# [AEE-706] 錯誤復原

## 情境

代理迴圈會失敗。工具會逾時 (timeout)。模型拒絕採取行動。歷史記錄不斷增長，直到超出情境視窗。模型陷入困境，一再重複相同動作。這些都不是邊緣案例——它們是任何正式代理部署中可預期的狀況。一個遇到第一次失敗就停止的 harness 是脆弱的；一個能在錯誤發生時維持迴圈前進能力的 harness 才是有韌性的。

錯誤復原 (error recovery) 不是為了防止失敗，而是為了在失敗發生時維持迴圈繼續推進的能力。

## 設計思維

**韌性**不是防止錯誤，而是在錯誤發生時維持迴圈前進的能力。遇到第一次失敗就停止的 harness 是脆弱的；具備錯誤預算 (error budget) 的 harness 才能持續發揮作用。

錯誤預算模式是核心概念：追蹤連續失敗次數。在連續 N 次失敗後，升級給人工處理或終止並附上診斷資訊。這能防止無限重試迴圈——這是最危險的 harness 失敗模式之一。無止盡重試的代理不只浪費資源，還可能造成複合性損害。

**RFC 2119：**

- Harness 在沒有延遲或條件改變的情況下，絕對不可 (MUST NOT) 重試失敗的工具呼叫。立即重試失敗的呼叫不是復原——而是放大問題。
- 當錯誤預算耗盡時，harness 必須 (MUST) 終止迴圈。超出預算後仍繼續執行的 harness，無法區分陷入停滯的迴圈與正常運作的迴圈。
- Harness 應該 (SHOULD) 記錄每一個錯誤及其復原動作。若沒有記錄失敗內容與嘗試方式，事後分析將無從進行。

## 深入探討

### 錯誤分類

| 錯誤類型 | 原因 | 可復原？ |
|---|---|---|
| 工具執行失敗 (tool execution failure) | 工具回傳錯誤結果或拋出例外 | 通常可以（若為暫時性錯誤） |
| 模型拒絕 (model refusal) | 模型拒絕行動（安全過濾器、能力限制） | 很少 |
| 情境溢位 (context overflow) | 歷史記錄超出情境視窗 | 可以——透過截斷或摘要 |
| 循環停滯 (loop stall) | 模型在 N 個回合內未產生有用輸出 | 有時候 |
| 逾時 (timeout) | 工具或模型呼叫超過時間限制 | 通常可以（若為暫時性錯誤） |

### 各類錯誤的復原策略

**工具執行失敗：**
在工具結果中回傳 `is_error: true`，讓模型決定是否重試、採用不同方式，或回報失敗。針對暫時性錯誤，在重試前加入退避 (backoff)。

```python
def handle_tool_failure(error: Exception, tool_name: str, attempt: int) -> ToolResult:
    is_transient = isinstance(error, (RateLimitError, NetworkError, TimeoutError))
    if is_transient and attempt < MAX_RETRIES:
        time.sleep(BACKOFF_SECONDS * (2 ** attempt))  # exponential backoff
        return None  # signal retry
    return ToolResult(
        is_error=True,
        content=f"{tool_name} failed: {str(error)}. Attempt {attempt}/{MAX_RETRIES}."
    )
```

**情境溢位：**
1. 將最舊的 N 個回合摘要成壓縮形式。
2. 以摘要取代歷史記錄中已摘要的回合。
3. 使用壓縮後的歷史記錄繼續迴圈。

**循環停滯：**
1. 偵測停滯（參見 AEE-701 停滯偵測 (stall detection)）。
2. 注入復原提示：「你似乎陷入困境。是什麼在阻礙你？請嘗試不同的方法。」
3. 若再經過 M 個回合仍無進展，終止並附上診斷資訊。

**逾時：**
取消工具呼叫，回傳帶有逾時訊息的 `is_error: true`，繼續迴圈。讓模型決定是否重試或改採其他方式。

**模型拒絕：**
模型對任務的拒絕表示該任務或工具組合超出模型的操作邊界。重試相同提示不會成功。記錄拒絕事件，升級給人工處理，並終止迴圈。

### 錯誤預算模式

追蹤連續失敗次數。在連續 N 次失敗後升級處理：

```python
class ErrorBudget:
    def __init__(self, max_consecutive: int = 3):
        self.max_consecutive = max_consecutive
        self.consecutive_failures = 0

    def record_success(self):
        self.consecutive_failures = 0

    def record_failure(self) -> bool:
        """Returns True if budget is exhausted."""
        self.consecutive_failures += 1
        return self.consecutive_failures >= self.max_consecutive

    def is_exhausted(self) -> bool:
        return self.consecutive_failures >= self.max_consecutive
```

當預算耗盡時：終止迴圈，並回傳診斷訊息，描述最後 N 次失敗並建議後續步驟。

### 可復原的 (recoverable) vs. 不可復原的 (unrecoverable) 錯誤

| 錯誤 | 可復原？ | 復原動作 |
|---|---|---|
| 超過速率限制 | 是 | 退避後重試 |
| 網路逾時 | 是 | 退避後重試 |
| 工具回傳非預期格式 | 有時候 | 讓模型以改寫的輸入重試 |
| 模型在每個回合都拒絕 | 否 | 終止；升級給人工 |
| 摘要後情境視窗仍然耗盡 | 否 | 終止；建立檢查點並在新工作階段重啟 |
| 工具無法使用（部署問題） | 否（修復前） | 終止；通知值班人員 |

### 每個失敗點應記錄的資訊

```json
{
  "timestamp": "2026-04-14T10:23:45Z",
  "session_id": "sess_abc123",
  "turn": 7,
  "error_type": "tool_execution_failure",
  "tool_name": "search_web",
  "error_message": "rate limit exceeded",
  "recovery_action": "backoff_retry",
  "attempt": 2,
  "budget_remaining": 2
}
```

## 最佳實踐

1. **將錯誤預算實作為第一類物件，而非單純的計數器。** 僅用整數表示的錯誤預算很容易忘記重置。具備 `record_success()`、`record_failure()` 和 `is_exhausted()` 方法的第一類 `ErrorBudget` 物件是明確、可測試且不易誤用的。在每次成功時重置，而不只是在迴圈重啟時重置。

2. **永遠不要在失敗後立即重試。** 立即重試失敗的呼叫會放大問題。基本原則是：第一次重試至少等待一秒，每次後續重試加倍等待。設定最大退避時間（例如 60 秒）以防止無限期停滯。

3. **以診斷資訊終止，而非靜默失敗。** 當錯誤預算耗盡時，回傳描述已嘗試內容、嘗試次數及失敗原因的診斷資訊。這能讓使用者有足夠資訊決定是否重試、重新定義任務或升級處理。
