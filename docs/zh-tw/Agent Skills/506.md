# [AEE-506] 技能治理 (Skill Governance)

## 情境

一個具備良好管理實踐的技能庫（AEE-505）——具名擁有者、版本化技能、登錄表——若缺乏治理仍可能失敗。治理是運營層：決定誰可以發布、技能如何演進、以及如何退場的政策、流程與執行機制。沒有治理，管理實踐就會走樣。

治理的需求出現在一個特定門檻：當技能數量超過任何個人能掌握的範圍，且一個損壞或過時的技能造成真實的運營問題時。對多數組織規模的技能庫而言，這個門檻大約在 20–30 個活躍技能時到來。

## 設計思維

核心主張：技能治理 (skill governance) 是防止技能管理在自身重量下崩潰的運營層。沒有治理，組織規模的技能庫會不斷累積過時技能、相互矛盾的標準，以及缺乏退場路徑的困境。治理定義了誰可以發布、技能如何審查，以及如何退場。

**何謂技能的重大變更 (breaking change)：**

對程式碼而言，重大變更會破壞 API 合約。對技能而言，重大變更會導致使用該技能的代理人以消費者未預期的方式產生不同行為：
- 移除消費者所依賴的觸發條件
- 縮小範疇，使技能不再適用於先前涵蓋的場景
- 新增覆寫消費者層級自訂設定的硬性限制
- 變更預期輸出格式（例如，原先提供逐行回饋的程式碼審查技能，改為僅提供摘要回饋）

不構成重大變更的情況：在不改變行為的前提下提升清晰度、新增「不適用時機」指引但不影響現有使用場景、更新範例。

**版本控制方案：**

語意化版本控制（semver）可直接套用：
- **MAJOR**：重大變更（行為變更、範疇縮減、限制新增）
- **MINOR**：非破壞性新增（新指引、範疇擴展、新範例）
- **PATCH**：非破壞性修正（清晰度改善、錯字更正、URL 更新）

組織範疇的 MAJOR 版本升級，需要在移除舊版本前經過棄用週期 (deprecation lifecycle)。消費者必須能夠鎖定至舊 MAJOR 版本，同時進行遷移。

**棄用週期：**

1. **已棄用 (Deprecated)**：仍可運作，但在登錄表中標記為已棄用。技能本文包含：「This skill is deprecated as of vX.0.0. Use `replacement-skill` instead. Will be removed on [date].」
2. **日落 (Sunset)**：不再更新；不修正錯誤。移除日期確認。
3. **已移除 (Removed)**：從登錄表和發布管道中移除。已鎖定版本的消費者可繼續運作；追蹤最新版本的消費者將收到技能缺失錯誤。

組織範疇的最短棄用期限：MINOR 替換（相同目的、更好的實作）為 30 天；MAJOR 替換（不同行為、需要消費者調整）為 90 天。

**RFC 2119：**

- Skills MUST NOT be removed from org scope without a deprecation period communicated in the skill itself and in the registry.
- Org skill publishers MUST document breaking changes between major versions, including: what changed, why it changed, and how consumers should migrate.
- Audit trails for agent sessions SHOULD include the active skill set and versions at session start.

## 深入探討

### 發布至組織範疇前的品質關卡 (Quality Gates)

在技能晉升至組織範疇之前：

```markdown
## Org Skill Publication Checklist

- [ ] Single primary purpose — stated in one sentence without "and"
- [ ] Invocation condition defined — specific enough to auto-select
- [ ] "When NOT to use" section present — at least one exclusion specified
- [ ] Scope defined — explicit boundaries on what the skill covers
- [ ] Version assigned — following semver, starting at 1.0.0
- [ ] Owner named — an individual or team accountable for maintenance
- [ ] Variation tested — 5 positive cases and 5 negative cases passing
- [ ] Conflict check — no invocation condition overlap with existing org skills
- [ ] Changelog entry — describes what this version adds or changes
```

高風險技能（每位工程師的工作階段中都會啟用、且可能影響正式環境系統的技能）需要額外審查：在團隊範疇進行試運行期、多位審查者，以及來自資安或合規利害關係人的核准。

### 存取控制 (Access Control)

**個人範疇**：無需控管。

**團隊範疇**：任何團隊成員皆可提案；至少需要一位其他團隊成員審查。PR 留言即已足夠。

**組織範疇**：由指定群組（DX 團隊、平台團隊或治理委員會）核准發布。角色分離：技能作者提案；核准者為不同的人。組織範疇不允許自我核准。

**維護組織技能**：具名擁有者可在無需額外核准的情況下發布 PATCH 和 MINOR 更新。MAJOR 更新需要與新技能相同的審查流程。

### 稽核軌跡 (Audit Trail)

一個在技能版本 X 下執行且產生非預期輸出的代理人工作階段，若不知道版本 X 處於活躍狀態，就無法診斷問題。工作階段稽核記錄必須包含：

```json
{
  "session_id": "sess_abc123",
  "timestamp": "2026-04-14T10:00:00Z",
  "active_skills": [
    { "name": "diagnose-root-cause", "version": "2.1.0", "scope": "org" },
    { "name": "backend:api-design", "version": "1.0.0", "scope": "team" }
  ],
  "active_tools": ["bash", "read_file", "search_web"]
}
```

這與 AEE-406（工具呼叫的稽核日誌）相互呼應：同一份擷取工具呼叫的工作階段記錄，也應擷取活躍技能集合。改變代理人行為的技能更新，與工具權限變更同等重要。

### 治理反模式

**技能墓地 (skill graveyard)：** 組織登錄表中沒有擁有者、沒有近期更新、也沒有消費者的技能——但因為沒有人願意費心去棄用它們，這些技能仍維持「活躍」狀態。墓地隨時間增長，降低登錄表的訊噪比。預防方式：每季進行擁有權稽核；任何超過 90 天沒有確認擁有者的技能，自動進入棄用狀態。

**影子個人技能 (shadow personal skill)：** 實踐者因為組織技能不符合自身工作流程，建立了一個覆寫組織技能的個人技能。這個覆寫從未被審查、從未被呈報給團隊，並在無聲之中偏離了組織標準。當該實踐者的代理人產出與其他人不同的結果時，影子技能在診斷上是隱形的。預防方式：組織技能被覆寫時應記錄日誌；應鼓勵實踐者向組織技能提案變更，而非維護靜默的覆寫。

**巨石技能 (monolith skill)：** 一個涵蓋某個領域所有事項的技能——組織層級的神技能。它起初是聚焦的，但隨時間不斷延伸，因為更新一個檔案比建立新技能並更新登錄表更容易。等到問題浮現時，這個巨石已大到無法在不破壞消費者的情況下拆解。預防方式：在組織發布時強制執行單一目的清單；將技能本文中的每個「且」視為拆解訊號。

## 最佳實踐

1. **在技能本文內發布棄用通知，而不只是在登錄表中。** 當代理人載入已棄用的技能時，棄用通知是它所讀取內容的一部分。代理人可以在使用當下，以行內方式將棄用通知呈報給實踐者。

2. **將改變行為的技能更新視為部署，而非編輯。** 組織技能的行為變更會影響組織中的每一位工程師。以與正式環境部署相同的謹慎態度處理：提前公告、提供遷移路徑、在過渡期間維護舊版本。

3. **將活躍登錄表與封存區分開來。** 已棄用和日落的技能應放在登錄表的獨立區段。正在尋找技能的實踐者不應在已棄用的內容中費力搜尋。
