# [AEE-502] 代理技能生態系統

## 情境

代理技能生態系統（agent skill ecosystem）存在於每一款主流 AI 程式開發工具之中，但各平台對它的稱呼不盡相同：斜線指令（slash commands）、規則、指示、工作流程、外掛。每個平台都有自己的呼叫模型、發佈機制，以及對行為引導範疇的不同定義。只熟悉自家工具系統的從業者，會錯過其他情境中已被驗證的模式與技術。

本文調查代理技能生態系統的現況——各平台現有的系統、各系統的運作方式，以及儘管名稱各異，各系統之間共享的模式。

## 設計思維

核心主張：代理技能生態系統目前仍屬分散，但正在朝同一方向收斂。各平台以不同名稱提供技能，但它們共享一個共同模式：在呼叫時將封裝好的行為引導載入代理的上下文。理解這個生態系的全貌，讓從業者能直接採用已驗證的模式，而不必從頭摸索。

收斂模式（convergence pattern）——每個平台的技能系統共有的元素：

1. 一個具名的行為引導單元
2. 一種將該引導載入代理上下文的機制
3. 某種呼叫觸發條件（明確指令、檔案存在，或條件比對）
4. 一種儲存格式（檔案、資料庫項目，或 API 設定）

各平台之間的差異在於程度：強制結構化的程度、技能的組合性（composability）、技能的發佈與版本管理（versioning）機制。

## 深入探討

### Agent Skills 開放標準

在調查各平台之前，有必要先介紹共同的基礎規格。[Agent Skills 開放標準](https://agentskills.io)對代理技能的定義是：「一個包含指示、腳本與資源的資料夾，讓代理能夠發現並使用，以更準確、更有效率地完成任務。」此標準最初由 Anthropic 開發，隨後作為開放標準向更廣泛的生態系釋出。截至 2026 年 4 月，已有超過 30 款工具實作此標準，包括 Claude Code、Gemini CLI、GitHub Copilot、Cursor、VS Code、Amp 等。

此標準下，技能的結構單元是一個以 `SKILL.md` 作為必要進入點的目錄，可包含選用的支援檔案（模板、範例、腳本、參考文件）。規格說明位於 [agentskills.io/specification](https://agentskills.io/specification)，GitHub 儲存庫位於 [github.com/agentskills/agentskills](https://github.com/agentskills/agentskills)。

下文中提到某平台「實作此標準」，意即該平台能識別此目錄結構，並在技能被呼叫時載入 `SKILL.md`。

### Claude Code 技能

Claude Code（命令列工具）實作 Agent Skills 開放標準。文件位於 [code.claude.com/docs/en/skills](https://code.claude.com/docs/en/skills)。

**儲存路徑：**

- 個人（使用者層級）：`~/.claude/skills/<skill-name>/SKILL.md`
- 專案層級：`.claude/skills/<skill-name>/SKILL.md`
- 外掛層級：`<plugin>/skills/<skill-name>/SKILL.md`（透過 Claude Code 外掛管理器安裝的技能，存放於外掛自身的目錄樹內）

舊版 `.claude/commands/` 路徑仍可向下相容使用。現有的 `.claude/commands/` 檔案持續有效，但 `.claude/skills/` 是新技能的建議路徑。

**呼叫方式：** 使用者輸入 `/skill-name` 可明確呼叫技能。當 Claude 判斷某技能的描述符合當前任務時，也可自動載入。明確呼叫與自動比對兩種方式皆支援。

**術語說明：** 早期的 Claude Code 文件以「自訂指令」（custom commands）和「斜線指令」（slash commands）稱呼此功能。目前的文件統一使用「技能」（skills）一詞，反映對 Agent Skills 開放標準的遵循。

### Superpowers

[Superpowers](https://github.com/obra/superpowers) 是由 Jesse Vincent（obra）與社群貢獻者共同建立的開源代理技能框架與軟體開發方法論。它並非 Anthropic 的產品，而是一款社群外掛，會將一套精選技能安裝至 Claude Code 中。

**安裝方式：** 透過 Claude Code 外掛系統：

```
/plugin install superpowers@claude-plugins-official
```

外掛列表位於 [claude.com/plugins/superpowers](https://claude.com/plugins/superpowers)。發佈管道為 Claude Code 外掛管理器與 GitHub，本專案並無獨立的 npm 套件。

**呼叫方式：** 技能在描述符合任務上下文時自動觸發，也可由使用者直接輸入 `/skill-name` 呼叫（例如 `/brainstorming`、`/execute-plan`）。自動呼叫與手動呼叫兩種方式皆支援。

**與 Claude Code 技能的關係：** Superpowers 安裝的技能符合 Claude Code 的技能系統規格。使用者透過 `/plugin install` 安裝 Superpowers 後，外掛的技能會與使用者個人或專案層級定義的技能共同使用。

### Cursor

Cursor 有兩套重疊的行為引導功能：Cursor Rules 與 Cursor Skills。

**Cursor Rules** 是較早推出、使用更廣泛的系統。文件位於 [cursor.com/docs/rules](https://cursor.com/docs/rules)。專案規則以 markdown 檔案形式存放於 `.cursor/rules`，並納入版本控制。Cursor 支援 `.md` 與 `.mdc` 副檔名。MDC 檔案支援 YAML 前言（frontmatter），欄位包含 `description`、`alwaysApply` 與 `globs`，可設定為永遠套用、依檔案路徑模式比對，或依描述比對。

```
.cursor/rules/
  commit-style.mdc
  api-conventions.mdc
```

典型的 MDC 規則檔案：

```yaml
---
description: "Apply when the user asks about API design or REST endpoints"
globs: ["src/api/**"]
---

Follow our REST conventions: ...
```

**Cursor Skills** 實作 Agent Skills 開放標準。技能文件位於 [cursor.com/docs/context/skills](https://cursor.com/docs/context/skills)，Cursor 在 agentskills.io 上被列為該網址的實作者。Cursor Rules 與 Cursor Skills 的關係是：Rules 是 Cursor 自有的行為引導系統，而 Skills 代表 Cursor 對跨平台 Agent Skills 標準的支援。截至 2026 年 4 月，兩套系統在 Cursor 中並存；使用 Cursor 管理團隊共享行為引導的從業者可依需求選擇任一方式。

目前的 Cursor 文件未提及舊版單一檔案格式 `.cursorrules`。目前的標準路徑為 `.cursor/rules/*.mdc`。

### Custom GPT 指示（OpenAI）

OpenAI Custom GPTs 將行為引導以持久系統層級指示（persistent system-level instructions）的形式載入，套用至該 GPT 的所有對話。與上傳的 Knowledge 檔案（用於檢索）不同，指示具有結構性——定義 GPT 的行為方式，而非它所知道的內容。

指示透過 GPT Builder 介面設定，儲存為 GPT 定義的一部分。此系統沒有檔案型儲存路徑；指示以文字形式在建構介面中輸入，並與 OpenAI 系統中的 GPT 記錄相關聯。

**呼叫方式：** 指示在每次對話時自動注入，無需逐次觸發條件；GPT 的指示始終處於啟用狀態。

**發佈：** Custom GPT 以 URL 形式分享，或發佈至 GPT Store。指示隨 GPT 定義一併傳遞，而非以獨立工件存在。

相關 OpenAI 文件：
- [Creating a GPT](https://help.openai.com/en/articles/8554397-creating-a-gpt)
- [Key guidelines for writing instructions](https://help.openai.com/en/articles/9358033-key-guidelines-for-writing-instructions-for-custom-gpts)

注意：截至 2026 年 4 月，OpenAI Help Center 封鎖自動化內容擷取。上述 URL 已透過搜尋索引確認存在；指示系統的具體行為描述來源為搜尋結果摘要。

### 平台比較

| Platform | Format | Invocation | Distribution | Composability | Versioning |
|----------|--------|------------|--------------|---------------|------------|
| Claude Code | Directory + `SKILL.md` (Agent Skills standard) | `/skill-name` or auto-match by description | File system; plugin manager | Multiple skills compose at invocation time | File-based; use git for version control |
| Superpowers (plugin) | Directory + `SKILL.md` (via Claude Code) | `/skill-name` or auto-match | `/plugin install superpowers@claude-plugins-official`; GitHub | Composes with other Claude Code skills | Plugin versioning via plugin manager |
| Cursor Rules | `.mdc` or `.md` files in `.cursor/rules/` | `alwaysApply`, glob match, or description match | Git-committed with repository | Multiple rules apply simultaneously | File-based; use git |
| Cursor Skills | Directory + `SKILL.md` (Agent Skills standard) | Auto-match or explicit invocation | File system; cross-platform | Composes per standard | File-based; use git |
| Custom GPT (OpenAI) | Text field in GPT Builder UI | Always active (system prompt injection) | GPT URL or GPT Store | Single instruction set per GPT | GPT record versioning (builder UI) |

### 收斂模式

儘管術語與工具各異，這些系統共享一套共同結構。每個平台的技能系統都：

1. **為引導單元命名。** 無論稱之為「技能」、「規則」、「Custom GPT」還是「指令」，每個平台都有一個具名的事物代表一個行為引導單元。

2. **將引導載入上下文。** 每個系統都會在代理處理請求之前，將引導注入代理的上下文——以系統提示、前置指示或載入檔案的形式。

3. **具備呼叫機制。** 機制各有不同：永遠啟用（Custom GPT）、明確指令（`/skill-name`）、條件比對（Cursor MDC frontmatter、Superpowers 自動比對）或路徑模式（glob）。但每個系統都有其機制。

4. **使用儲存格式。** 檔案（Claude Code、Cursor）、文字欄位（Custom GPT Builder）或資料庫項目。檔案型系統可順帶獲得版本控制的好處；API 儲存型系統需要額外的版本管理紀律。

已遷移至 Agent Skills 標準的平台（Claude Code、Cursor、Gemini CLI 等）之所以向目錄加 `SKILL.md` 的格式收斂，正是因為這種格式解決了臨時文字欄位與單一 markdown 檔案無法處理的版本管理與發佈問題。以目錄形式結構化的技能，可在核心指示之外包含範例、模板、參考文件與自動化腳本——這是文字欄位難以企及的豐富工件形式。

剩餘的分散性主要在於發佈與組合性。Custom GPTs 以應用實例的形式發佈；符合 Agent Skills 標準的平台以檔案或套件形式發佈技能。Custom GPT 的組合性有限（每個 GPT 僅有一套指示）；在符合 Agent Skills 標準的平台上，多個技能可在呼叫時自然組合。

## 最佳實踐

1. **依發佈需求選擇技能格式，而非以熟悉度為準。** 若一個技能需要隨儲存庫傳遞並套用至整個團隊，檔案型格式（Claude Code 技能、Cursor rules/skills）是自然的選擇——它們與所管理的程式碼一同進行版本控制。若你要建立一個供非技術使用者透過 URL 存取的獨立助理，Custom GPT 指示則是合適的選擇。應依發佈管道選擇格式，而非依你最常用的工具。

2. **即使平台不要求，也要加入明確的呼叫條件。** Cursor 的 `alwaysApply: true` 與 Custom GPT 的永遠啟用注入方式很方便，但這表示每次互動都要承載所有啟用引導的全部重量。明確的呼叫條件——Cursor MDC frontmatter 中的 `description` 欄位、`SKILL.md` 標頭中的呼叫條件——能讓 harness 或代理為正確的任務選取正確的技能，降低雜訊並提升可靠性。先寫呼叫條件，再寫技能內容。

3. **將組合性視為需要主動設計的功能。** 單一技能系統（一個大型規則檔案、一個 Custom GPT 指示區塊）雖然容易起步，但難以維護。支援多個組合技能的平台，能讓你建立模組化引導函式庫——偵錯、提交訊息撰寫、程式碼審查、領域慣例等技能各自獨立，可分別更新。採用新平台時，從一開始就將引導設計為多個具名單元，而非單一的龐大區塊。
