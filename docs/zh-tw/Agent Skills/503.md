# [AEE-503] 技能設計

## 情境

光是撰寫代理會遵循的行為指引還不夠。這些指引必須被封裝成技能，才能可靠地被呼叫、隨時間維護，並與其他技能組合使用。封裝時的決策——技能如何命名、觸發條件（invocation condition）如何描述、範疇邊界（scope boundary）劃在哪裡——決定了這個技能是可重用的資產，還是一次性的提示詞。

大多數技能庫累積的是雜亂產物，而非資產。差別在於設計紀律：資產是以特定目的、明確觸發條件和定義好的限制來設計的；雜亂產物則是為了某個當下而撰寫，從未賦予那些東西。

## 設計思維

核心主張：做太多事的技能會被忽略或誤用。最常見的技能設計失敗是範疇膨脹（scope creep）——一個原本是「如何除錯」的技能，變成了「出問題時如何做所有事」。技能設計是一門在真實使用情境下劃定並維持範疇邊界（scope boundary）的紀律。

**範疇邊界問題：**

技能之所以成長，是因為追加指引比建立新技能更容易。除錯技能被擴充了撰寫測試的指引（因為測試能防止 bug），再被擴充了重構指引（因為乾淨的程式碼更容易除錯），再被擴充了程式碼審查指引（因為審查能抓到 bug）。每次新增在局部都合理；累積起來卻成了一個適用於所有事情的技能。

當一個萬能技能（god-skill）被載入時，它會與每個其他技能競爭代理的注意力。結果是行為不一致——代理遵循萬能技能的某些指令，忽略其他指令，而實踐者無法預測哪些會被遵循。

**單一職責原則（single responsibility）：**

每個技能應該只有一個主要目的。測試方法：你能用一句話、不帶「和」字來陳述技能的目的嗎？如果不行，技能就有超過一個目的。

**命名慣例：**

- **動詞-名詞命名（verb-noun naming）格式**：`diagnose-root-cause`、`write-commit-message`、`review-pull-request`。動詞編碼代理的行為；名詞指定領域。
- **組織技能加上領域前綴**：`backend:api-design`、`frontend:component-review`。防止跨團隊的命名衝突。
- **具體優於一般**：`generate-jest-test-cases` 比 `testing` 更好。具體的名稱本身就是隱含的範疇邊界。

**觸發條件作為設計限制：**

撰寫觸發條件能強制釐清範疇。如果你無法用一句話寫出它，技能的範疇就尚未妥善界定。

- 太寬泛：「當使用者的程式碼有問題時」
- 太狹隘：「當使用者貼上來自 Linux 上 Django 應用程式的 Python 堆疊追蹤時」
- 範疇恰當：「當使用者回報非預期的行為並要求協助找出原因時」

**撰寫「不應使用的情境」段落：**

這是技能設計中最常被跳過的部分。負面觸發條件能防止技能被應用於其並非為之設計的請求——尤其是那些看起來相似但需要不同技能的請求。

除錯技能應指定：「當使用者要求對正常運作的程式碼進行程式碼審查、當使用者想要撰寫測試，或當使用者想要重構時，請勿使用本技能。」

- 技能 MUST 只有一個主要目的，且可在不使用「和」字的情況下用一句話陳述。
- 技能 MUST 指定至少一個不應呼叫它們的條件——特別是當另一個技能可能與它混淆時。
- 技能名稱 SHOULD 以動作為導向且無歧義。

## 深入探討

### 實作範例：從萬能技能到三個技能

**重構前：** 一個隨時間成長的單一 `engineering-assistant` 技能。

```markdown
---
name: engineering-assistant
description: Help with engineering tasks
---

When the user has an engineering problem, help them solve it.

If there's a bug: ask for the error message and stack trace. Identify the root cause. Suggest a fix.

If there's a test to write: analyze the code under test. Write tests for the happy path, edge cases, and error conditions.

If there's code to review: check for correctness, readability, performance, and security. Provide feedback with specific line references.

If there's a refactor: identify code smells. Propose a cleaner structure. Preserve behavior.

When in doubt, ask the user what they need.
```

（技能本文以英文撰寫，與語言環境無關。）

問題：
- **觸發條件**：缺失。「協助工程任務」太寬泛，無法自動選擇。
- **範疇**：未定義。涵蓋除錯、測試、程式碼審查和重構——四個截然不同的活動。
- **「不應使用的情境」**：缺席。
- **衝突風險**：任何針對這些活動的專門技能都將與此技能衝突。

**重構後：** 三個範疇明確的技能。

```markdown
---
name: diagnose-root-cause
description: Identify the root cause of unexpected software behavior. Use when the user reports a bug or unexpected output and asks why it is happening.
invocation: when the user reports unexpected behavior and asks for help identifying the cause
scope: root cause identification only — does not write fixes, tests, or refactors
---

When diagnosing a bug:
1. Ask for the error message, stack trace, and minimal reproduction steps if not provided.
2. Identify the failing assumption: what did the code expect vs. what happened?
3. Trace the execution path to the point of divergence.
4. State the root cause as a specific claim: "The bug is caused by X when Y."
5. Stop here. Do not propose a fix unless explicitly asked.

Do NOT write test cases. Do NOT refactor the code. Do NOT conduct a code review.
Use `write-test-cases` or `review-pull-request` for those tasks.
```

（技能本文以英文撰寫，與語言環境無關。）

```markdown
---
name: write-test-cases
description: Write test cases for a specific function or module. Use when the user wants test coverage written, not when they are debugging existing failures.
invocation: when the user asks for tests to be written for specific code
scope: test case authoring only — does not debug, refactor, or review
---

When writing tests:
1. Identify the function or module under test and its public interface.
2. List the scenarios: happy path, boundary values, error conditions, edge cases.
3. Write tests in the project's existing test framework (detect from imports or ask).
4. Each test should test one behavior. Name tests after the behavior: `test_returns_empty_list_when_no_items`.

Do NOT diagnose bugs in existing tests. Do NOT refactor the code under test.
Use `diagnose-root-cause` if the tests are failing unexpectedly.
```

（技能本文以英文撰寫，與語言環境無關。）

```markdown
---
name: review-pull-request
description: Review code changes for correctness, readability, and maintainability. Use when the user shares code for review, not when they are debugging or writing tests.
invocation: when the user asks for a code review or shares a diff/PR for feedback
scope: code review feedback only — does not debug, write tests, or implement changes
---

When reviewing code:
1. Read the full diff before commenting.
2. Organize feedback by severity: blocking issues, important suggestions, minor notes.
3. For each issue: state what, why it matters, and how to fix it.
4. End with an overall assessment: approve, approve with suggestions, or request changes.

Do NOT write replacement code. Do NOT debug issues found during review — flag them.
Do NOT write tests — recommend them.
```

（技能本文以英文撰寫，與語言環境無關。）

這三個技能合在一起涵蓋了與萬能技能相同的領域。每個技能都可以獨立呼叫、獨立測試，並可組合使用。實踐者可以使用 `diagnose-root-cause` 而不需要載入測試或審查指引。

### 發佈前測試技能

在將技能從個人範疇推廣至團隊範疇之前：

1. **正向測試**：提出 5 個應該觸發此技能的請求。代理是否對每個請求都正確應用了技能的指引？
2. **負向測試**：提出 5 個看起來相似但不應觸發此技能的請求。代理是否避免應用它？
3. **邊界測試**：提出 3 個位於範疇定義邊緣的請求。行為是否一致？

若任何正向測試失敗，技能主體需要釐清。若任何負向測試失敗，「不應使用的情境」段落不夠充分。這與 AEE-306（提示詞穩健性測試）的方法論相同，應用於技能。

## 最佳實踐

1. **每次技能設計都從撰寫觸發條件和範疇開始，而非主體。** 若先撰寫主體，範疇將會配合你所寫的內容，而非技能實際需要涵蓋的內容。

2. **明確測試負向情境。** 未經過「不應處理的請求」測試的技能，不應推廣至團隊或組織範疇。一個對所有事都觸發的技能，比沒有技能更糟糕。

3. **在第一次出現「和」字的跡象時就進行分解。** 當你發現自己在技能主體中撰寫「若 X 則如此，若 Y 則如此」時，停下來。提取 X 技能和 Y 技能；讓執行框架（harness）負責路由。
