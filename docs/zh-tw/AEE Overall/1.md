# [AEE-1] 詞彙表

:::info
AEE 全系列使用的標準術語定義。如果業界用法存在差異，本詞彙表會明確說明 AEE 的含義。
:::

## Context

智能體 AI 工程從多個學科繼承了術語——強化學習、分散式系統、軟體工程和自然語言處理——這些術語的含義往往相互衝突或過度使用。本詞彙表確立了 AEE 全系列使用的定義。文章引用本詞彙表，而不是在行文中重複定義術語。

## 術語

### 智能體 (Agent)

一種以 LLM 作為推理引擎的軟體系統，能夠感知目標、規劃多步驟工作流程、呼叫外部工具，並以最少的人工監督自主完成任務。

來源：[Google Cloud — What Is Agentic AI](https://cloud.google.com/discover/what-is-agentic-ai)

### 自主性頻譜 (Autonomy Spectrum)

描述 AI 系統行使獨立決策程度的概念軸，從完全由人類主導（副駕駛 / 自動完成）、到有監督的自主性（人在迴路中），再到完全自主運作（無監督、長時程執行）。

注意：請參閱 AEE-106。

來源：[Agentic AI: A Comprehensive Survey (arXiv 2510.25445)](https://arxiv.org/abs/2510.25445)

### 上下文視窗 (Context Window)

LLM 在單次前向傳遞中能夠處理的最大 token 數量（輸入 + 輸出）；決定了模型一次能看到多少歷史記錄、指令和檢索到的資料。截至 2024–2025 年，主流模型提供 128K–1M+ token 的視窗。

注意：請參閱 AEE-108。

來源：[IBM Think — Context Window](https://www.ibm.com/think/topics/context-window)

### 微調 (Fine-tuning)

在任務專屬或領域專屬的資料集上繼續訓練預訓練基礎模型，以專化其輸出；會更新模型權重（不同於 RAG 在推理時注入知識）。常見子類型包括監督式微調（SFT）、RLHF 和 RLVR。

來源：[Hugging Face — Fine-tuning a Pretrained Model](https://huggingface.co/docs/transformers/training)

### 函數呼叫 (Function Calling)

OpenAI 對工具使用的原始術語——LLM 產生一個結構化 JSON 負載，指定函數名稱及其參數；宿主應用程式執行該函數，並將返回值送回上下文。強調離散的、可執行的程式碼塊，而非更廣泛的工具抽象。

注意：在協議層面與工具使用（Tool Use）有所不同。

來源：[Fireworks AI — Function Calling](https://fireworks.ai/blog/function-calling)

### 基礎接地 (Grounding)

將 LLM 的回應錨定在可驗證的事實資料上的實踐——通常透過 RAG、工具呼叫或在權威語料庫上進行微調——使輸出可追溯至真實世界的知識來源，而非僅依賴參數化記憶。

來源：[K2view — What Is Grounding and Hallucinations in AI](https://www.k2view.com/blog/what-is-grounding-and-hallucinations-in-ai/)

### 幻覺 (Hallucination)

當 LLM 生成事實上不正確、內部不一致或完全捏造的文字，同時表現得流暢且自信；起因於 LLM 被訓練為生成合理的 token 序列，而非驗證真實性。

注意：請參閱 AEE-110。

來源：[K2view — What Is Grounding and Hallucinations in AI](https://www.k2view.com/blog/what-is-grounding-and-hallucinations-in-ai/)

### 執行框架 (Harness)

所有包裹 LLM 但本身不是模型的程式碼、設定和執行邏輯：編排迴路、工具登錄表、記憶體管理、上下文組裝、狀態持久化、錯誤處理和防護機制。「智能體 = 模型 + 執行框架。」

注意：請參閱 AEE-700。

來源：[Parallel AI — What Is an Agent Harness](https://parallel.ai/articles/what-is-an-agent-harness)

### 推理 (Inference)

對新輸入執行已訓練模型的前向傳遞以生成輸出的過程；對於自迴歸 LLM，這是逐 token 取樣的迭代過程。與訓練不同——推理期間不更新模型權重。

來源：[BentoML — How Does LLM Inference Work](https://bentoml.com/llm/llm-inference-basics/how-does-llm-inference-work)

### 編排 (Orchestration)

管理 LLM 智能體如何排序動作、委派給子智能體或工具、跨步驟追蹤狀態，並將輸出路由回推理迴路的協調層；多步驟智能體工作流程的「指揮家」。

注意：請參閱 AEE-900 系列。

來源：[IBM Think — LLM Agent Orchestration with LangChain and Granite](https://www.ibm.com/think/tutorials/llm-agent-orchestration-with-langchain-and-granite)

### 提示 (Prompt)

提交給 LLM 以引發回應的文字（或多模態輸入）。在智能體系統中，提示由執行框架動態組裝，通常包含系統提示、檢索到的上下文、工具定義、對話歷史和當前用戶訊息。

來源：[Prompting Guide — Function Calling](https://www.promptingguide.ai/agents/function-calling)

### RAG（檢索增強生成）

一種透過在推理時從外部知識庫檢索相關文件並將其附加到提示前面，將 LLM 輸出錨定於事實性、最新資訊的技術。由 Lewis 等人（NeurIPS 2020）提出。

注意：請參閱 AEE-600 系列。

來源：[Lewis et al. — Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks (arXiv 2005.11401)](https://arxiv.org/abs/2005.11401)

### ReAct

一個提示 / 智能體迴路框架（Yao 等人，ICLR 2023），交替執行推理（Reasoning）軌跡和行動（Acting）步驟——模型在生成思維鏈和發出工具呼叫之間交替，每次工具的觀察結果在下一步思考前送回。與純思維鏈相比，能改善接地性並減少幻覺。

注意：請參閱 AEE-701。

來源：[Yao et al. — ReAct: Synergizing Reasoning and Acting in Language Models (arXiv 2210.03629)](https://arxiv.org/abs/2210.03629)

### 從可驗證獎勵的強化學習 (RLVR)

一種訓練機制，其中獎勵信號來自客觀的、可程式驗證的真實答案檢查（例如單元測試通過/失敗、數學答案相等性），而非學習到的人類偏好模型。獎勵為二元（0/1）、防篡改，特別適合程式碼和數學領域。

注意：請參閱 AEE-102。

來源：[The AI Navigator — What Is Reinforcement Learning with Verifiable Rewards (RLVR)](https://www.theainavigator.com/blog/what-is-reinforcement-learning-with-verifiable-rewards-rlvr)

### 技能 (Skill)

透過執行框架向智能體公開的封裝、可重用行為模式——與原始工具（函數呼叫）不同，它將意圖、參數和執行邏輯封裝為一個具名的、可組合的單元，供智能體呼叫以完成特定子任務（例如 `web_search`、`run_code`、`send_email`）。

注意：這是 AEE 專案定義的術語。此概念也存在於 Microsoft Semantic Kernel 外掛程式中。請參閱 AEE-500。

來源：[Microsoft Semantic Kernel — Plugins](https://learn.microsoft.com/en-us/semantic-kernel/concepts/plugins/)

### 系統提示 (System Prompt)

附加在每次對話前面的特權模型級指令區塊；設定角色、規則、可用工具、輸出格式和安全約束。對最終用戶不可見，但模型在任何用戶輪次之前即會讀取。

來源：[Anthropic — System Prompts](https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/system-prompts)

### 分詞 / 標記化 (Tokenization)

將原始文字分割為 token 的預處理步驟——token 是 LLM 處理的原子單位。Token 大致對應詞片段（透過 BPE 或類似演算法的子詞）；約 1 token ≈ 0.75 個英文單字。決定文字長度如何對應至模型容量與成本。

來源：[NVIDIA Blog — AI Tokens Explained](https://blogs.nvidia.com/blog/ai-tokens-explained/)

### 工具使用 (Tool Use)

LLM 在運行時呼叫外部能力（API、資料庫、程式碼直譯器、檔案系統）的機制；模型發出一個結構化指令，指定要呼叫哪個工具及其參數，然後將結果作為觀察值接收。

注意：請參閱 AEE-400 系列。

來源：[Prompting Guide — Function Calling](https://www.promptingguide.ai/agents/function-calling)
