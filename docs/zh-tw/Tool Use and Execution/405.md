# [AEE-405] 工具選擇

## 情境

工具選擇 (tool selection) 是模型在每個回合中決定要呼叫哪個工具（如果有的話）的過程。這個決策完全由工程師提供的工具描述驅動。模型無法直接存取工具的實作內容——它只能閱讀描述，並據此推斷各工具的適用時機。當選擇失敗時，失敗的根源幾乎都指向描述問題，而非模型問題。

將工具選擇理解為一個提示驅動的決策——不是查找表，不是分類器，而是基於自然語言引導的推理步驟——會根本改變工程師處理問題的方式。解決選擇失效的方法不是更努力地提示模型，而是撰寫更好的描述。

## 設計思維

核心主張：工具選擇失效幾乎都是描述失效。當模型呼叫了錯誤的工具，或在應該呼叫工具時卻未呼叫，原因幾乎都是工具描述不明確、描述重疊或不完整——而非模型能力不足。

**模型如何選擇工具：**

模型在決定是否呼叫工具之前，會先閱讀 `tools` 陣列中所有工具的名稱與描述。它不會執行工具，也不會檢查工具的實作。選擇是一個推理步驟：給定使用者的請求和可用工具，應該呼叫哪個工具（如果有的話），以及傳入什麼參數？

這意味著工具選擇品質受到描述品質的限制。設計良好的工具，如果描述不明確，就無法被正確選擇。設計不佳的工具，如果描述過於寬泛，可能會被誤觸呼叫。

**選擇失效模式 (selection failure modes)：**

1. **選錯工具（描述重疊）：** 兩個工具的描述涵蓋相似的範圍。模型選了其中一個，有時選錯。修正方式：加入「何時不應使用」的子句，明確區分這些工具彼此的差異。

2. **應呼叫卻未呼叫（觸發條件模糊）：** 描述未清楚說明何時應呼叫工具。模型默認從訓練資料回答，而非呼叫工具。修正方式：加入具體的觸發條件（「當使用者詢問當前價格、即時資料或近期事件時，使用此工具」）。

3. **過度呼叫 (over-invocation)（描述過於寬泛）：** 模型對不適用的請求也呼叫了工具。常見於描述中出現「對任何 X 問題使用」——「任何」的範圍夠廣，會拉進鄰近的查詢。修正方式：縮窄觸發條件；加入「不應用於」的子句。

4. **呼叫不足 (under-invocation)（未標示即時資料需求）：** 模型應呼叫工具，卻改從訓練資料回答。常見於即時資料工具，當描述未標示資料具有時效性時。修正方式：加入「此工具提供當前即時資料——對於需要最新資訊的問題，請勿在未呼叫此工具的情況下嘗試回答。」

**選擇準確性與工具數量：**

隨著可用工具數量增加，選擇準確性會下降。工具集較小時，模型能精確推理每個工具的適用範圍。工具集較大時，相關工具獲得的關注減少，描述重疊的問題也更難區分。緩解策略：
- **子集選擇：** 只呈現與當前任務相關的工具
- **動態工具載入 (dynamic tool loading)：** 使用分類器或嵌入搜尋，根據使用者請求選擇要包含哪些工具
- **工具路由層 (tool routing layer)：** 第一次呼叫先將請求路由至正確的工具子集

- 工具描述 MUST（必須）在同一工具集中存在可能混淆的其他工具時，明確說明何時不應使用該工具。
- 超過 10 個工具的工具集 SHOULD（應該）使用動態工具載入或路由層，而非同時呈現所有工具。
- 工具選擇準確性 MUST（必須）在部署前針對具代表性的生產輸入樣本進行測試。描述品質無法僅靠閱讀 schema 來驗證。

## 深入探討

### 測試工具選擇

工具選擇是可測試的。建立一個測試套件，驗證針對一組具代表性的輸入，哪個工具會被選中：

```python
import anthropic

client = anthropic.Anthropic()

def test_tool_selection(tools: list[dict], test_cases: list[dict]) -> dict:
    """
    tools: list of tool definitions (name, description, input_schema)
    test_cases: list of {"input": str, "expected_tool": str | None}
      expected_tool is None when no tool should be called
    Returns: {"total": int, "passed": int, "pass_rate": float, "failures": list}
    """
    results = {"total": len(test_cases), "passed": 0, "failures": []}

    for case in test_cases:
        response = client.messages.create(
            model="claude-opus-4-6",
            max_tokens=256,
            tools=tools,
            messages=[{"role": "user", "content": case["input"]}]
        )

        selected_tool = None
        for block in response.content:
            if block.type == "tool_use":
                selected_tool = block.name
                break

        if selected_tool == case["expected_tool"]:
            results["passed"] += 1
        else:
            results["failures"].append({
                "input": case["input"],
                "expected": case["expected_tool"],
                "got": selected_tool
            })

    results["pass_rate"] = results["passed"] / results["total"] if results["total"] > 0 else 0.0
    return results
```

以兩個描述重疊的工具為例：

```python
tools = [
    {
        "name": "search_web",
        "description": "Search the web for current information. Use for recent events, current prices, or information that changes over time. Do NOT use for questions about internal company policies or documents.",
        "input_schema": {
            "type": "object",
            "properties": {
                "query": {"type": "string", "description": "The search query."}
            },
            "required": ["query"]
        }
    },
    {
        "name": "search_documents",
        "description": "Search internal company documents and knowledge base. Use for questions about company policies, procedures, products, and internal processes. Do NOT use for general web search or current events.",
        "input_schema": {
            "type": "object",
            "properties": {
                "query": {"type": "string", "description": "The search query."}
            },
            "required": ["query"]
        }
    }
]

test_cases = [
    {"input": "What's the current price of gold?", "expected_tool": "search_web"},
    {"input": "What is our company's refund policy?", "expected_tool": "search_documents"},
    {"input": "What is the capital of France?", "expected_tool": None},
    {"input": "What were our Q3 sales numbers?", "expected_tool": "search_documents"},
    {"input": "Who won the championship last night?", "expected_tool": "search_web"},
]

results = test_tool_selection(tools, test_cases)
print(f"Pass rate: {results['pass_rate']:.0%}")
for failure in results["failures"]:
    print(f"FAIL: '{failure['input']}' → expected {failure['expected']}, got {failure['got']}")
```

每次修改工具名稱或描述後，都應執行此測試。

### 描述反模式

**反模式一：鏡像描述**

```
search_web:       "Search for information on the web"
search_documents: "Search for information in documents"
```

兩個描述都說「搜尋資訊」——模型無法判斷哪個適用於給定的查詢。加入區分上下文：哪個 web、哪些文件、哪類查詢屬於各自範疇。

**反模式二：缺少觸發條件**

```
get_stock_price: "Gets the current stock price"
```

這無法告訴模型何時應呼叫它。若使用者問「蘋果公司的股價是多少？」，模型可能從訓練資料回答（資料已過時）。修正方式：加入「每當問題需要當前或即時股價時，使用此工具。不要嘗試從訓練資料回答股價問題——股價隨時變動。」

**反模式三：過度寬泛的包含範圍**

```
answer_question: "Answers any question the user asks"
```

「任何問題」匹配所有輸入。模型會對所有事情都呼叫此工具——包括應路由至更具體工具的查詢。絕對不要在工具觸發條件中使用「任何」。

## 最佳實踐

1. **在撰寫描述之前，先寫好選擇的測試案例。** 列出「應該」觸發工具的輸入，以及「不應該」觸發的輸入。這在撰寫描述之前就強制釐清了工具的適用範圍，並在描述撰寫完成後立即提供測試套件。

2. **每次修改描述後都執行選擇測試套件。** 修正一個失效的變更可能引入另一個。選擇失效往往具有非局部性——更改一個工具的描述可能影響工具集中其他工具的選擇行為。

3. **在新增指令之前先診斷失效原因。** 當選擇失敗時，直覺是在系統提示中加入更多指令。這只是增加雜訊，無法解決根本原因。先診斷：是缺少觸發條件（加入「使用時機」）、描述重疊（加入「不應使用的時機」），還是包含範圍過廣（縮窄範圍）？
