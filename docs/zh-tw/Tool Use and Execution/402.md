# [AEE-402] 工具結構描述設計

## 情境

工具結構描述 (tool schema) 是模型用來判斷是否呼叫某個工具、以及傳遞哪些引數的依據。`description` 欄位不是給人類閱讀的文件——它是模型選擇與呼叫工具時最主要的訊號。參數描述 (parameter description) 不是可有可無的附加資訊——它是模型決定每個欄位應填入什麼值的指令。

大多數工具呼叫失敗的根源在於結構描述品質，而非模型能力。當模型呼叫了錯誤的工具、傳入不正確的引數型別，或遺漏了必要參數 (required parameter)，幾乎都是在回應一份模糊或不完整的結構描述，而不是工具使用本身的能力問題。

## 設計思維

核心主張：工具結構描述是一種自然語言介面，而不只是型別契約。模型透過參數描述來決定何時、如何呼叫工具——描述寫得不好，造成誤呼叫的機率遠高於型別不符。

**工具名稱 (tool name) 設計：**

工具名稱是工具選擇時的第一個訊號，應當：
- 以動作為導向：`search_web`、`create_calendar_event`、`get_user_profile`——而非 `tool1`、`calendar`、`user`
- 足夠明確以區分類似工具：`search_web` vs. `search_knowledge_base` vs. `search_documents` 清楚表達各自的適用範圍
- 在同一組工具中保持一致的命名慣例：全部使用 snake_case；全部動詞優先

工具名稱必須符合正規表示式 `^[a-zA-Z0-9_-]{1,64}$`——只允許字母、數字、底線與連字號，最長 64 個字元。

**工具描述 (tool description) 品質：**

工具描述必須回答三個問題：
1. 這個工具做什麼？
2. 什麼時候應該使用它？
3. 什麼時候**不應該**使用它？

第三個問題最常被省略，也最常被需要。在多工具環境中，模型必須區分功能重疊的工具。少了負向引導，它就無法做到。

**參數描述：**

每個參數都必須有描述，說明：
- 該值代表什麼（不只是它的名稱）
- 預期格式（ISO 8601 日期、URL、逗號分隔清單）
- 有效範圍或限制（最多 200 個字元、必須為正整數）
- 邊界情況的行為（「傳入 null 以取得所有紀錄；省略則使用呼叫者的預設值」）

**型別選擇：**

- 當有效值為固定集合時，使用列舉型別 (enum type) 而非 `string`。模型會嚴格在列舉值範圍內作答。
- 是非欄位使用 `boolean` 而非 `string`。
- 當扁平參數就足夠時，避免深層巢狀物件。扁平結構讓模型更容易正確呼叫。
- 當工具無法在缺少某參數的情況下運作時，將其標記為必要參數。在描述中記錄選填參數的預設值。

**RFC 2119：**

- 每個工具參數 MUST（必須）有描述。沒有描述的參數會迫使模型猜測其用途——而猜測結果可能與工具的實際行為不符。
- 當同一組工具中有其他工具可能被混淆時，工具描述 MUST（必須）說明何時不應使用該工具。
- 工具名稱 MUST（必須）以動作為導向且具體明確。通用名稱（`tool1`、`helper`、`process`）是結構描述設計的失敗。
- 當參數的有效值為已知固定集合時，SHOULD（應該）使用列舉型別 (enum type)。

## 深入探討

### 結構描述剖析

以下是一個完整的文件搜尋工具定義：

```json
{
  "name": "search_documents",
  "description": "Search the internal knowledge base for relevant documents. Use when the user asks about company policies, product documentation, or internal processes. Do NOT use for general web search or external information — use search_web for those queries.",
  "input_schema": {
    "type": "object",
    "properties": {
      "query": {
        "type": "string",
        "description": "The search query using key terms from the user's question. Avoid filler words. Maximum 500 characters."
      },
      "document_type": {
        "type": "string",
        "enum": ["policy", "product", "process", "all"],
        "description": "Filter by category. Use 'policy' for HR/compliance documents, 'product' for technical specs, 'process' for workflows. Use 'all' when the category is unclear."
      },
      "max_results": {
        "type": "integer",
        "description": "Maximum number of documents to return. Must be between 1 and 20. Defaults to 5 if not specified."
      }
    },
    "required": ["query"]
  }
}
```

這份結構描述之所以有效：
- 工具名稱將其與 `search_web` 區分（內部 vs. 外部）
- 描述同時說明何時使用以及何時**不該**使用
- `query` 指定了格式與長度限制
- `document_type` 使用列舉型別，並為每個值記錄語意
- `max_results` 指定有效範圍與預設值

### 實例：重新設計一個有問題的結構描述

**原始結構描述（導致呼叫失敗）：**

```json
{
  "name": "calendar",
  "description": "Calendar tool",
  "input_schema": {
    "type": "object",
    "properties": {
      "action": {
        "type": "string"
      },
      "data": {
        "type": "object"
      }
    },
    "required": ["action", "data"]
  }
}
```

失敗模式：
- `name: "calendar"` 是名詞而非動作——不清楚呼叫這個工具會做什麼
- `description: "Calendar tool"` 完全沒有提供工具選擇的引導
- `action` 沒有描述也沒有列舉型別——模型必須猜測有效值
- `data` 是未定型的物件——模型完全不知道要傳入什麼結構
- 兩個參數都是必要參數，卻沒有任何說明

**重新設計為三個具體工具：**

```json
[
  {
    "name": "create_calendar_event",
    "description": "Create a new event on the user's calendar. Use when the user asks to schedule a meeting, add a reminder, or block time. Do NOT use to modify or delete existing events.",
    "input_schema": {
      "type": "object",
      "properties": {
        "title": {
          "type": "string",
          "description": "Event title. Keep concise (under 100 characters)."
        },
        "start_time": {
          "type": "string",
          "description": "Event start time in ISO 8601 format with timezone offset (e.g., '2026-04-15T14:00:00-07:00')."
        },
        "end_time": {
          "type": "string",
          "description": "Event end time in ISO 8601 format. Must be after start_time."
        },
        "attendees": {
          "type": "array",
          "items": { "type": "string" },
          "description": "Email addresses of attendees to invite. Pass an empty array or omit for personal reminders."
        }
      },
      "required": ["title", "start_time", "end_time"]
    }
  },
  {
    "name": "list_calendar_events",
    "description": "Retrieve upcoming events from the user's calendar. Use when the user asks what is scheduled, when they are free, or wants to review their calendar. Do NOT use to create or modify events.",
    "input_schema": {
      "type": "object",
      "properties": {
        "start_date": {
          "type": "string",
          "description": "Start of the date range to query, in YYYY-MM-DD format."
        },
        "end_date": {
          "type": "string",
          "description": "End of the date range, in YYYY-MM-DD format. Defaults to start_date + 7 days if omitted."
        }
      },
      "required": ["start_date"]
    }
  },
  {
    "name": "delete_calendar_event",
    "description": "Permanently delete a calendar event by ID. Use only when the user explicitly asks to delete or cancel a specific event. Do NOT use to modify event details — deletion is irreversible. Obtain event_id from list_calendar_events first.",
    "input_schema": {
      "type": "object",
      "properties": {
        "event_id": {
          "type": "string",
          "description": "The unique identifier of the event. Obtain this from list_calendar_events before calling."
        }
      },
      "required": ["event_id"]
    }
  }
]
```

改動說明：
- 三個具體工具取代一個通用工具——每個都有明確的動作與獨立的適用範圍
- 每個參數都有包含格式與限制的描述
- `start_time` 與 `end_time` 明確指定 ISO 8601 格式
- `delete_calendar_event` 說明「刪除操作不可逆」——降低意外執行破壞性呼叫的風險
- 每個工具都有明確的「Do NOT use」子句

## 最佳實踐

1. **先寫「何時不應使用」子句，再寫描述主體。** 在多工具環境中，負向引導才是模型區分功能重疊工具時真正需要的東西。先找出可能與這個工具混淆的相鄰工具，為每個寫好排除子句，再填寫正向引導。

2. **用不應觸發此工具的輸入來測試結構描述。** 完成描述後，發送幾個不應呼叫此工具的請求，確認模型不會誤觸。過於寬泛的描述會吸引相鄰查詢，導致誤呼叫。

3. **以多個具體工具取代帶有 action 參數的單一通用工具。** 使用單一工具搭配 `action` 字串參數的模式，會破壞模型的選擇機制。模型依據描述來選擇工具——提供各自有獨立描述的具體工具，比起一個沒有說明 `action` 欄位的單體工具，能產生更可靠的行為。
