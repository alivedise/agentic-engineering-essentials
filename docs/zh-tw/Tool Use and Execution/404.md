# [AEE-404] 瀏覽器與電腦使用

## 情境

瀏覽器自動化 (browser automation) 與電腦使用 (computer use) 讓 agent 能透過為人類設計的介面與世界互動。函式呼叫是呼叫工程師自己定義的 API；瀏覽器自動化與電腦使用則是操作工程師未曾建構、也無法控制的介面：網站、桌面應用程式、表單與 GUI。這項能力填補了結構化 API 無法觸及的缺口——遺留系統、第三方網站，以及僅提供人機介面的系統。

代價是爆炸半徑 (blast radius)。函式呼叫工具只能執行實作中允許的操作。一個能夠點擊、輸入與瀏覽的電腦使用 agent，可以做到任何人類使用者能做的事——包括破壞性、不可逆，或超出預期範圍的操作。這類工具類型的安全架構不是選項，而是前提。

## 設計思維

核心主張：瀏覽器自動化與電腦使用讓 agent 能透過為人類設計的介面與世界互動——但它們的爆炸半徑 (blast radius) 遠高於函式呼叫。部署前必須明確建立安全架構。

**瀏覽器自動化 vs. 電腦使用：**

| | 瀏覽器自動化 | 電腦使用 |
|---|---|---|
| 介面 | 程式化（DOM） | 視覺化（截圖 + 動作） |
| 可靠性 | 高——選擇器具確定性 | 較低——取決於截圖品質 |
| 範圍 | 僅限網頁 | 任何可見應用程式 |
| Token 成本 | 低（無截圖） | 高（每個動作需截圖） |
| 適用情境 | HTML 結構穩定時 | 遺留應用程式、非網頁 GUI |

**截圖-動作循環 (screenshot-action loop)：**

電腦使用以感知-動作循環 (perception-action loop) 的方式運作：
1. 執行環境對當前畫面狀態截圖
2. 截圖以圖片內容區塊的形式傳送給模型
3. 模型決定下一個動作：點擊、輸入、捲動、按鍵或截圖
4. 執行環境執行該動作
5. 拍攝新截圖，循環繼續
6. 當模型輸出文字回應表示任務完成時，循環結束

每次迭代都會傳送一張 base64 編碼的完整截圖——依解析度不同，通常需要數百至數千個 token。需要大量畫面互動的任務可能產生可觀的 token 成本。部署前務必預估費用。

**不應使用瀏覽器/電腦使用的情境：**

- 當穩定的 API 存在時：改用函式呼叫。對有可用 API 的網站使用電腦使用，是不必要的複雜度。
- 當目標介面頻繁變更時：DOM 選擇器與點擊座標 (coordinates) 會在 UI 更新後失效。
- 在沒有隔離環境的生產環境中：兩種工具類型都必須在隔離環境中運行。

**RFC 2119：**

- 電腦使用 agent MUST（必須）在與主機系統隔離的環境（VM、容器）中運行。在不受控制的桌面環境中執行電腦使用 agent，意味著該 agent 可存取畫面上可見的所有應用程式、憑證與檔案。
- 瀏覽器自動化 SHOULD（應該）限定在特定網域，而非授予無限制的網路存取權。能夠任意瀏覽的 agent，可能被導向專門透過頁面內容執行提示詞注入的網站。
- 電腦使用任務的 token 成本 MUST（必須）在投入生產部署前預估。需要大量畫面互動的任務具有相當可觀的 token 成本，僅從任務描述往往無法察覺。

## 深入探討

### 使用 Playwright 進行瀏覽器自動化

Playwright 提供對 Chromium、Firefox 和 WebKit 的程式化控制，是目標網站具有穩定 HTML 結構時進行結構化網頁自動化的正確選擇。

```python
from playwright.sync_api import sync_playwright

def extract_product_prices(url: str) -> list[dict]:
    with sync_playwright() as p:
        browser = p.chromium.launch(headless=True)
        page = browser.new_page()
        page.goto(url)
        page.wait_for_load_state("networkidle")

        products = page.query_selector_all(".product-card")
        results = []
        for product in products:
            name = product.query_selector(".product-name").inner_text()
            price = product.query_selector(".price").inner_text()
            results.append({"name": name, "price": price})

        browser.close()
        return results
```

Playwright 的優勢：
- 確定性：當 HTML 結構穩定時，CSS 選擇器具有穩定性
- 速度快：無截圖開銷；DOM 查詢直接執行
- 結構化擷取：回傳有型別的資料，而非原始文字
- 網路攔截：可模擬或攔截請求

### 使用 browser-use 進行 Agent 友善的自動化

browser-use 是建構在 Playwright 之上的函式庫，為 LLM 驅動的網頁瀏覽提供更高層次的介面，負責將自然語言任務描述轉譯為瀏覽器動作。

```python
from browser_use import Agent
from langchain_anthropic import ChatAnthropic

agent = Agent(
    task="Find the current price of the MacBook Pro 16-inch on apple.com",
    llm=ChatAnthropic(model="claude-opus-4-6"),
)

result = await agent.run()
print(result)
```

當任務需要多步驟瀏覽並包含決策邏輯，而非固定的擷取模式時，browser-use 是合適的選擇。

### Anthropic 電腦使用

Anthropic 的電腦使用能力讓 Claude 能透過截圖與動作與完整的桌面環境互動。它使用標準訊息 API 搭配 beta 標頭。

可用的動作類型：
- `screenshot`：擷取當前畫面（用於觀察）
- `left_click`、`right_click`、`double_click`：在指定的螢幕座標 (coordinates) 點擊
- `type`：輸入一段文字
- `key`：按下鍵盤按鍵或組合鍵（例如 `"ctrl+c"`）
- `scroll`：在指定位置按指定量捲動
- `mouse_move`：移動游標而不點擊

```python
import anthropic
import base64

client = anthropic.Anthropic()

def computer_use_loop(task: str, take_screenshot, execute_action) -> str:
    """
    take_screenshot: () -> base64-encoded PNG string
    execute_action: (dict) -> None
    """
    # Initial message with task and first screenshot
    initial_screenshot = take_screenshot()
    messages = [
        {
            "role": "user",
            "content": [
                {"type": "text", "text": task},
                {
                    "type": "image",
                    "source": {
                        "type": "base64",
                        "media_type": "image/png",
                        "data": initial_screenshot
                    }
                }
            ]
        }
    ]

    while True:
        response = client.beta.messages.create(
            model="claude-opus-4-6",
            max_tokens=1024,
            tools=[{
                "type": "computer_20251124",
                "name": "computer",
                "display_width_px": 1280,
                "display_height_px": 800
            }],
            messages=messages,
            betas=["computer-use-2025-11-24"]
        )

        messages.append({"role": "assistant", "content": response.content})

        if response.stop_reason == "end_turn":
            for block in response.content:
                if block.type == "text":
                    return block.text

        # Execute all tool use actions and capture resulting screenshots
        tool_results = []
        for block in response.content:
            if block.type == "tool_use" and block.name == "computer":
                execute_action(block.input)
                # Include screenshot of the screen state AFTER the action
                post_action_screenshot = take_screenshot()
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": [
                        {
                            "type": "image",
                            "source": {
                                "type": "base64",
                                "media_type": "image/png",
                                "data": post_action_screenshot
                            }
                        }
                    ]
                })

        messages.append({"role": "user", "content": tool_results})
```

**安全設定：** 電腦使用 agent 必須在隔離環境中運行。建議的做法是使用 Docker 容器或 VM，且不可存取主機系統的檔案、憑證或應用程式。Anthropic 的電腦使用文件提供了參考用的 Docker 設定。

## 最佳實踐

1. **當目標網站有穩定 HTML 時，優先選擇 Playwright 而非電腦使用。** Playwright 速度更快、成本更低、可靠性更高，也更容易除錯。電腦使用是在結構化自動化不可行時的正確選擇——而不是通用的瀏覽器策略。

2. **將瀏覽器自動化限定在最小必要的網域範圍內。** 將 Playwright 設定為只瀏覽特定來源，可降低透過頁面內容進行提示詞注入的風險。能夠任意瀏覽的 agent，可能被導向專門用來覆寫其指令的頁面。

3. **在無法存取主機憑證的容器中執行電腦使用。** 最常見的部署錯誤，是在測試期間於開發者的機器上執行電腦使用，然後在未更改隔離設定的情況下移至生產環境。測試時請使用與生產環境相同的隔離設定。
