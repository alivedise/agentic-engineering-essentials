# [AEE-403] 程式碼執行 (Code Execution)

## 情境

程式碼執行 (code execution) 是 agent 工具箱中能力最強、也最危險的工具類型。它能夠執行任意計算：資料分析、演算法執行、動態邏輯，以及生成並運行程式碼來解決模型無法在上下文中直接解決的問題。但它同時也執行來自一個無法完全信任的來源的任意程式碼——也就是模型的輸出。這兩個事實密不可分。

工程師在沒有沙箱 (sandbox) 的情況下為 agent 加入程式碼執行功能，並不是在做效能上的取捨。他們實際上是在賦予模型在應用程式進程中執行任意程式碼的能力——使用該進程的權限，存取該進程的檔案系統與網路。

## 設計思維

核心主張：程式碼執行是能力最強、也最危險的工具類型。它可以執行任意計算，但必須搭配沙箱——在不受控制的環境中執行模型生成的程式碼是嚴重的安全失敗。

**程式碼執行 vs. 函式呼叫：**

兩者都是工具類型，差異在於特定性：
- **函式呼叫**呼叫一個已知的、預先定義的操作，具有明確的 schema。模型指定的是工程師撰寫並審查過的函式參數。
- **程式碼執行**呼叫一個直譯器，執行模型生成的任意程式碼。模型指定的是程式碼本身——而這段程式碼可以做直譯器允許的任何事情。

當操作定義明確且有界時，使用函式呼叫。當任務需要無法以固定函式表達的動態計算時，使用程式碼執行：例如對未知結構輸入的資料轉換、在執行時才陳述的問題的演算法執行，或模型無法在上下文中完成的數學計算。

**沙箱的必要性：**

沙箱環境 (sandboxed environment) 強制執行程式碼與主機系統之間的隔離。沙箱至少必須強制執行：
- **檔案系統隔離 (filesystem isolation)**：程式碼只能讀寫指定路徑；無法存取主機檔案系統
- **網路隔離 (network isolation)**：封鎖或限制對外連線至定義的允許清單；程式碼無法外洩資料
- **進程隔離 (process isolation)**：程式碼在與主機應用程式獨立的進程或容器中執行；無法存取主機進程 (host process) 的記憶體或憑證
- **資源限制 (resource limits)**：CPU 時間、記憶體與執行逾時有明確上限；失控程式碼無法耗盡主機資源

問題不在於是否要使用沙箱——而在於哪個提供者適合該使用情境。

**RFC 2119：**

- 模型生成的程式碼 MUST NOT（絕不能）在沙箱環境之外執行。在主機進程中執行模型輸出，無論提示詞如何約束，都是嚴重的安全失敗。
- 沙箱 MUST（必須）強制執行資源限制，包括 CPU 時間、記憶體、執行逾時以及網路存取政策。沒有資源限制的沙箱不能稱之為沙箱。
- 程式碼執行工具 MUST（必須）以結構化輸出的形式回傳標準輸出 (stdout)、標準錯誤 (stderr) 與退出代碼 (exit code)。模型需要完整的執行上下文才能判斷是否要重試或回報失敗。

## 深入探討

### 提供者選項

**E2B（雲端沙箱）**

E2B 提供按需建立的隔離沙箱環境來執行程式碼。每個沙箱在一個隔離容器中運行，配備 Python 或 JavaScript 執行環境、隔離的檔案系統，以及可設定的網路政策。

```python
import e2b_code_interpreter as e2b

sandbox = e2b.Sandbox()

execution = sandbox.run_code("""
import pandas as pd
import json

data = [{"name": "Alice", "score": 92}, {"name": "Bob", "score": 78}]
df = pd.DataFrame(data)
result = df.sort_values("score", ascending=False).to_dict(orient="records")
print(json.dumps(result))
""")

print(execution.text)   # stdout
print(execution.error)  # stderr (if any)
sandbox.kill()
```

E2B 特性：
- 網路：可設定；預設允許對套件倉庫的 HTTPS 外連
- 持久性：沙箱在同一 session 中持久存在，並在同一 session 的多個程式碼區塊間共享狀態
- 費用：按實際計算秒數計費

**Modal（無伺服器容器）**

Modal 以可設定的 CPU、記憶體和選用 GPU 按需執行容器化工作負載。對於 agent 中的程式碼執行，Modal 提供具有強隔離性的短暫函式執行。

```python
import modal

app = modal.App()

@app.function(
    cpu=1,
    memory=512,
    timeout=30,
    network_file_systems={}  # no persistent storage
)
def execute_code(code: str) -> dict:
    import subprocess, sys
    result = subprocess.run(
        [sys.executable, "-c", code],
        capture_output=True, text=True, timeout=25
    )
    return {
        "stdout": result.stdout,
        "stderr": result.stderr,
        "returncode": result.returncode
    }

with app.run():
    output = execute_code.remote("print(2 + 2)")
```

Modal 特性：
- GPU 支援：可用於需要 GPU 計算的機器學習工作負載
- 容器映像：完全可設定；可安裝任意套件
- 隔離性：每次呼叫都是全新容器，除非明確設定持久性
- 預設逾時：5 分鐘；最長 24 小時

**Docker（自行託管）**

對於無法將資料送至外部提供者的團隊，Docker 容器可在自行託管的基礎架構上提供相同的隔離效果：

```python
import docker

client = docker.from_env()

def run_in_sandbox(code: str, timeout: int = 10) -> dict:
    try:
        output = client.containers.run(
            "python:3.11-slim",
            command=["python", "-c", code],
            remove=True,
            mem_limit="128m",
            cpu_quota=50000,       # 50% of one CPU
            network_disabled=True,
            read_only=True,
            tmpfs={"/tmp": "size=64m"},
            timeout=timeout
        )
        return {"stdout": output.decode("utf-8"), "returncode": 0}
    except docker.errors.ContainerError as e:
        return {"stdout": "", "stderr": str(e), "returncode": e.exit_status}
```

根據您的任務調整 `cpu_quota` 和 `mem_limit` — 50,000（單一 CPU 的 50%）和 128MB 記憶體是適合不信任程式碼的保守預設值。資料分析工作負載可能需要更高的限制。

關鍵 Docker 參數：
- `--network=none` / `network_disabled=True`：封鎖所有網路存取
- `mem_limit`：強制記憶體上限
- `cpu_quota`：限制 CPU 使用量
- `--read-only`：根檔案系統為唯讀
- `tmpfs /tmp`：可寫入的記憶體內暫存目錄

### 將程式碼執行定義為工具

在函式呼叫協定中定義程式碼執行：

```json
{
  "name": "execute_python",
  "description": "Execute Python code in a sandboxed environment and return the output. Use for data analysis, mathematical computation, algorithm execution, or any task requiring dynamic computation. The code runs in an isolated container. Network access is restricted — external connections are blocked except to package registries. Only use when the computation cannot be expressed as a direct answer.",
  "input_schema": {
    "type": "object",
    "properties": {
      "code": {
        "type": "string",
        "description": "Valid Python code to execute. Print results to stdout — expression return values are not captured. Available packages: numpy, pandas, matplotlib. Maximum 10 seconds execution time."
      }
    },
    "required": ["code"]
  }
}
```

### 輸出處理

以結構化工具結果的形式回傳完整執行上下文：

```json
{
  "type": "tool_result",
  "tool_use_id": "toolu_01ABC",
  "content": "stdout:\n[{\"name\": \"Alice\", \"score\": 92}, {\"name\": \"Bob\", \"score\": 78}]\n\nstderr:\n(empty)\n\nexit_code: 0"
}
```

若執行失敗：

```json
{
  "type": "tool_result",
  "tool_use_id": "toolu_01ABC",
  "is_error": true,
  "content": "stdout:\n(empty)\n\nstderr:\nNameError: name 'df' is not defined\n\nexit_code: 1"
}
```

回傳 stdout、stderr 與 exit_code，讓模型有足夠的上下文來決定是否要以修正後的程式碼重試。

## 最佳實踐

1. **在工具描述中明確列出可用套件清單。** 模型需要知道它可以 import 哪些套件。描述中寫明「可用套件：numpy、pandas、matplotlib、json」可以防止模型嘗試 import 沙箱中未安裝的套件，避免不必要的 NameError。

2. **無狀態任務使用全新沙箱；多步驟 session 則重複使用沙箱。** 跨多次工具呼叫持久保存狀態的沙箱，適合用於每個步驟都建立在上一步結果之上的多步驟資料分析。對於獨立任務，每次呼叫使用全新沙箱，以避免不同呼叫之間的狀態洩漏。

3. **在 stdout 之外一併回傳 exit code。** 只收到 stdout 的模型無法區分「執行完成但無輸出」與「靜默失敗」之間的差異。始終回傳 exit code。當 exit code 非零時，附上 stderr。
