# [AEE-109] LLM 的運作方式——從業者的心智模型

## Context

缺乏 LLM 運作心智模型的工程師會做出錯誤的架構決策。他們會在模型根本無法可靠執行的任務上過度信任模型，在模型處理良好的任務上不信任模型，並且在根本原因是模型的結構性特性時，將失敗歸咎於應用程式代碼中的 bug。正確的心智模型不需要深厚的 ML 知識——但確實需要理解 LLM 如何生成輸出的五個基礎屬性。本文提供了這個最小可行模型，針對推理 agent 行為、失敗模式和設計取捨而校準。

## Design Think

核心主張：LLM 是具有固定 context window 的自回歸 token 預測器。理解這一句話就能解釋從業者在生產中遇到的大多數 agent 失敗模式。

**1. Token，不是單詞。**

LLM 對 token 進行操作——這是在模型看到任何文字之前，由分詞演算法產生的子詞單元。平均而言，一個 token 對應約 0.75 個英文單詞，但比例並不均勻。非英文語言、原始碼、數學符號和特殊字符通常以顯著不同的密度進行分詞。CJK 字符（中文、日文、韓文）每個字符產生的 token 數通常比英文文字多。成本、context window 限制和每次請求的延遲都以 token 計量，而非字符或單詞。以字符數估計 context 使用量的工程師將產生系統性的錯誤估計。

**2. 自回歸生成。**

每個輸出 token 都被預測為模型詞彙表上的概率分佈，以 context 中所有前置 token 為條件（提示加上已生成的任何 token）。模型從左到右逐個 token 生成。在生成開始之前沒有全局規劃過程，沒有修訂機制，也沒有對已生成輸出的重新讀取。模型按順序提交每個 token 且無法修訂它。這就是為什麼模型可以看似自信地開始一個句子，然後在四個 token 後產生矛盾——後面的 token 是從前面的內容預測的，而不是來自獨立驗證的聲明。

**3. Context window 是模型唯一的記憶。**

API 調用之間沒有持久狀態。模型在推理時知道的所有內容必須存在於輸入 token 中。之前的對話、用戶偏好、先前調用的中間推理，以及任何不在 context 中的世界知識，都對模型不可見，除非明確包含在內。被給予 128K token context window 的模型，每次推理調用恰好有 128K token 的記憶——僅此而已。

**4. 注意力，不是檢索。**

模型不在資料庫中查找事實或遍歷知識圖譜。在推理期間，注意力層計算當前 context 中 token 之間的加權關係，前饋層應用從訓練資料中學習的轉換。當模型產生事實聲明時，它是在生成給定訓練語料庫統計上最可能的延續——而不是在檢索已驗證的事實。這就是為什麼模型可以產生流暢、詳細且完全錯誤的事實聲明：該機制優化的是 token 概率，而非真實性。

**5. 模型無法推理它不知道自己不知道的事情。**

模型沒有內建的不確定性信號。它沒有元數據層來追蹤哪些輸出是高置信度的，哪些是推測性的。無論模型對聲明的訓練資料基礎是否充分，它都會生成流暢、格式相似的輸出。聽起來自信的輸出是生成過程的屬性，而非可靠性信號。一個錯誤的模型聽起來與一個正確的模型完全相同。

**RFC 2119：**

- 設計 agentic 系統的工程師 MUST 內化這個心智模型。上述五個屬性——分詞、自回歸、受 context 限制的記憶、基於注意力的預測和缺失的不確定性信號——直接解釋了主導生產 agentic 系統失敗的幻覺、context 溢出、不一致行為和意外延遲問題。
- 工程師在估計成本和 context window 使用量時 MUST 計算 token，而非字符。基於字符的估計系統性地不準確。
- 工程師 SHOULD NOT 將流暢、自信的模型輸出解讀為正確性的證據。輸出風格與輸出準確性無關。

## 深入探討

### Transformer 架構一段話

輸入文字首先被分詞為整數 ID 序列。每個 ID 被映射到高維嵌入向量。Transformer 層堆疊然後處理完整序列：在每一層中，多頭自注意力為每個 token 位置計算所有其他 token 位置值的加權和——權重由學習的查詢-鍵點積決定。這讓每個 token 能「注意」context 中的每個其他 token。注意力之後，前饋子層應用位置式學習轉換。在所有層之後，最後一個 token 位置的輸出通過線性層和 softmax 投影，產生完整詞彙表上的概率分佈。模型從這個分佈中採樣以產生下一個 token，將其附加到 context，並重複。這個順序過程就是自回歸循環。提出完整 transformer 架構的奠基論文是 Vaswani 等人的 "Attention Is All You Need"（arXiv 1706.03762，NeurIPS 2017），截至 2025 年已積累超過 173,000 次引用。

### 從業者的分詞

現代 LLM 的主流分詞演算法是字節對編碼（Byte Pair Encoding，BPE）。BPE 通過從單個字節或字符的詞彙表開始，並迭代地將最頻繁出現的相鄰對合並為單個新 token 來工作。這個過程重複，直到達到目標詞彙表大小（通常為 50,000–100,000 個 token）。結果是一個子詞單元詞彙表——常見的英文單詞成為單個 token，較罕見的單詞被拆分為兩三個子詞 token，非常罕見的字符串被拆分到單個字節。

實際工程啟示：

- 相同的單詞可能根據 context 以不同方式分詞（大小寫、換行後的位置、是否在空格後面）。
- 英文文字以大約每個單詞 1 個 token 的效率分詞。
- 代碼的分詞密度因語言和風格而異：長變量名、縮進重的代碼和具有複雜語法的語言可能比緊湊代碼中的等效邏輯產生顯著更多的 token。
- CJK 文字通常每個字符產生 1–2 個 token，相比英文常見單詞每約 4 個字符約 1 個 token。
- 數字和算術表達式的分詞不一致：數字「1234」可能根據分詞器和 context 被分詞為一個、兩個或四個 token。

Hugging Face LLM 課程提供了 BPE 的權威逐步說明，Karpathy 的 `minbpe`（GitHub）以約 200 行 Python 實現了完整演算法——對於從代碼中學習的工程師來說是最有用的資源。

### 為什麼幻覺會發生

幻覺不是傳統意義上的 bug。它是模型根據其訓練目標正確行為的結果。

在預訓練期間，模型被優化為在非常大的語料庫上預測給定前置 context 的下一個 token。該語料庫包含事實錯誤、聽起來自信的推測，以及在寫作時是真實但已變得不真實的陳述。模型學習以類似流暢度重現所有這些內容。它沒有機制來區分「這是我確信的事實」和「這是我生成的聽起來像事實的東西」。

Xu 等人的 "Hallucination is Inevitable: An Innate Limitation of Large Language Models"（arXiv 2401.11817）提供了理論基礎：因為 LLM 無法從有限的訓練資料中學習所有可計算函數，當用作通用問題求解器時，它們必然會產生不正確的輸出。這是結構性屬性，而非可修復的訓練工件。

arXiv 2510.06265 的 2025 年綜合調查將注意力機制弱點和能力不對齊識別為架構貢獻因素。另外，arXiv 2509.18970 記錄了幻覺如何在多步驟 agentic context 中累積：在 10 步推理鏈的第 2 步引入的錯誤被每個後續步驟攜帶和構建，將原始錯誤放大為大型最終錯誤。這種複合效應是為什麼 agentic 系統比單輪聊天機器人更容易受到幻覺損害，即使每步錯誤率相同。

## 最佳實踐

1. **計算 token，不是字符。** 在構建任何涉及 context window 的生產系統之前，整合分詞器庫（例如模型提供商發布的庫）以檢測實際 token 計數。基於字符的估計大幅低估 CJK 和代碼輸入，並產生不正確的成本和限制預測。

2. **不要從模型置信度或輸出流暢度推斷模型準確性。** 措辭的流暢性和置信度是生成過程的輸出，而非可靠性信號。在輸出將被採取行動的地方以程式方式測試正確性：運行斷言、執行生成的代碼、根據 schema 驗證結構化輸出。請參閱 AEE-102 了解可驗證獎勵在訓練自我檢查推理系統中的作用。

3. **在不失去精確性的情況下最小化提示。** 冗長的提示並不比簡潔的提示更安全或更可靠。不必要的長度消耗 context 容量，增加成本，並可能稀釋模型對重要指令的注意力。工程原則是：提示中的每個 token 都應該在那裡，因為刪除它會降低輸出品質。
