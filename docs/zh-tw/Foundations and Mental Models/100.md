# [AEE-100] 什麼是智能體？

## Context

「智能體」一詞在 AI 討論中被隨意使用——有時指帶有系統提示的簡單聊天機器人，有時指能自主運行數小時的完全自主系統。缺乏共同定義，工程師就無法清晰推理自己在建構什麼，也無法評估系統是否符合目的。

## Design Think

**智能體**是一個感知環境、做出決策、並採取行動以實現目標的系統——可能跨越多個步驟，使用文字生成之外的一個或多個工具或能力。

區別智能體與普通 LLM 呼叫的關鍵屬性：

1. **多步驟推理** -- 系統能規劃並執行一系列行動，而不僅僅是一次性回應
2. **工具使用** -- 系統能對世界採取行動（讀取檔案、呼叫 API、執行程式碼、瀏覽網頁）
3. **狀態** -- 系統在步驟之間維護上下文（對話歷史、記憶、中間結果）
4. **目標導向** -- 系統朝著目標行動，而不僅僅是回答問題

系統越自主地使用這些屬性，其「智能體性」就越強。聊天機器人具有狀態。RAG 系統具有工具使用。完全自主的程式碼撰寫智能體具備全部四項。

### 智能體分類

傳統 AI 智能體文獻區分三大類別：

- **反應式智能體（Reactive agents）** -- 無內部世界模型；直接對刺激做出回應（刺激-反應）。速度快、結構簡單，但無法對未來進行推理。
- **審議式智能體（Deliberative agents）** -- 維護顯式的世界模型；在行動前進行規劃。能跨越時間維度推理，但速度較慢，且需要準確的世界模型。
- **混合型智能體（Hybrid agents）** -- 大多數生產環境中的 LLM 智能體屬於此類。模型作為隱式的習得世界模型，規劃從上下文推理中浮現，而非依賴顯式符號邏輯。

### 邊界定義

> 在 AEE 定義下，一個系統**必須（MUST）**至少具備工具使用或持續性多步驟推理，才能被稱為智能體。單次 LLM 呼叫加上系統提示**不是（NOT）**智能體。

此 RFC 2119 邊界定義的目的是使工程討論保持精確。「我們正在建構智能體」應帶有具體的工程含義——執行框架設計、評估策略、狀態管理、權限模型——這些都不適用於單次呼叫的 LLM 整合。

## 深入探討

### 反應式 vs. 審議式智能體

傳統 AI 智能體文獻區分反應式智能體（對刺激做出回應，無內部世界模型）與審議式智能體（維護環境模型，行動前進行規劃）。LLM 智能體模糊了這一界限：模型作為習得的世界模型，規劃從提示中浮現，而非來自顯式符號推理。在實際生產中，大多數 LLM 智能體是混合型的：在每一步對工具觀測做出反應，同時利用模型的上下文推理進行輕量級審議。

### PEAS 框架

學術界的 PEAS 框架（Russell & Norvig）為描述任何智能體系統提供了形式化視角：

- **Performance（績效）** -- 定義成功的指標（通過的測試、完成的任務、用戶評分）
- **Environment（環境）** -- 智能體運行的世界（可用工具、外部 API、檔案系統）
- **Actuators（執行器）** -- 智能體如何對世界採取行動（工具呼叫、文字輸出、程式碼執行）
- **Sensors（感測器）** -- 智能體如何感知世界（輸入 token、工具結果、觀測值）

將 PEAS 應用於新的智能體設計可以發現缺口：沒有明確績效指標的智能體沒有定義成功的標準；沒有定義環境的智能體將面臨無邊界的範疇擴張。

### 「智能體程度」刻度盤

與其進行二元的智能體/非智能體分類，不如思考智能體程度——系統使用了多少核心屬性，以及自主程度如何：

| 系統 | 工具使用 | 多步驟 | 狀態 | 目標導向 | 智能體程度 |
|------|---------|-------|------|---------|---------|
| 單次 LLM 呼叫 | No | No | No | No | 極低 |
| RAG 管道 | Limited | No | No | No | 低 |
| 函式呼叫助手 | Yes | Limited | Session | Partial | 中 |
| ReAct 工具循環 | Yes | Yes | Task | Yes | 高 |
| 長時程程式碼智能體 | Yes | Yes | Persistent | Yes | 完整 |

此刻度盤比喻很重要，因為許多生產系統處於中間範圍。工程決策——執行框架設計、評估策略、權限模型——應與實際所需的智能體程度相符。

## 最佳實踐

1. **在設計執行框架之前，先確定智能體程度。** 只需要一次工具呼叫的系統不需要完整的 ReAct 循環與重新規劃。執行框架的複雜度應與智能體程度要求相符。
2. **為每個新的智能體設計定義 PEAS。** 未解答的 PEAS 組件就是設計缺口。如果無法定義績效，就無法評估智能體；如果無法定義環境，範疇就會意外擴張。
3. **狀態管理是最困難的部分。** 現代模型 API 對多步驟推理和工具使用的支援已相當完善。狀態——了解之前步驟發生了什麼、當前目標是什麼、已嘗試了什麼——才是大多數生產環境中智能體失敗的根源。
