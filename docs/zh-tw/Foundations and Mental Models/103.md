# [AEE-103] Agent vs. Chatbot

## Context

把 agent 當 chatbot 設計的工程師會低估系統需求：他們省略了 harness、跳過權限建模、沒有建立 eval 基礎設施，然後對系統產生無法稽核的副作用感到驚訝。把 chatbot 當 agent 設計的工程師則會過度工程化：他們為一個本質上是 request-response 的迴圈加上了 tool routing、狀態機和生命週期管理，增加了成本與脆弱性，卻沒有帶來任何好處。這個架構上的差異有充分的文獻依據：chatbot 是「無狀態預測」系統（request-response 迴圈），而 agent 則作為「有狀態推理」系統（感知-行動迴圈）運作，兩者的分水嶺在於工具使用加上跨步驟的持久狀態。在設計時正確進行這個分類，是一個團隊能做出的最高槓桿決策之一。

## Design Think

核心主張：agent 與 chatbot 之間的差異是**架構性的**，由可量測的具體屬性定義——而非由 UI、行銷標籤或品牌名稱決定。

**Chatbot** 是一個針對與人類文字交流而最佳化的系統。它每輪產生一個回應。它可能維護 session 記憶（對話歷史），但它不在外部系統中採取行動，也不被設計為執行多步驟計劃。其失敗模式有邊界：一個壞回應就是一個壞回應。

**Agent** 是一個行使以下一項或多項能力的系統：工具使用、多步驟推理、持久狀態管理或目標導向執行。關鍵標誌是能夠採取在系統外部產生效果的行動——寫入資料庫、呼叫 API、執行程式碼、發送訊息。arXiv 2503.12687 的框架很有用：「chatbot 提供協助，agent 提供勞動。」Chatbot 幫你思考；agent 代表你行動。

工程上的後果直接由此衍生：

- Chatbot 在 session 內 MAY 有記憶，但 MUST NOT 被設計為預期它會在外部系統上自主行動。
- Agent MUST 以適合其 agentness 程度的基礎設施來設計：包裹執行迴圈的 harness、限制 agent 可呼叫什麼的 tool routing、限制爆炸半徑的權限模型，以及驗證結果的 eval harness。
- 一個在 UI 上看起來像 chatbot、但進行自主 tool call 的系統，就**工程目的而言就是 agent**，MUST 相應地進行設計和評估——UI 呈現與架構分類無關。
- 工程師 MUST NOT 將 agent 基礎設施套用於真正的 chatbot：額外的複雜度是不必要的，且會在沒有解決任何實際問題的情況下產生新的失敗模式。

MarkTechPost 對各廠商定義的整合進一步明確了這點：chatbot 專注於「解釋問題並返回答案」，而 agent 則「跨越連接的系統將請求從意圖帶向結果」。從前者到後者的轉變不是漸進的——它是一個需要重新設計而非升級的相變。

## 深入探討

### 現代產品中的模糊地帶

許多產品在架構上是 agent，卻以「AI 助理」或「AI 副駕駛」自我行銷。一個預約日曆、提交支援票據或執行程式碼變更的產品就是 agent，無論其介面是否為聊天視窗。UI 呈現製造了 chatbot 的感知，而工程現實卻是 agent。這種不匹配是低度工程化部署的根本原因之一。

「停止建構 Chatbot」的實踐者分析從操作層面描述了這個差異：chatbot「由人類輸入觸發，使用 NLP/檢索產生回應」；agent「由目標或事件觸發，使用推理、規劃和工具使用產生結果」。觸發器和輸出類型都是不同的。

### 為什麼 UI 不決定架構類別

架構類別由系統**做什麼**決定，而非它如何呈現自己。一個呼叫單一 API 的命令列工具不是 agent。一個透過聊天介面協調五個 tool call、跨輪次維護目標並將結果寫入資料庫的系統是 agent。介面是一種渲染選擇；架構是系統責任的結果。

這很重要，因為工具、可觀測性和評估需求都遵循架構類別。Agent 需要對其 tool call 進行分散式追蹤、對外部行動的稽核日誌，以及針對代表性任務序列的迴歸測試。Chatbot 需要回應品質評估和延遲監控。對任一系統套用錯誤的工具集都會產生缺口。

### 轉型案例：增加工具使用的 Chatbot

考慮一個客戶服務 chatbot。在其原始形式中：它接收訊息、查找知識庫文章並返回建議回應。架構類別：chatbot。失敗模式：不好的建議。爆炸半徑：零——它無法採取行動。

團隊隨後添加了一個工具：自動發放最高 $50 退款的能力。系統現在進行外部狀態變更。架構類別：agent。失敗模式：不正確或欺詐性退款。爆炸半徑：真實且具有財務影響。

這個轉變需要刻意的重新設計——而非漸進式的變更。新系統需要：權限模型（哪些帳戶、哪些閾值）、稽核日誌（誰提出請求、發放了什麼、何時發放）、eval harness（它是否拒絕違反政策的請求？）以及速率限制（每個 session 多少退款）。這些對 chatbot 都不需要。對 agent 則全部必要。

## 最佳實踐

1. **依架構屬性分類，而非 UI。** 詢問：這個系統進行 tool call 嗎？它是否跨步驟維護狀態以達成目標？如果任一答案為是，它就是 agent。無論其呈現給使用者的方式如何，都將其歸類為 agent。
2. **不要對真正的 chatbot 套用 agent 基礎設施。** Harness、tool routing 和權限模型增加了複雜度。一個只產生文字回應的系統不需要它們，若以投機方式添加，維護起來只會更困難。
3. **將 chatbot 到 agent 的轉變視為架構里程碑，而非漸進式變更。** 當一個 chatbot 獲得第一個在外部系統上行動的工具時，停下來，以完整的 agent 基礎設施（harness、權限模型、稽核日誌、eval harness）重新設計系統，並在部署前重新審查其威脅模型。
