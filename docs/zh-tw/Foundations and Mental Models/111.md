# [AEE-111] Agentic 任務的模型選擇

## Context

為 agentic 任務選擇正確的模型並不明顯，選錯的後果會迅速複合。前沿模型昂貴且緩慢；較小的模型可能在需要多步驟推理或可靠工具使用的任務上失敗。對每項任務都默認使用最強大模型的工程師，在不需要前沿能力的子任務上每項任務多付 10-50 倍的費用，在用戶體驗到的關鍵路徑上累積延遲，並構建難以優化的系統，因為模型層級不是設計變量。模型選擇是一個具有成本、延遲、品質和可靠性後果的架構決策——而且可以在不犧牲品質的情況下進行優化。

## Design Think

核心主張：agentic 任務的模型選擇需要跨三個軸進行推理——能力、成本和延遲——並將它們與任務需求匹配。默認最強大的可用模型不是架構；它是架構的缺失。

**1. 能力軸。**

在選擇模型之前，將任務映射到其能力層級需求（完整的能力層級分類法請參閱 AEE-104）。作為工作啟發：

- **Tier 0–1（分類、提取、格式化、簡單摘要）：** 這些任務由較小、較便宜的模型良好服務。它們涉及對明確輸入 schema 的模式匹配，不需要多步驟推理或規劃。Tier 1 和前沿模型在這些任務上的品質差異很小；成本差異很大。
- **Tier 2（多步驟推理、工具使用、代碼生成）：** 這些任務需要跨多個步驟的可靠指令遵循，並從更強的模型中受益。這裡的錯誤會傳播——失敗的工具調用或不正確的中間結果可能級聯到後續步驟。
- **Tier 3+（目標分解、長期規劃、模糊的開放性任務）：** 前沿模型在這裡是合適的。成本因任務的複雜性和長執行序列中錯誤的複合風險而合理。

**2. 成本和延遲軸。**

發布的基準測試分數不能替代特定任務的測量。同一模型在您的任務配置文件上的表現可能與標準學術基準測試上非常不同。在承諾使用某個模型之前，在您實際工作負載的代表性樣本上進行測量。

兩個測量優先級因部署上下文而異：

- **同步、面向用戶的任務：** 延遲主導。在前沿模型上需要 8 秒而在較小模型上需要 2 秒的任務，是一個用戶體驗選擇，而非僅僅是成本選擇。在實際並發下測量 p50、p95 和 p99 延遲。
- **批次和異步任務：** 每任務成本主導。測量代表性輸入上的 token 消耗，並乘以每個候選模型的每 token 價格。批次規模下 10 倍的成本差異是每月基礎設施預算項目，而非捨入誤差。

**3. 模型路由。**

複雜的 agentic 系統應將不同的子任務路由到不同的模型，而非對所有操作運行單一模型。這是在設計時做出的架構決策，而非運行時優化。

實用的路由分類法：

- **協調器/規劃器：** 前沿模型——負責目標分解、長期推理和決定調用哪些工具。這裡的品質具有高槓桿效應，因為錯誤會傳播到每個子任務。
- **執行子任務：** 較便宜的模型——分類、結構化提取、重新格式化、簡單摘要。這些是高量、低風險的操作，較便宜的模型在其中表現相當。
- **驗證/批評者：** 中級或前沿模型——檢查中間輸出的品質或一致性。這是一個量較少的操作，品質比成本更重要。

路由和級聯研究（arXiv 2410.10347）展示了最優級聯策略——首先嘗試較便宜的模型，如果不符合品質閾值則升級到更有能力的模型——與對每個請求運行前沿模型相比，在可忽略的整體性能損失下實現 5 倍或更大的成本節省。關鍵見解是，真實世界查詢的很大一部分足夠簡單，較便宜的模型就能成功；只有較難的查詢才需要升級。

- 工程師在為生產提交到某個模型之前，MUST 在其特定任務和 token 配置文件上測量成本和延遲。標準資料集上的已發布基準測試不能替代在代表性生產輸入上的測量。
- 工程師在具有多樣任務類型的系統中 SHOULD 使用模型路由，而非對所有任務使用單一模型。單模型架構在簡單任務上系統性地多付費用，在複雜任務上可能表現不足。
- 系統 MUST NOT 在合成或演示輸入上進行基準測試並在未對代表性生產輸入進行測量的情況下部署到生產。演示輸入通常比生產輸入更短、更清晰、更結構化；演示上的性能不能預測生產性能。

## 深入探討

### 模型路由架構

最簡單的路由架構是靜態路由：在設計時按類型對每個傳入任務進行分類，並將其分配給模型層級。這需要任務分類法，但在成本和延遲上完全可預測。

動態路由添加了一個輕量級分類器，在運行時標記每個請求並將其路由到適當的模型。分類器本身應該是一個小型、快速的模型（或基於規則的系統），以避免增加延遲。LLM 路由生產指南（TianPan.co，2025）記錄了將 50% 的查詢從前沿模型轉移到較小模型，在該部分將成本降低約 14 倍——關鍵條件是必須首先對任務混合進行配置分析，以識別哪 50% 可以安全降級。

### LLM 級聯模式

級聯模式為靜態路由添加了品質門控。請求首先由較便宜的模型嘗試。如果輸出符合定義的品質閾值（例如，置信度分數、schema 驗證檢查或次要模型的評估），則返回它。如果不符合，請求升級到更有能力的模型。

級聯策略的學術分析（arXiv 2410.10347）推導了在品質約束下最小化成本的最優級聯策略，並展示最優策略在使用較便宜模型方面出奇地激進——因為真實世界查詢難度的分佈嚴重偏向簡單查詢。實際啟示：品質閾值，而非模型層級，是主要的設計槓桿。太高的閾值將不必要地升級；太低的閾值將接受差的輸出。在保留資料上實證校準閾值。

### 建立成本/延遲/品質測量矩陣

在生產部署之前，建立具有以下維度的測量矩陣：

- **任務類型：** 列出您的系統執行的不同任務類別（例如，意圖分類、文件摘要、代碼生成、工具調用規劃）。
- **候選模型：** 您正在考慮的 2–4 個模型層級。
- **每個單元的測量：** 代表性輸入上的品質分數（使用您的特定任務評估標準）、中位延遲、p95 延遲，以及在您預期 token 配置文件下每 1,000 項任務的成本。

矩陣使取捨明確，並創建了有文件記錄的基線。它也使模型升級和降級可審計：當新的模型版本聲稱改善的性能時，您可以重新運行矩陣，並根據相同的評估標準進行比較。

### Harness 設計和模型抽象

不同的模型可能需要不同的提示約定、工具調用格式、函數調用 schema 和錯誤處理模式。與特定模型 API 格式緊密耦合的 harness，每次模型選擇更改時都需要大量返工。

工程原則是模型無關的介面設計：harness 應定義一個模型適配器介面，抽象於提供商特定的 API 格式之上。交換模型——或將不同任務路由到不同模型——應該需要更改配置值，而非重寫工具路由邏輯、提示模板或響應解析代碼。這在多個模型可能同時使用的路由架構中尤其重要。

## 最佳實踐

1. **在生產部署之前，為您的任務類型建立成本/延遲/品質測量矩陣。** 在代表性生產輸入上測量，而非合成或演示輸入。合成輸入通常比生產輸入更短、更清晰、更均勻，它們上的性能不能預測生產性能。矩陣也是一個活躍的工件：在新模型版本發布或任務分佈更改時更新它。

2. **設計模型無關的 harness 介面。** Harness 應抽象於模型特定的 API 格式，以便交換模型——或將不同任務路由到不同模型——需要更改配置參數，而非重寫工具路由、提示模板或響應解析。將 harness 與特定模型的約定耦合會產生架構債務，每次模型選擇更改時都會複合。

3. **從前沿模型開始建立品質基線，然後逐任務系統性地降級以找到最低可行層級。** 最低可行層級是在該類別的代表性任務上符合您品質閾值的最便宜模型。「我們可以在這裡使用較便宜的模型」應該是從測量中得出的結論，而非假設。逐任務類型工作，使每次降級都是孤立的，並在下一次之前測量其品質影響。
