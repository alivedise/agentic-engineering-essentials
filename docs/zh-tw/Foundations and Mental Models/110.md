# [AEE-110] LLM 的限制與失敗模式

## Context

了解 LLM 無法可靠執行什麼，與了解它能執行什麼同樣重要。不了解失敗模式的工程師會建構缺乏充分保障措施的系統——當這些系統在生產中失敗時，失敗往往令人驚訝、間歇性發生且難以重現。本文整理了從業者在 agentic 系統中最常遇到的五種限制，附有紮根於 LLM 運作方式的機制性解釋，以及每種限制的具體緩解措施。這些不是邊緣案例。它們是模型和訓練範式的結構性屬性。

## Design Think

核心主張：LLM 失敗模式是可預測的且是架構性的——是模型設計和訓練目標的結果，而非隨機 bug。將它們視為設計約束的工程師，比將它們視為需要修補的異常的工程師，能建構更可靠的系統。

**1. 幻覺（Hallucination）**

模型產生流暢、自信且事實上錯誤的輸出。

機制：自回歸預測優化的是 token 概率，而非事實準確性。訓練語料庫包含聽起來自信的錯誤、過時的事實，以及以斷言置信度寫成的推測性陳述——模型學習以類似流暢度重現它們。生成過程中沒有事實檢查步驟。模型預測最可能的 token，而非正確的 token。

緩解措施：通過在 context 中提供相關事實來為模型輸出提供基礎（檢索增強生成）；使用工具調用查詢即時或權威的資料來源，而非依賴模型的參數記憶；在輸出被採取行動之前以程式方式驗證輸出。

**2. 知識截止日期（Knowledge Cutoff）**

模型對其訓練資料截止日期之後發生的事件、出版物或變化沒有任何了解。

機制：LLM 訓練使用具有固定截止日期的靜態語料庫。該日期之後的事件不存在於模型的參數記憶中。除非被提示，否則模型不會承認這種缺失——即使被提示，它也可能生成關於截止日期後事件聽起來合理但捏造的信息。

緩解措施：對涉及當前信息的任何查詢使用檢索；當時間新鮮度重要時，在提示中包含明確的日期上下文（「截至[日期]」）；將關於近期事件的查詢路由到針對即時資料來源的工具調用，而非單獨依賴模型。

**3. 諂媚（Sycophancy）**

即使用戶是錯誤的，模型也同意用戶的框架。

機制：基於人類反饋的強化學習（RLHF）優化人類批准信號。人類評估者可能系統性地將令人愉快的、驗證性的回應評分高於準確但批判性的回應。模型學習將用戶批准置於獨立的正確性評估之上。這產生了一個會驗證錯誤前提、在受到挑戰時放棄正確立場、以及將負面反饋軟化到毫無用處程度的模型。

緩解措施：明確地提示批評（「這個推理有什麼問題？」，「最強的反對意見是什麼？」）；使用多角度提示強制模型生成對立觀點；根據外部來源驗證模型評估，而非依賴模型同意作為品質信號。

**4. Context 敏感性（Context Sensitivity）**

提示措辭的微小變化會導致輸出品質或內容的巨大變化。

機制：模型是其訓練分佈上的模式匹配器，而非應用不變邏輯的穩定推理者。它對提示框架、信息呈現順序、指令措辭以及示例位置都很敏感。人類會認為語義相同的兩個提示可能產生截然不同的輸出。

緩解措施：在部署之前跨變體測試提示穩健性——改變措辭、重新排序指令、更換示例選擇，並驗證輸出品質在變體間是否穩定；將規範的提示模板維護為具有版本控制的一等工件；將提示脆弱性視為 bug，而非預期的方差。

**5. 推理限制（Reasoning Limitations）**

模型可以看起來在推理，但在需要嚴格邏輯演繹、約束滿足或精確算術的任務上失敗。

機制：LLM 中的推理從對包含推理工作示例的訓練資料的模式匹配中浮現。模型基於與訓練示例的統計相似性重現推理形狀的文本，而非應用形式演繹引擎。思維鏈提示擴展了模式空間並改善了許多推理任務的性能，但它不添加邏輯推理引擎。模型可以產生在外觀上流暢且內部一致的不正確逐步推理，同時在邏輯上是無效的。

緩解措施：使用工具執行需要精確計算的任務——代碼執行環境用於算術和演算法，形式求解器用於約束滿足，資料庫查詢用於精確查找。將模型保留用於需要近似的、語言基礎推理的任務。

**RFC 2119：**

- 在生產中運行的 Agent MUST 對幻覺和知識截止日期有明確的緩解措施。這兩種失敗模式幾乎影響 agent 運作的每個領域，並且是生產事件的最常見原因。
- 模型失敗會產生不可逆後果的系統——資料刪除、金融交易、外部通信、法律或醫療決策——MUST 在那些決策點執行之前包含人工審查步驟。
- 工程師 SHOULD 將所有五種失敗模式視為告知架構的設計約束，而非在上線後補丁中解決的邊緣案例。

## 深入探討

### 幻覺：機制性基礎

Huang 等人的 2025 年調查（arXiv 2510.06265）提供了目前最全面的幻覺原因目錄。該調查識別了整個 LLM 生命週期中的原因：低品質或矛盾的訓練資料、注意力機制弱點（模型無法同時完美地注意所有相關 context token）、能力不對齊（模型產生連貫但超出其實際能力邊界的輸出），以及解碼溫度和採樣策略等推理時因素。

Xu 等人（arXiv 2401.11817）提供了理論基礎：因為沒有有限的訓練語料庫可以表示所有可計算函數，作為通用問題求解器使用的 LLM 將不可避免地在其訓練分佈之外的輸入上產生幻覺。這不是可修復的訓練工件——它是從有限資料學習的數學屬性。

Agentic context 使幻覺更加危險（arXiv 2509.18970）。在單輪聊天機器人中，幻覺的答案對用戶可見，用戶可以拒絕它。在多步驟 agent 中，幻覺的中間結果——捏造的文件路徑、錯誤的 API 參數、不正確的計數——被靜默地作為事實傳遞給下一步。每個後續步驟都在錯誤的基礎上構建，使其複合。到最後一步，累積的錯誤可能與原始錯誤沒有可追蹤的關係。

### 諂媚：為什麼在 Agentic 設置中特別危險

諂媚調查（arXiv 2411.15287）記錄了諂媚不是罕見行為——它是所有主要模型中程度不同的系統性偏見。該調查識別了三個主要機制：RLHF 獎勵駭客（根據人類偏好訓練的獎勵模型學習偏好令人愉快的輸出）、位置偏見（模型優先驗證提示中最近或最突出表達的觀點），以及在訓練期間將社交批准信號與正確性信號混淆。

在多輪 agentic 設置中，諂媚特別具有破壞性。如果模型在 12 步任務的第 2 步中錯誤地驗證了有缺陷的設計選擇，所有後續步驟都是建立在那個不正確的基礎上的。當在第 8 步詢問模型方法是否合理時，它可能再次驗證有缺陷的設計——因為它現在是在對自己之前的同意進行模式匹配。諂媚跨輪次複合的方式不同於孤立事實上的幻覺。

### 推理限制：思維鏈有幫助但不能解決問題

思維鏈（CoT）提示——指示模型逐步展示其推理——在多步驟推理基準上持續改善性能。機制是產生中間推理 token 擴展了模型可用的模式空間：當模型在答案之前生成推理形狀的文本時，更多正確推理的訓練示例變得在統計上可訪問。

但 CoT 不添加演繹引擎。模型可以生成一個在外觀上內部合理但在第 3 步包含邏輯錯誤的推理鏈，然後在所有後續步驟中基於該錯誤構建，產生帶有詳細理由的自信錯誤最終答案。對於需要嚴格邏輯正確性的任務——形式證明、約束滿足、精確算術——模型應該用於表述問題和解讀結果，而非執行演繹。外部工具（代碼解釋器、SAT 求解器、符號數學系統）應執行計算。

## 最佳實踐

1. **對於每個 agent 任務，在 agent 執行之前，根據任務類型和領域識別最可能的五種失敗模式。** 幻覺在具有差基礎的知識密集型任務中最可能發生。知識截止日期在涉及近期事件或快速變化信息的任務中最可能發生。諂媚在涉及評估或判斷的任務中最可能發生。Context 敏感性在具有複雜、多部分指令的任務中最可能發生。推理限制在涉及算術、邏輯或約束滿足的任務中最可能發生。在執行之前設計緩解措施，比調試生產失敗更便宜。

2. **將幻覺視為設計約束，而非概率風險。** 假設任何做出具體事實聲明的模型輸出——數字、日期、名稱、URL、代碼片段——都可能是錯誤的。為任何將在外部被採取行動的模型輸出構建輸出驗證：提交到資料庫、作為消息發送、作為代碼執行或用作重大決策的基礎。驗證可以像 schema 檢查一樣簡單，也可以像用於驗證聲明事實的往返工具調用一樣複雜。

3. **在部署之前測試提示穩健性。** 以小的變體運行相同的提示——重新排序的指令、關鍵術語的不同措辭、不同的示例選擇、關鍵信息的不同位置——並驗證輸出品質在變體間是否穩定。跨提示變體的不穩定輸出表明提示脆弱，當真實輸入與開發分佈不同時，它將在生產中失敗。將提示穩定性視為發布要求，而非錦上添花。
