# [AEE-107] Goals, Tasks, and Plans

## Context

Agent 從人類接收目標並必須將其分解為可執行的步驟。這種分解的品質決定了 agent 是否成功：一個具有清晰任務邊界和可驗證完成標準的明確目標，幾乎總是比交給有能力模型的模糊目標產生更好的結果。大多數生產中的 agent 失敗都可以追溯到三個根本原因之一：目標規格不佳（目標模糊或矛盾）、缺少完成標準（「完成」沒有定義所以 agent 無法驗證成功）、或當步驟失敗時無法重新規劃的剛性計劃。這些是工程失敗，而非模型失敗。

## Design Think

核心主張：目標 → 任務 → 計劃是一個在每個層次都有不同工程關切的層次結構。混淆這三個概念會產生要麼無法啟動（目標太模糊無法分解）要麼無法恢復（計劃太剛性在步驟失敗時無法適應）的系統。

**目標（Goal）：** 使用者想要的東西。目標通常是模糊的、規格不足的或相互衝突的。它們經常包含關於背景、限制和可接受的取捨的隱式假設，而使用者沒有明確表達。Agent 必須在行動前澄清歧義，或採用合理的、明確的解釋並將其浮現給使用者。目標不是工程製品——它是意圖的表達。在執行開始之前，它必須被翻譯成工程製品。

**任務（Task）：** 具有可驗證完成標準的離散、可執行工作單元。目標分解為任務。沒有完成標準的任務不是任務——它是一個方向。「研究這個主題」是一個方向。「為查詢 X 的前 5 個搜索結果返回 3 個要點摘要，每個要點不超過 30 個字」是一個任務：它有一個起始狀態（查詢 X）、一個輸出規格（3 個要點、5 個結果、30 個字）和一個可驗證的完成標準（輸出要麼符合規格，要麼不符合）。完成標準是運行時評估——它們是 harness 在繼續下一步之前驗證每個步驟是否正確完成的機制。

**計劃（Plan）：** Agent 為實現目標將執行的有序任務序列。計劃可以是靜態的（預先生成並按原樣執行）或動態的（根據已完成任務的輸出在執行過程中重新規劃）。靜態計劃適用於有充分理解的、穩定的工作流程。當後期任務以無法預先完全指定的方式依賴於早期任務的結果時，就需要動態計劃——這是大多數非平凡的 agentic 工作的情況。無法在任務失敗時重新規劃的 agent 是附加了語言模型的脆弱腳本。

**RFC 2119：**

- 每個 agent 任務 MUST 有一個可以被程式化驗證或由模型驗證的完成標準。
- 無法分解為具有完成標準的任務的目標 SHOULD 在執行開始前升級給人類進行澄清。
- 計劃 MUST 是可修訂的：無法在任務失敗時重新規劃的 agent 在任何非平凡目標上都會失敗。
- Agent SHOULD 在開始執行之前浮現目標中的歧義，而非在產生不正確的輸出之後。

## 深入探討

### 層次化任務規劃 vs. 扁平任務列表

扁平任務列表是一個獨立行動的序列。當任務沒有依賴關係時它是有效的——當任務 N 的輸出不是任務 N+1 所需要的時候。大多數非平凡的 agent 目標涉及依賴任務：任務 N+1 需要任務 N 的結果，或者任務 N+3 需要在決定做什麼之前知道任務 N+1 是否成功。扁平列表在這裡失敗，因為它們沒有依賴關係的表示，也沒有條件分支的機制。

層次化任務規劃將高層次目標分解為子目標，再將子目標分解為任務，再將任務分解為行動。層次結構使依賴關係明確：任務 N+1 是任務 N 的子節點，而非同等優先級的兄弟節點。這種表示支持依賴追蹤和選擇性重新規劃——當任務 N 失敗時，只有以任務 N 為根的子樹需要重新規劃，而不是整個計劃。

BabyAGI 架構展示了最簡單的動態任務管理循環：執行當前任務、評估結果、根據結果創建新任務、重新排列任務列表、並重複。即使是這個最小循環也處理了扁平列表無法處理的特性：任務列表本身在執行過程中發生變化。

### 動態重新規劃

重新規劃決策在任務失敗、任務的輸出出乎意料、或新資訊改變了未來任務的有效性時出現。三種重新規劃策略涵蓋了大多數情況：

1. **帶修改的重試：** 用調整後的參數重新執行失敗的任務（不同的搜索查詢、不同的 API 端點、縮小範圍）。適用於任務規格有效但執行遇到暫時性或可糾正錯誤的情況。
2. **子樹替換：** 用通過不同路徑實現相同子目標的替代子計劃替換失敗的任務及其依賴項。適用於原始任務根本被阻塞的情況。
3. **目標升級：** 向人類浮現計劃無法按設計完成的情況，並請求澄清或新指令。適用於 agent 能力範圍內不存在替代路徑的情況。

Harness 必須將重新規劃實現為明確的邏輯，而不是作為模型的隱式屬性。模型在提示時可能生成新計劃，但 harness 必須決定何時需要重新規劃、在重新規劃提示中包含哪些資訊，以及如何將新計劃與已完成的工作整合。

### 生產中的目標歧義

目標歧義以最糟糕的失敗模式表現：模型在不正確的解釋上繼續，產生有信心的、內部一致的、錯誤的輸出。這比拋出錯誤的任務失敗更糟糕，因為 harness 可能無法檢測到它。輸出在表面上看起來正確；只有當下游系統消費輸出或人類審查最終結果時，錯誤才變得明顯。

模式：使用者提供一個帶有隱式假設的目標，而 agent 不共享這個假設。Agent 根據其訓練分佈選擇最合理的解釋，執行計劃，並交付一個技術上與其解釋一致但與使用者意圖不符的結果。沒有引發錯誤；沒有產生信號。失敗是靜默的。

緩解措施：提示模型在執行開始之前識別不清楚的要求並將其作為問題浮現。將目標澄清視為 agent 循環中的一流步驟，而非例外處理器。如果使用者無法或不願澄清，記錄採用的解釋並在 agent 的輸出中浮現，以便人類可以評估它是否正確。

### 完成標準作為運行時評估

完成標準不是行政文件——它們是可執行規格。每個標準應該可以實現為 harness 在任務執行後運行的檢查：輸出是否符合類型和結構規格？它是否滿足領域特定的限制？它是否滿足長度、格式或內容要求？

這是與 AEE-102（RL + 可驗證獎勵）的連接：完成標準是可驗證獎勵信號的運行時等價物。完成標準可以自動驗證的任務是可以測量的任務，而可以測量的任務是可以改進的任務。沒有完成標準的任務是一個黑盒子：harness 無法知道它是否成功或失敗，也無法提供隨時間改進性能所需的反饋信號。

## 最佳實踐

1. **在編寫 agent 任務時同時編寫完成標準，而非之後。** 如果你無法說明如何驗證任務是否正確完成，則任務規格不足。編寫完成標準的行為通常會在任務定義本身中浮現歧義——這比在執行後發現要好。
2. **從一開始就將重新規劃內建到 harness 中。** 無法在步驟失敗時調整計劃的 agent 是脆弱的腳本，而非 agent。重新規劃不是邊緣案例——它是非平凡目標的正常路徑。設計 harness 以期待和處理計劃修訂，而非將其視為錯誤條件。
3. **在執行之前提示模型識別不清楚的要求。** 將歧義作為結構化的澄清請求浮現，而非作為糟糕的輸出。在執行前提出的問題只需要一次往返；在完整執行運行後產生的糟糕輸出可能需要整個運行和不正確結果的下游後果的代價。
