# [AEE-304] 指令遵循

## 情境

當模型沒有遵循指令時，直覺反應是責怪模型。但在大多數情況下，問題出在指令本身。經過指令遵循 (instruction following) 資料訓練的模型，對於清晰、具體、內部一致的指令有很強的執行能力。真正讓它們可靠地失敗的，是那些模糊、互相矛盾、過度冗長，或預設了模型並不具備的共同情境的指令。診斷指令遵循失敗，需要站在模型解析的角度來閱讀指令，而非工程師預設的意圖。

## 設計思維

核心主張：指令遵循的失敗，通常是工程師的問題，而非模型的問題。大多數失敗源於模糊的行動動詞 (action verb)、互相矛盾的約束、過長的指令，或是預設了模型並不具備的共同情境。

**讓指令能夠被可靠解析的要素：**

1. **具體的行動動詞。** "Summarize"（摘要）是明確的：產生輸入的精簡版本。"Handle"（處理）則不然：它是指回應、轉發、分類、解決，還是記錄？明確的行動動詞範例：summarize（摘要）、extract（擷取）、classify（分類）、rewrite（改寫）、translate（翻譯）、evaluate（評估）、list（列舉）、compare（比較）、generate（生成）。模糊的行動動詞範例：handle（處理）、process（處理）、work with（處理）、deal with（應對）、manage（管理）、address（處理）。

2. **明確的範疇。** "Summarize the document"（摘要這份文件）留下了許多開放問題：哪些章節？長度為何？格式為何？"Summarize the Executive Summary and Key Findings sections of the document in three bullet points, each under 20 words"（用三個要點摘要文件中的執行摘要與關鍵發現章節，每點不超過 20 個字）就是有範疇的指令。

3. **明確的成功標準。** "Make this clearer"（讓這個更清楚）不是一個指令——它是一個方向，但沒有定義「清楚」的含義。"Rewrite this paragraph so that a reader with no domain knowledge can understand the main claim in one reading"（改寫這段話，使沒有領域知識的讀者能一次閱讀就理解主要論點）才定義了成功標準。

4. **不預設隱含情境。** 參考「我們的風格指南」、「慣用格式」或「標準做法」的指令，預設模型知道這些是什麼。除非在提示中明確提供，否則模型並不知道。

**失敗模式：**

1. **長度超載。** 單一指令區塊包含超過 7 至 10 個不同約束，對模型在長篇回應中持續追蹤是一大挑戰。模型並非未能理解指令——它成功執行了前幾條，然後將其餘的降低優先順序。緩解方法：減少約束數量，或使用編號清單讓模型更容易追蹤已應用的項目。

2. **內部矛盾 (contradiction)。** 指令 A 說「要簡短」，而指令 B 說「涵蓋所有相關考量」。模型必須二擇一——而它會根據近因性與顯著性啟發式規則來選擇，而非工程師的意圖。緩解方法：在部署前審核指令是否存在矛盾。

3. **隱含情境 (implicit context)。** 指令預設模型知道某些它實際上不知道的事：領域背景、公司慣例、讀者背景、輸出目的。緩解方法：將隱含假設在情境區塊中明確說明（參見 AEE-301）。

4. **格式模糊性 (format ambiguity)。** "Respond formally"（正式地回應）沒有指定在此情境下「正式」是什麼意思。正式可以是不使用縮寫、可以是法律文件語氣、可以是只用項目符號，也可以是特定的章節標題。緩解方法：提供格式範例，或以機械式的方式描述格式需求。

**指令衝突時的優先順序：**

當多個指令可能產生衝突時，模型使用兩種啟發式規則：近因性（提示中較後面的指令權重較高）與顯著性（明確標記的優先順序，例如「Most important:」或編號清單，比散文段落中的陳述更具權重）。工程師可以善用這些規則：將最重要的約束放在指令區塊的最後，並在衝突可能發生時明確標記優先順序。

**RFC 2119：**

- 指令 MUST（必須）在部署前審核是否存在內部矛盾。兩位工程師應能各自獨立閱讀指令集，並對合規回應的樣貌達成共識。
- 當多個約束在生產輸入中可能衝突時，指令 SHOULD（應該）使用明確的優先標記 (priority marker)，例如「Most important:」或「If in doubt, prioritize X over Y」。
- 指令 SHOULD（應該）使用具體的行動動詞。使用模糊動詞（handle、process、address）的指令 MUST（必須）在部署前改寫。

## 深度探討

### 指令審核

指令審核 (instruction audit) 是部署前對指令集進行的結構化審查，針對四種失敗模式進行檢核：

1. **模糊性 (ambiguity) 檢核：** 每條指令是否能以兩種或以上的方式解讀？若是，選定一種解讀並明確表述。

2. **矛盾檢核：** 是否有任何指令與其他指令衝突？常見衝突：簡潔 vs. 完整、正式 vs. 易讀、速度 vs. 精確。若存在衝突，以明確的優先順序聲明來解決。

3. **隱含情境檢核：** 是否有任何指令預設了模型無法從提示中推斷的前提？若是，將該假設移至情境區塊。

4. **範疇檢核：** 每條指令是否都有明確的範疇？"Extract key points"（擷取重點）是沒有範疇的。"Extract up to five key points, each as a single sentence under 15 words"（擷取最多五個重點，每點為一個不超過 15 個字的句子）是有範疇的。

在部署前執行審核，能在源頭捕捉大多數指令遵循的失敗，而非等到生產環境才發現。

### 實作範例

**含有三種失敗模式的指令集：**

```
System: You are an AI assistant. Help users with their questions. 
Be professional but friendly. Always be thorough in your responses 
and cover all relevant information. Keep responses concise. 
Use our standard format. Make sure to address any follow-up questions 
the user might have.
```

存在的失敗模式：
- **矛盾：** "thorough... cover all relevant information"（涵蓋所有相關資訊）與「keep responses concise"（保持簡潔）互相衝突
- **隱含情境：** "our standard format"（我們的標準格式）——模型不知道這是什麼
- **隱含情境：** "follow-up questions the user might have"（使用者可能有的後續問題）——在沒有明確約束的情況下，預測未陳述的問題是不可能的
- **模糊動詞：** "Help users"（協助使用者）——如何協助？解釋？修復？分類？轉介？

**改寫後的指令集：**

```
System: You are a customer support specialist for a SaaS product. 
Answer questions about account management, billing, and product features.

For each response:
1. Answer the stated question directly in the first paragraph.
2. If the answer requires steps, use a numbered list.
3. End with one sentence offering to clarify if needed.
4. Maximum 200 words.

Do not speculate about questions the user has not asked.
Do not provide information about competitor products.
```

改動說明：
- 角色已明確（客服專員，限定領域範疇）
- "Help"（協助）替換為「Answer questions about [scoped domain]"（回答關於特定領域的問題）
- 矛盾已解決：以編號格式加 200 字上限，取代「thorough"（完整）與「concise"（簡潔）的模糊組合
- "Standard format"（標準格式）替換為明確的結構規則
- "Follow-up questions"（後續問題）移除，以具體的「提供澄清」句子取代

## 最佳實踐

1. **每個指令集在部署前都要執行雙工程師測試。** 請同事獨立閱讀你的指令，並描述他們認為合規的回應樣貌。如果描述與你的有所不同，指令就是模糊的——而模型也會產生同樣程度的變異性。

2. **超過三個約束的指令集，優先使用編號清單而非散文。** 編號清單讓你和模型都更容易追蹤已應用的約束。將多個約束嵌入段落的散文指令更難審核，模型也更難一一列舉。

3. **透過改善指令來修復指令遵循失敗，而非增加更多指令。** 當模型未遵循某條指令時，直覺是加上「我說過要 [X]！」之類的強調。增加指令來修補不合規，只會讓指令集更長、更矛盾。正確做法是：診斷原始指令失敗的原因（模糊性、矛盾、隱含情境），並從源頭修正。
