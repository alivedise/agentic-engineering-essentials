# [AEE-301] 提示結構基礎

## 情境

大多數提示詞 (prompt) 失敗的根源在於結構問題。模型產生不一致的輸出，並非因為它不可靠，而是因為提示詞沒有足夠精確地傳達意圖——使用了模糊的行動動詞 (action verb)、省略了輸出格式限制 (output format constraint)，或將本應分開的資訊混雜在一起。了解提示詞結構的工程師能夠系統性地診斷這些失敗，並從根源修正，而非在下游增加更多的變通方案。

## 設計思維

核心主張：提示詞不是自由格式的文字——它是一個有結構的工件，各個元件具有不同的工程功能。沒有刻意結構的提示詞會產生不一致的結果，並非因為模型不可靠，而是因為工程師沒有明確地傳達意圖。

一個結構良好的提示詞有四個元件：

1. **角色/人設 (role/persona)** — 模型在此互動中扮演的角色。設定行為預設值、語調風格，以及隱含的知識領域。「你是一位正在審查 pull request 的資深軟體工程師」所激活的回應模式，與未指定角色相比截然不同。

2. **情境區塊 (context block)** — 模型完成任務所需、但無法單從指令推斷的資訊。包含當前文件、使用者的狀況、所在系統的限制條件。情境區塊是有範疇的：它只包含與當前任務相關的內容，而非一般性的知識傾倒。

3. **指令 (instruction)** — 模型必須執行的事項。一個主要行動動詞（摘要、擷取、分類、改寫、評估），加上明確的範疇與成功標準。指令是一份合約：它必須足夠具體，讓模型無法以多種互不相容的方式來滿足它。

4. **輸出格式限制 (output format constraint)** — 輸出必須如何組織。JSON 綱要、Markdown 章節標題、最大長度、必填欄位。這個元件是最常被省略的，也是缺失時造成最多下游失敗的元件。

**元件與對話輪次的對應關係：**

系統輪次 (system turn) 保存角色與穩定的情境（適用於所有互動的指令）。使用者輪次 (user turn) 保存特定於任務的情境與指令。助理預填 (assistant prefill) 在使用時，透過以所需格式開始回應來設定輸出格式。

**元件之間的互動：**

寫得好的指令可能被衝突的輸出格式限制所破壞。「清楚解釋這個概念」與「以 JSON 格式回應」相互衝突。缺少輸出格式限制會導致模型自行選擇格式，而格式在不同執行之間會有所不同。

**負空間：**

過度規格化的提示詞與規格不足的提示詞同樣脆弱。一個列舉所有可能邊緣情況的 2,000 token 系統提示，表明工程師不信任模型——並且產生一個對那些列舉的確切措辭高度敏感的提示詞。正確的規格化程度是：足以消除歧義，但不至於過度限制模型的推理空間。

**RFC 2119：**

- 部署於生產環境的提示詞 MUST（必須）具有明確的輸出格式限制。沒有輸出格式限制的提示詞，在不同執行之間將產生不同的格式。
- 指令 SHOULD（應該）使用明確的行動動詞（摘要、擷取、分類、改寫），而非模糊的指令（處理、操作、負責、應對）。
- 當任務需要特定的行為風格或知識領域時，SHOULD（應該）加入角色/人設規格。對於範疇明確的簡單擷取任務，若角色對輸出沒有實質影響，MAY（可以）省略。

## 深度探討

### 元件結構詳解

**角色/人設：** 最有效的角色規格是具體且有根據的：「你是一位為開發者受眾審查文件的技術寫作員」比「你是一個有幫助的助理」效果更好。角色設定了關於語調、詞彙、預設讀者知識，以及良好回應樣貌的隱性先驗。角色規格之所以有效，是因為它們激活了與該角色溝通風格相關的訓練模式。

角色規格不會增加模型原本沒有的能力。告訴模型它是專家，並不會讓它在訓練分佈範圍之外的任務上更加準確。角色規格的作用是將模型引導至更一致的行為風格。

**情境區塊：** 情境區塊是放置基礎資訊的地方：要摘要的文件、要審查的程式碼、使用者的明確目標、系統的相關限制。它不是放置一般背景知識（模型從訓練中已具備）或指令（屬於指令元件）的地方。

一個常見的結構性錯誤是混淆情境與指令：「這是合約（情境）。審查它並找出任何產生責任風險的條款（指令）。此外，我們的管轄地是加州（更多情境），我們偏好白話摘要（輸出格式限制）。」將這些內容分別整理成明確的元件，使提示詞更易於測試、版本控制和除錯。

**指令：** 指令元件必須通過所謂的「雙工程師測試」：兩位工程師獨立閱讀這個指令，是否會對正確回應的樣貌達成共識？若否，指令規格不足。常見的失敗模式：模糊動詞（「處理」、「應對」）、缺少範疇（「摘要這份文件」——哪些章節？多少字？）、以及缺乏成功標準（「讓這個更好」——以什麼標準衡量更好？）。

**輸出格式限制：** 輸出格式限制應指定：結構（JSON 物件、Markdown 列表、純文字段落）、必填欄位或章節，以及任何長度或數量限制。對於 JSON 輸出，應附上綱要。對於 Markdown 輸出，應展示所需章節結構的範例。

### 實例演示

**弱提示詞（無刻意結構）：**

```
Help me with this email. I need to respond to a customer complaint about a delayed shipment.
The customer is upset and wants a refund. Here is their email: [EMAIL]
```

問題所在：沒有角色。沒有輸出格式限制。「Help me with」不是一個行動動詞。「退款」一詞被提及，但沒有限制是否要提供退款。模型可能以草稿郵件、策略建議、要點列表或混合形式來回應。

**重新結構化的提示詞（四個元件明確列出）：**

```
[Role]
You are a customer service specialist at an e-commerce company. You write professional,
empathetic responses that de-escalate situations while following company policy.

[Context]
Company policy: issue a 10% discount on the next order for delays under 7 days;
offer a full refund for delays over 7 days. This shipment is 4 days delayed.

[Instruction]
Write a response to the following customer complaint email that acknowledges the delay,
applies the correct policy, and closes on a positive note.

[Output format]
Respond with a single email draft. Subject line on the first line, blank line, then body.
150 words maximum. No bullet points.

[Customer email]
[EMAIL]
```

改變了什麼：角色激活了客服溝通風格。情境區塊包含政策限制（防止模型自行發明退款政策）。指令使用了帶有範疇的具體行動動詞。輸出格式限制指定了結構、長度與格式。

## 最佳實踐

1. **在撰寫指令之前先撰寫輸出格式限制。** 決定輸出必須是什麼樣子，能迫使對指令實際要求的內容更加清晰。先撰寫指令的工程師往往會產生模糊的指令，因為他們尚未確定成功的樣貌。

2. **單獨測試每個元件。** 改變角色，觀察輸出品質是否改變。移除輸出格式限制，觀察變異情況。加入衝突的指令，觀察模型是否能察覺。元件層級的測試能揭示提示詞中哪些部分是關鍵的。

3. **將情境區塊最小化至當前任務實際需要的內容。** 情境區塊中的每個 token，都是無法用於推理的 token。如果相同的背景資訊出現在每個提示詞中，它應屬於系統提示詞。如果是特定於任務的，則只包含與任務相關的子集。
