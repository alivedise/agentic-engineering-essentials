# [AEE-205] 微調基礎

## 情境

微調（fine-tuning）往往被過早採用，且原因不正確。工程師用它來修正幻覺問題、提升推理品質，或補充模型缺乏的知識——結果往往事與願違，甚至讓情況更糟。正確決策的前提，是理解微調實際上做了什麼，以及它做不到什麼。

## 設計思維

微調在特定資料集上調整模型權重，以提升模型在特定分布上的表現。它不是通用的改善機制——它改善的是模型原本就能執行的任務的一致性、風格與效率。它不能教導模型新的事實，也無法可靠地修正推理失誤。

**三個合理的微調理由：**

1. **提示詞無法穩定達成的風格與格式一致性。** 當任務需要高度特定的輸出結構、語氣或格式，而少樣本提示詞（few-shot prompting）的表現不穩定時，微調可以將這種一致性訓練進模型。
2. **透過模型蒸餾降低成本與延遲。** 大型模型在特定任務上的行為可以蒸餾到較小的微調模型中，降低每次呼叫的成本與延遲，同時不顯著損失品質。
3. **模型預設處理不佳的領域特定詞彙或輸出結構。** 落在模型訓練分布之外的專業術語、不尋常的輸出結構或領域慣例，都是微調適合介入的場景。

**三個不合理的微調理由：**

1. **補充模型缺乏的知識。** 微調從範例中訓練行為模式——它不是知識注入機制。若模型缺乏事實或最新資訊，請使用 RAG（見 [AEE-108](../Foundations and Mental Models/108)）。在包含目標事實的資料集上進行微調，無法讓模型在推理時可靠地回憶這些事實。
2. **修正幻覺（hallucination）。** 微調可以降低幻覺在訓練分布上的發生頻率，但無法消除問題。在一般任務中產生幻覺的模型，微調後仍會在邊緣案例和分布外輸入上產生幻覺。
3. **替代提示詞最佳化。** 在考慮微調之前，必須先窮盡提示詞工程（prompt engineering）的可能性。許多被歸因於模型能力的問題，實際上是提示詞設計問題。在提示詞未最佳化的情況下進行微調，只會複合錯誤。

- 工程師**必須（MUST）**在微調前建立目標任務的評估基準（baseline benchmark）。沒有量測基準，就無法判斷微調是否帶來改善。
- 在提示詞工程窮盡之前，**絕對不能（MUST NOT）**嘗試微調。請先執行結構化的提示詞實驗——少樣本範例、思維鏈（chain-of-thought）、明確的格式指令——並量測結果，再決定是否進行微調。

## 深度探討

### 完整微調 vs. 低秩適應（LoRA）與參數高效微調（PEFT）

**完整微調（full fine-tuning）**會更新模型的所有參數。代價高昂：需要足夠的 GPU 記憶體來保存所有可訓練狀態的模型權重、每個參數的最佳化器狀態，以及梯度。對於大型模型，這對大多數團隊而言是不切實際的成本。完整微調也帶來最高的災難性遺忘（catastrophic forgetting）風險——因為每個權重都被調整，模型的通用能力最容易被狹窄的訓練分布所取代。

**低秩適應（LoRA，Low-Rank Adaptation）**由 Hu 等人在論文《LoRA: Low-Rank Adaptation of Large Language Models》（arXiv:2106.09685，2021）中提出，直接解決了這個問題。LoRA 不更新原始權重，而是凍結預訓練模型的權重，並在凍結層旁注入可訓練的低秩分解矩陣（low-rank decomposition matrices）。只有這些小型矩陣被訓練，原始權重從不被修改。

效率提升相當顯著。論文報告顯示，與使用 Adam 對 GPT-3 175B 進行完整微調相比：LoRA 將可訓練參數數量減少了 10,000 倍，GPU 記憶體需求減少了 3 倍。儘管如此，論文顯示 LoRA 在 RoBERTa、DeBERTa、GPT-2 和 GPT-3 任務上的表現與完整微調相當甚至更好，且由於訓練後的矩陣可以在服務時合併回基礎權重，不會增加額外的推理延遲。

**參數高效微調（PEFT，Parameter-Efficient Fine-Tuning）**是 Hugging Face 的函式庫，在統一的 API 下提供 LoRA 及許多相關技術的實作——包括前綴調整（prefix tuning）、提示詞調整（prompt tuning）等。PEFT 是大多數微調工作的實務起點，提供工具、文件，以及與 transformers 生態系統的預整合。

### 何時選擇 LoRA，何時選擇完整微調

針對風格適應、格式一致性和領域術語，從 LoRA 開始。LoRA 迭代更快、所需算力更少，也較不容易破壞基礎模型的通用能力。凍結的基礎權重作為自然的正則化器（regularizer）：模型只能透過低秩矩陣調整行為，這限制了它偏離原有能力的程度。

只有在 LoRA 可量測地無法達到所需效能，且算力預算允許的情況下，才應升級至完整微調。這意味著要先執行 LoRA 實驗、對比基準量測，並確認效能差距無法透過資料品質改善或超參數調整來彌補。對於大多數風格與格式適應任務，完整微調不應作為起點。

### 資料需求

**品質重於數量**是微調資料的主導原則。資料集大小與微調品質之間的關係不是線性的：少量高品質、多樣化、去重後的範例，會持續勝過大量含雜訊的資料集。訓練資料中的雜訊會教導模型重現這些雜訊。

幾個最重要的屬性：

- **格式一致性。** 訓練範例中格式不一致，會導致輸出格式不一致。若一半範例使用 JSON，一半使用純文字，微調後的模型將無法穩定選擇任一格式。整個訓練集應強制統一格式。
- **分布覆蓋。** 只涵蓋常見案例的訓練資料，只能產出處理常見案例的模型，邊緣案例則會失敗。應刻意納入罕見輸入、邊緣案例和預期失效模式的範例。微調後的模型在訓練分布之外的輸入上，其行為在很大程度上是不確定的。
- **去重（deduplication）。** 重複範例在訓練中會對這些案例賦予過高的權重。包含 50 個近乎相同模式範例的訓練集，會產出偏向該模式的模型，而犧牲其他模式。

沒有普遍適用的最小範例數量——需求取決於任務複雜度、模型大小和目標效能水準。從小量開始，量測，並只在量測顯示更多資料有幫助時才擴大規模。

### 微調前評估基準的要求

評估基準（baseline benchmark）不是可選項目。在撰寫第一個訓練範例之前：

1. 定義任務的評估指標——精確匹配、BLEU、人工評分、任務成功率，或適合領域的自訂評分函數。
2. 構建一個代表完整目標分布的留出評估集（held-out evaluation set），包含邊緣案例。
3. 以你能產出的最佳提示詞，針對基礎模型執行評估。這就是基準分數。
4. 記錄基準分數，以及模型版本、提示詞版本和評估集版本。

微調後，對微調模型執行完全相同的評估。只有評估完全一致，比較才有意義。若與基準相比的改善不顯著，微調沒有幫助——應回頭診斷實際問題所在（資料品質、評估設計，或微調的原始前提）。

這個紀律也能防範一種隱性失效模式：微調改善了訓練分布上的分數，卻降低了實際目標分布上的效能。若評估集是從訓練相鄰的範例中構建，改善可能不具泛化性。

### 災難性遺忘與過度特化

在狹窄任務上進行微調，可能降低模型在它原本能夠良好處理的任務上的效能——這個現象被稱為**災難性遺忘（catastrophic forgetting）**。模型的權重編碼了廣泛的通用能力，這些權重在微調中被調整趨向狹窄的訓練分布。未在微調資料中被強化的通用能力可能因此退化。

LoRA 可以緩解但不能消除這個風險：因為基礎權重是凍結的，通用能力受到更多保護。完整微調帶來最高的災難性遺忘風險。

一個實際的緩解措施：在微調前後，針對通用基準，或針對模型預期處理的完整任務分布的代表性樣本進行評估。一個在目標任務上有所改善但在通用任務上明顯退化的模型，若這些通用任務屬於其操作範疇，可能不適合部署到生產環境。

相關的問題是**過度特化（over-specialization）**：微調後的模型在訓練資料相同分布的範例上表現良好，但在生產中稍有差異的輸入上表現更差。部署後應監控生產輸入分布。若生產輸入偏移了訓練分布——不同的措辭方式、不同的領域、不同的使用者族群——微調模型的優勢可能削弱甚至逆轉。需要定期針對生產樣本進行評估以偵測這種情況。

## 最佳實踐

1. **在微調之前，先窮盡提示詞工程。** 執行結構化的提示詞實驗——少樣本範例、思維鏈、明確的格式指令。許多「需要微調」的問題，實際上是提示詞設計問題。在沒有良好最佳化基準提示詞的情況下進行微調，會使你無法判斷微調是否真的是正確的介入方式。

2. **從 LoRA 或 QLoRA 開始，而非完整微調。** 迭代更快、所需算力更少，且災難性遺忘的風險更低。只有在 LoRA 在真正的資料品質和超參數迭代後，可量測地仍無法達到所需效能時，才升級至完整微調。

3. **在接觸訓練資料之前，先建立評估基準。** 定義評估方式，針對有提示詞的基礎模型執行評估，記錄分數。這個數字是微調是否合理的標準。沒有它，就無法知道微調是否有幫助、有害，還是毫無差別。
