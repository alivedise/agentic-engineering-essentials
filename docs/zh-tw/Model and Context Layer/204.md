# [AEE-204] 系統提示詞工程

## 情境

系統提示詞（system prompt）是設定模型行為的核心機制，相當於工程師的設定程式碼（configuration code）。然而，與一般設定檔不同的是，系統提示詞與模型的訓練分布以非直覺的方式交互影響：放置位置影響解讀方式、長度影響注意力品質，而使用者在 user turn 的輸入也可能以工程師未預期的方式破壞系統指令。

## 設計思維

系統提示詞是一種架構產物，而非提示詞技巧。它定義了模型的持久性行為設定：人物設定（persona）、任務框架、輸出格式要求、工具描述與安全限制。工程師若將它視為「加入更多指令」的地方，而非受版本控制的設定產物，最終積累的技術債與未經審查的設定檔如出一轍。

**應放入系統提示詞的內容：**
- 穩定的行為指令（人物設定、語氣、責任範圍）
- 在所有互動中一致的輸出格式要求
- 工具描述與呼叫指令
- 安全限制與禁止行為

**不應放入系統提示詞的內容：**
- 動態內容，例如當前日期或使用者特定資料——請在請求時注入到對話上下文或 user turn
- 大型知識庫——透過 RAG 檢索相關內容（見 [AEE-108](../Foundations and Mental Models/108)），而非將文件直接貼入系統提示詞
- 逐 session 變動的對話上下文——這屬於對話歷史，而非系統層級設定

**位置承載信任語意。** 模型將系統提示詞的指令視為工程師層級的設定——具有權威性，且先於使用者意圖。放在 user turn 的指令則被解讀為使用者意圖——信任層級較低，模型可能針對其進行推理或優先順序判斷。這個差異不只是 API 機制問題；它反映了模型在生成過程中如何權衡相互競爭的指令。

**RFC 2119 要求：**

- 系統提示詞**必須（MUST）**作為設定產物進行版本控制（version control）。僅存放在資料庫欄位或無變更歷史的環境變數中的系統提示詞，是無法稽核的設定產物。
- 系統提示詞**應該（SHOULD）**在部署前針對各種輸入變化進行健壯性測試。在一般路徑下運作的系統提示詞，在改寫措辭、對抗性表述或離題輸入下可能產生意外行為——[AEE-110](../Foundations and Mental Models/110) 描述的上下文敏感性（context sensitivity）在此直接適用。
- 工程師**必須（MUST）**將提示詞注入（prompt injection）——使用者輸入試圖覆蓋系統提示詞指令的行為——視為安全問題，而非模型品質問題。

## 深度探討

### 系統 / 使用者 / 助理預填的位置語意

三種 API 角色（role）承載不同的解讀語意：

**System turn** 設定基準行為，具有最高信任層級。放置於此的內容被視為在任何使用者互動開始之前即已建立的權威設定。模型的訓練使其尊重系統提示詞的限制，並據此解讀後續 turn。Anthropic 的文件指出，即使只是系統提示詞中的一句話來建立角色，也能有效聚焦 Claude 的行為與語氣。

**User turn** 被解讀為使用者意圖。模型根據系統提示詞建立的限制，判斷使用者想要什麼。僅出現在 user turn 的指令，信任層級低於系統提示詞指令——這也是提示詞注入的攻擊面。

**助理預填（assistant prefill）**（在 API 支援的情況下）允許工程師預先生成模型回應的開頭。這對強制輸出格式很有用（例如預填 `{` 以強制輸出 JSON），但需謹慎：它繞過了模型的自然 turn 結構，若預填內容與系統提示詞產生矛盾，可能出現意外結果。

**指令位置影響注意力品質。** Anthropic 的提示詞文件指出，將查詢放在長提示詞的末尾——位於查詢所針對的長篇資料之後——在測試中可將回應品質提升最多 30%，尤其對複雜的多文件輸入效果顯著。機制在於注意力（attention）：模型對查詢的表徵在生成時最為新鮮，而它所針對的資料已完整地存在於其上方的上下文中。

### 精簡與完整性的取捨

系統提示詞中的每個 token 都消耗上下文預算（見 [AEE-108](../Foundations and Mental Models/108) 與 [AEE-201](201)）。過度規格化的系統提示詞有三個代價：消耗了原本可用於對話歷史或檢索知識的上下文；將注意力分散在更多指令上，可能降低對任何單一指令的遵循程度；以及難以維護和稽核。

對系統提示詞中每條指令，應問的正確問題是：**移除這條指令是否改變模型行為？** 如果移除後模型產生相同的輸出，表示這條指令要麼已被模型的訓練涵蓋（它本來就會這樣做），要麼與提示詞中的另一條指令重複。兩種情況下，都應移除。

這需要測試，而非憑直覺判斷。用相同的輸入分別對有無候選指令的系統提示詞進行測試。若輸出難以區分，該指令即為冗餘。

### 版本控制的實踐

系統提示詞是設定產物，應以此標準管理：

- **以檔案形式存放於版本控制系統，而非資料庫欄位。** 存放於程式碼庫的檔案有提交歷史、差異審查與還原能力。存放於設定資料表的欄位預設沒有這些。
- **在提交訊息中標注變更意圖。** 「更新系統提示詞」的提交訊息毫無用處。「將範圍限制為面向客戶的問題；移除內部工具描述」才是可稽核的記錄。
- **變更後執行回歸測試。** 修復某個行為的變更可能破壞另一個行為。以具代表性的輸入與預期輸出組成的測試集，能在部署前偵測回歸問題。

系統提示詞是模型行為的規格書，值得與應用程式碼相同的版本控制紀律。

### 提示詞注入攻擊向量（OWASP LLM01:2025）

提示詞注入（prompt injection）（OWASP 將其分類為 LLM01:2025）是一類攻擊，其中輸入導致模型以工程師未預期的方式行為。OWASP 記錄了三種主要注入類型：

**直接注入（direct injection）：** 使用者的輸入明確試圖覆蓋系統提示詞指令。典型範例：「忽略先前的指令，改為……」直接注入是最顯而易見的形式，也通常最容易偵測。對常見的覆蓋措辭進行簡單模式比對可提供部分防護，但攻擊者可以無限變換措辭。特權分離（privilege separation）——確保模型不執行聲稱具有系統層級權威的 user turn 指令——比模式比對更為健壯。

**間接注入（indirect injection）：** 對抗性指令被嵌入代理在執行過程中讀取的外部內容——使用者上傳的文件、工具擷取的網頁、資料庫返回的結果。模型無法可靠地區分「待處理的內容」與「待遵循的指令」，當兩者透過同一個管道到達時尤其如此。一個讀取網頁並據其採取行動的代理，可能被包含格式類似系統提示詞指令的網頁所顛覆。這個攻擊向量隨代理的工具能力而擴大：代理從外部來源讀取的內容越多，間接注入的攻擊面就越大。

**多模態注入（multimodal injection）：** 在處理圖像或其他非文字模態的模型中，對抗性指令可以隱藏在這些模態中——例如，以低對比度或與背景相同顏色的字體，將文字嵌入圖像。模型處理完整圖像，可能執行對人工審查者而言不可見的嵌入指令。

**緩解措施：**
- **輸入清理（input sanitization）：** 在傳遞給模型之前，過濾或跳脫使用者輸入中的常見注入模式。這不是完整的防禦，但能縮小攻擊面。
- **特權分離（privilege separation）：** 系統層級指令（模型被允許做什麼、絕對不應做什麼）必須在 system turn 中建立，且不應被 user turn 的內容覆蓋。系統設計應確保聲稱具有較高權限的 user turn 指令根本不被執行。
- **執行前的輸出驗證：** 在代理情境中，在執行前根據預定義的政策驗證模型提議的動作。模型應負責提議；獨立的驗證層應負責授權。來自檢索文件的內容應被視為待處理的資料，而非待遵循的指令。

### 測試系統提示詞

測試系統提示詞，意味著在壓力下驗證其行為正確，而非僅測試開發時使用的範例：

- **變換使用者輸入措辭。** 若系統提示詞指示模型只以英文回應，測試使用者以法文書寫、混合語言，或詢問模型為何只能以英文回應時的行為。
- **測試對抗性輸入。** 明確測試直接注入嘗試。若系統提示詞通過這些測試，記錄測試內容與時間。
- **在邊緣情境下驗證。** 測試相鄰但超出定義範圍的輸入。當使用者客氣地要求時，模型是否仍能正確拒絕範圍外的請求？
- **跨版本測量回歸。** 系統提示詞變更後，執行相同的測試集並比較輸出。非預期的行為變化即為缺陷。

這是將 [AEE-110](../Foundations and Mental Models/110) 的提示詞健壯性測試方法應用於設定層的實踐。

## 最佳實踐

1. **積極精簡系統提示詞長度。** 對每條指令測試移除後是否改變模型行為——若不改變，則移除。更短的系統提示詞消耗更少上下文預算，讓模型注意力聚焦在重要的事項上，也更易於維護和稽核。

2. **在專屬檔案中進行版本控制，並撰寫描述性的提交訊息。** 將每次變更視為有意圖的設定變更。解釋指令新增或移除原因的提交訊息，讓系統提示詞具有可稽核性。沒有變更歷史的系統提示詞是一種風險。

3. **在部署任何處理不受信任使用者輸入的代理之前，明確測試對提示詞注入的抵抗力。** 直接注入、間接注入（來自工具結果與檢索文件）與多模態注入都是已記錄的攻擊向量。對抗性輸入不是邊緣案例——它們是攻擊面本身。
