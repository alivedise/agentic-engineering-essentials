# [AEE-202] 上下文視窗架構

## 情境

AEE-108 確立了上下文是有限資源、必須主動管理的概念。本文進一步說明為何上下文同時也是一種非均勻資源：資訊在上下文視窗中的位置，會影響模型能否可靠地關注到該資訊。一個 128K token 的視窗，並不代表每一個 token 位置都能得到同等程度的注意力（attention）。Transformer 的注意力機制（attention mechanism）具有位置感知性，研究顯示模型從上下文開頭與結尾檢索資訊的可靠性，遠高於埋在中段的資訊。理解這個技術底層——Transformer 如何使用位置資訊——是做出正確架構決策、判斷「什麼資訊該放在哪裡」的必要前提。

## 設計思維

**Transformer 的注意力機制具有位置感知性。** 資訊出現在上下文中的哪個位置，直接影響模型能否可靠地檢索它。這不是可調的參數，也不是特定供應商的實作差異，而是 Transformer 訓練方式與注意力梯度流動的結構性特質。

**「迷失在中間」現象（Lost in the Middle）。** Liu 等人（arXiv 2307.03172，發表於 TACL 2023）針對多文件問答與鍵值檢索任務，研究語言模型如何使用長上下文。研究發現：當相關資訊出現在輸入上下文的開頭或結尾時，模型表現最好；而當模型必須從長上下文的中段存取相關資訊時，即便是專門針對長上下文訓練的模型，也會出現顯著的準確率下降。這一現象在上下文位置上呈現出特有的 U 型效能曲線。

**上下文衰退（Context Rot）。** 除了位置效應之外，隨著 token 總數增長，推理品質也會獨立地逐漸退化。隨著上下文長度增加，單一 token 所能獲得的注意力權重會成比例地稀釋——在 100K token 的上下文中，一個 token 獲得的注意力大約是 10K token 上下文中的十分之一。這種注意力稀釋會降低訊號雜訊比，隨著對話累積，輸出品質也會日益下降。Morph（morphllm.com/context-rot）將此現象稱為「上下文衰退」：模型本身沒有損壞，但有用的訊號愈來愈被無關的 token 淹沒。

**RFC 2119 要求：**

- 工程師**不得（MUST NOT）**假設整個上下文視窗的注意力是均勻的。128K token 的視窗，並不代表每個位置都有相同的檢索能力。
- 對位置敏感的關鍵資訊（當前任務描述、核心限制條件、最近一次工具呼叫的結果）**應（SHOULD）**放在上下文靠近結尾的位置，而不是埋在漫長的對話歷史之後。
- 工程師**不得（MUST NOT）**把「放得進上下文視窗」等同於「一定會被可靠地關注到」。資訊存在於視窗中是必要條件，但不是充分條件。

## 深度探討

### 位置編碼（Positional Encoding）

Transformer 本身沒有順序的概念。如果把一串 token 當作集合輸入，模型會對稱地處理它們。位置資訊必須透過位置編碼（positional encoding）明確注入，在計算注意力之前加到 token 嵌入向量上。

原始 Transformer（Vaswani 等人，〈Attention Is All You Need〉，2017 年）使用固定的正弦函數絕對位置編碼：序列中每個位置都被指定一個由不同頻率的正弦與餘弦函數推導出的確定性向量。這種方式在訓練序列長度範圍內運作良好，但無法可靠地外推到更長的序列，因為模型從未在訓練中見過超出其訓練最大長度的絕對位置值。

### 旋轉位置嵌入（RoPE，Rotary Position Embedding）

Su 等人（2021 年）在〈RoFormer: Enhanced Transformer with Rotary Position Embedding〉（arXiv 2104.09864）中提出了旋轉位置嵌入（RoPE）。RoPE 不是將固定的位置向量加到 token 嵌入上，而是在計算注意力之前，對查詢向量（query）與鍵向量（key）施加一個旋轉變換來編碼位置資訊。其核心洞察在於：兩個 token 之間的相對位置，在它們旋轉後的向量內積中得以保留，讓模型能夠自然地推理相鄰程度，而無需依賴絕對位置整數。

RoPE 具備三個使其特別適合長上下文模型的特性：能優雅地處理可變序列長度、token 間的依賴關係隨著相對距離增加而自然衰減（距離較遠的 token 彼此關注程度較低），以及可與線性注意力結合使用。這些特性使 RoPE 成為現代開放權重模型（包括 LLaMA、Mistral、Gemma）中主流的位置編碼方案。

### 滑動視窗注意力（Sliding Window Attention）

標準的自注意力（self-attention）複雜度與序列長度的平方成正比：每個 token 都關注其他所有 token，因此序列長度加倍，計算量就增加為四倍。Beltagy、Peters 和 Cohan（2020 年）在 Longformer 架構（arXiv 2004.05150）中以局部視窗注意力加上任務導向的全域注意力，取代了全域注意力。

在滑動視窗注意力（sliding window attention）中，每個 token 只關注其固定大小的鄰近 token。這將注意力的複雜度從 O(n²) 降低到 O(n)，使任意長度的序列都變得可計算。少數指定的全域 token（通常是任務相關的特殊 token，如 `[CLS]`）仍然關注所有位置，讓模型能夠整合整份文件的資訊。代價是：視窗範圍外的 token 資訊只能透過多個連續層層傳遞，而非在單一注意力操作中一步到位。Longformer 在發表時於多個長文件基準測試上取得了當時的最佳成果。

### 中間位置為何吃虧：機制說明

在標準全域注意力中，U 型檢索曲線有其訓練時期的解釋。在自迴歸訓練過程中，每個後續 token 都會回看之前的所有 token，因此序列早期的 token 從遠比晚期 token 更多的預測步驟中獲得梯度訊號。這讓模型受到更強的訓練壓力，傾向關注上下文開頭附近的位置（早期 token 被許多次預測「見到」）以及結尾附近（近因效應為下一個 token 的直接檢索提供了強烈訊號）。中間位置的 token 兩頭皆不佔：它們既不像早期 token 那樣被大量序列所關注，也不像末尾 token 那樣享有近因偏誤的好處。

結果是模型學會可靠地檢索位於上下文端點的資訊，但對落在兩端之間的內容，檢索訊號則較為嘈雜。這並非規模不足造成的問題——Liu 等人發現，即使在專門針對長上下文訓練的模型中，此現象依然存在。

### 多輪對話的結構性影響

在典型的 API 呼叫中，token 序列的排列順序是：系統提示詞 → 較早的對話輪次 → 最近一條使用者訊息。隨著對話增長，最早的輪次會被推向總 token 序列的開頭和中段。最近的使用者訊息則自然落在結尾——也就是可靠性最高的位置。

實際意義在於：在較長的對話中，早期輪次會逐漸漂移到中段的衰退區。因此，滾動摘要（AEE-108）不只是 token 預算管理，也是位置管理。將舊輪次壓縮為摘要並捨棄逐字的對話記錄，可以將舊資訊從不斷增長的「迷失在中間」區域中移出，並重置近因梯度。如果 Agent 依賴逐字對話歷史來保存關鍵事實，隨著對話增長，即使這些事實仍在上下文視窗範圍內，模型對它們的關注可靠性也會逐漸降低。

## 最佳實踐

1. **將當前任務描述與最關鍵的限制條件放在上下文靠近結尾的位置**——作為最後一條使用者訊息，或緊接在它之前——而不是埋在對話歷史深處。上下文的結尾是檢索可靠性最高的位置。

2. **不要假設更大的上下文視窗就能消除檢索問題。** 一個 128K 視窗中相關內容位於第 60K 位置，其表現將不如一個 32K 視窗中相同內容位於第 28K 位置。「放得進視窗」不等於「會被關注到」。

3. **將上下文長度視為品質風險，而不只是成本風險。** 隨著對話增長，要主動監控輸出品質——品質下降是漸進的，且不會產生任何錯誤訊號。在長對話中運作的 Agent 可能在 API 回應完全正常的情況下，悄悄地遺忘了早期設定的限制條件。
