# [AEE-206] 生產環境的模型選擇

## 情境

大多數團隊在開發時預設使用前沿（frontier）模型，之後就再也不重新審視這個決策。結果是生產系統為用不到的能力付出代價。生產環境的模型選擇是一個需要特定任務量測的實證決策，而不是根據排行榜排名的聲望選擇。

## 設計思維

生產任務中「正確」的模型，是在最低成本和可接受延遲下通過你的特定任務品質門檻的那個。通用基準測試排名是縮小候選清單的起點，而不是選擇標準。MMLU 分數幾乎相同的兩個模型，在你的實際任務上可能產生截然不同的結果。

**能力過剩（capability overshoot）**是最常見的成本失效模式。前沿模型擅長複雜的多步驟推理、模糊指令和全新任務。對於較小模型能可靠處理的任務——分類、擷取、結構化輸入的摘要、格式轉換——它們是過度設計且過度定價的。對這些任務使用前沿模型是不必要的成本與延遲。

**生產模型選擇的三個量測維度**（延伸自 [AEE-111](../Foundations and Mental Models/111) 的框架）：

1. **特定任務品質。** 在你實際任務的代表性樣本上量測。不是基準測試分數——是你的任務。程式碼基準測試預測的是在程式碼基準測試上的表現；它未必能預測你特定程式碼審查任務的表現。
2. **成本。** 輸入 token 單價 × 輸入量 + 輸出 token 單價 × 輸出量，並按適用情況調整快取折扣。是完整的成本結構，而不只是標題的 token 費率。
3. **延遲。** 互動式應用程式的首 token 時間（TTFT，Time to First Token）；批次工作負載的吞吐量（tokens per second）。

**RFC 2119 要求：**

- 模型選擇**必須（MUST）**在生產部署前於代表性任務樣本上驗證。通用基準測試可以用來通知候選清單；它們不能替代特定任務的量測。
- 工程師**絕對不能（MUST NOT）**僅憑通用基準測試排名選擇模型。基準測試表現不能預測特定任務的表現。

## 深度探討

### 批判性解讀基準測試

基準測試分數帶有三個系統性風險，使其作為主要選擇標準時不可靠。

**基準測試汙染（benchmark contamination）。** 若模型的訓練資料包含了基準測試集中的問題或答案，該模型在這個基準測試上的分數，相對於其在真正未見輸入上的表現是虛高的。這是業界記載良好的風險。實際後果是：高基準測試分數並不能可靠地預測模型在未見過的輸入（也就是生產環境中重要的那類輸入）上的表現。

**任務分布不匹配。** 基準測試量測的是其特定任務分布上的表現。頂尖的程式碼基準測試分數量測的是競技程式設計問題、演算法實作等任務的表現，而不是你的程式碼審查工作流程、內部 API 文件生成，或測試案例合成任務。領域與任務的特定性非常重要。

**基準測試飽和。** 當多個前沿模型在某個基準測試上趨近天花板表現（超過 90%）時，這個基準測試就失去了區分能力。接近天花板的分數細微差異，反映的可能是量測雜訊而非真正的能力差距。即時的特定任務評估（AEE-800 系列涵蓋此主題）才是生產決策的適當工具。使用 Artificial Analysis（<https://artificialanalysis.ai/>）追蹤跨供應商的即時多維度效能數據——包括延遲、吞吐量和品質——比靜態排行榜快照更有用。

### 成本建模實務

Token 計費需要分別對輸入和輸出建模，因為供應商通常對同一模型的兩者收取不同費率。輸出 token 比輸入 token 貴。

**提示詞快取（prompt caching，Anthropic）：** 對於相同的大型提示詞前綴在多個請求中重複使用的工作負載——系統提示詞、文件情境、少樣本範例——提示詞快取可以大幅降低輸入成本。快取讀取（cache read）的計費為標準輸入 token 單價的 0.1 倍（九折優惠）。快取寫入（cache write）對 5 分鐘 TTL 收費為標準輸入 token 單價的 1.25 倍，1 小時 TTL 則為 2 倍。淨成本降低取決於快取命中率：根據你預期的提示詞結構和請求量，明確地建模這個數字。

**批次 API（batch API，OpenAI 及其他供應商）：** 許多供應商以即時定價的約 50% 提供批次處理，並有 24 小時的延遲 SLA。對於不受延遲限制的工作負載——大量文件處理、離線評估、每日資料強化——批次定價可以將每次請求成本減半，且不犧牲任何品質。OpenAI 批次 API（<https://platform.openai.com/docs/guides/batch>）是參考實作；其他供應商有同等方案。

在高量工作負載上，候選模型之間 2 倍的價格差距會累積成顯著的絕對成本。一個成本低一半且通過品質門檻的模型，即使在基準測試上分數較低，也是正確的選擇。

### 延遲建模

**首 token 時間（TTFT，Time to First Token）**是從發送請求到收到第一個回應 token 的時間。對於串流應用程式——聊天介面、互動工具、即時分析——TTFT 決定了感知的回應速度。使用者在生成完成之前就看到了第一個 token，因此 TTFT 比總生成時間更能驅動使用者體驗。

**吞吐量**（第一個 token 之後的 tokens per second）對長篇生成很重要：報告、摘要、文件。吞吐量較高的模型在更短的時間內產出相同的輸出。

較小的模型在相同硬體預算下，通常具有更低的 TTFT 和更高的吞吐量。若你的任務對延遲敏感，且較小的模型通過了品質門檻，則延遲差異往往足以支撐選擇決策，不論成本差異如何。

### 供應商鎖定風險

供應商在 API 結構、認證、可用模型、速率限制、棄用時程和定價上各有不同。這造成了在專案初期容易低估的**供應商鎖定（vendor lock-in）**風險。

**API 格式差異**意味著除非加以抽象，否則切換供應商需要修改程式碼。OpenAI、Anthropic 和 Gemini 儘管有部分標準化的努力，仍有不同的請求和回應格式。

**模型棄用**是例行之事。供應商通常在退役某個模型版本前提供 3–6 個月的通知，但若替代模型產生不同的輸出特性，遷移時程並不總是可以商量。

**定價變更**不需事先通知。今天經濟實惠的模型可能重新定價。

**緩解方式：使用抽象層。** LiteLLM（<https://github.com/BerriAI/litellm>）為 100 多個 LLM 供應商提供統一的 OpenAI 相容介面。對 `completion(model="anthropic/claude-sonnet-4-20250514", ...)` 的呼叫使用與 `completion(model="openai/gpt-4o", ...)` 相同的介面。切換供應商或模型變成了設定變更，而非程式碼變更。這也讓在生產環境中對模型進行 A/B 測試成為可能，無需修改應用程式層。

### 生產環境中的路由（Routing）

生產系統中並非每個請求都有相同的複雜度。要求一句話摘要的使用者查詢，與需要綜合多份文件中相互矛盾資訊的查詢，是不同的任務。對所有請求一視同仁，意味著為簡單請求支付前沿模型的費用。

**LLM 路由（routing）**透過根據複雜度或類型將請求導向不同模型層級來解決這個問題。關於統一路由與串聯方法的研究（arXiv:2410.10347）顯示，結合路由和串聯策略，在維持品質的同時實現了顯著的成本降低。論文證明串聯路由（cascade routing）持續優於單獨使用任何單一模型。

**基於分類器的路由：** 一個小型、快速的模型將每個傳入請求按類型或複雜度分類，並路由到適當的層級。簡單請求前往較便宜的模型；複雜請求則升級到前沿層級。

**基於成本的路由：** 路由到歷史上已通過特定請求類別品質門檻的最便宜模型。在生產中追蹤每個類別的品質指標以維護路由表。

**降級鏈（Fallback chains）：** 若模型 A 失敗（超時、錯誤或輸出低於品質門檻），則在模型 B 上重試。這提供了對供應商中斷的韌性，以及在較便宜的模型無法處理邊緣案例時的優雅降級。

## 最佳實踐

1. **在為生產選擇模型之前，從你的實際任務中建立 50–200 個範例的代表性評估集。** 針對它執行所有候選模型。以最低成本通過你品質門檻的模型勝出。通用基準測試排名用來通知候選清單；特定任務評估決定最終選擇。

2. **在做出決定之前，對完整成本結構建模。** 計入輸入單價、輸出單價、若工作負載結構支援則計入快取折扣，若延遲需求允許則計入批次折扣。在高量工作負載上，2 倍的價格差距會累積成顯著的絕對成本——在生產部署前明確計算這個數字是值得的。

3. **從一開始就建立供應商抽象層。** 切換模型或供應商應該是設定變更，而非程式碼變更。這也讓在生產環境中對模型進行 A/B 測試成為可能。LiteLLM 是希望擁有統一介面而不自行建構抽象層的團隊的實務起點。
