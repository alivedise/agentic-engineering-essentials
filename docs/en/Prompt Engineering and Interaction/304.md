# [AEE-304] Instruction Following

## Context

When a model does not follow an instruction, the instinct is to blame the model. In most cases, the instruction is the problem. Models trained on instruction-following data are very capable of following clear, specific, internally consistent instructions. What they reliably fail at is parsing instructions that are ambiguous, contradictory, excessively long, or that assume shared context they do not have. Diagnosing instruction following failures requires reading the instruction as the model parses it — not as the engineer intended it.

## Design Think

The core claim: instruction following failures are usually the engineer's fault, not the model's. Most failures trace to ambiguous verbs, contradictory constraints, excessive length, or instructions that assume shared context the model does not have.

**What makes an instruction parse reliably:**

1. **Specific action verbs.** "Summarize" is unambiguous: produce a shorter representation of the input. "Handle" is not: does it mean respond to, forward, classify, resolve, or log? Unambiguous action verbs: summarize, extract, classify, rewrite, translate, evaluate, list, compare, generate. Ambiguous action verbs: handle, process, work with, deal with, manage, address.

2. **Explicit scope.** "Summarize the document" leaves open: which sections? To what length? In what format? "Summarize the Executive Summary and Key Findings sections of the document in three bullet points, each under 20 words" is scoped.

3. **Unambiguous success criteria.** "Make this clearer" is not an instruction — it is a direction without a definition of what "clear" means. "Rewrite this paragraph so that a reader with no domain knowledge can understand the main claim in one reading" defines the success criterion.

4. **No implicit shared context.** Instructions that refer to "our style guide," "the usual format," or "standard practice" assume the model knows what these mean. It does not unless that context is provided explicitly.

**Failure patterns:**

1. **Length overload.** A single instruction block containing more than 7–10 distinct constraints is difficult for the model to track across a long response. The model does not fail to understand the instructions — it succeeds at the first several and deprioritizes the rest. Mitigation: reduce constraint count or use numbered lists that help the model track what it has applied.

2. **Internal contradiction.** Instruction A says "be brief" while instruction B says "cover all relevant considerations." The model must choose one — and it will choose based on recency and salience heuristics rather than the engineer's intent. Mitigation: audit instructions for contradictions before deployment.

3. **Implicit context.** The instruction assumes the model knows something it does not: the domain, the company's conventions, the reader's background, the purpose of the output. Mitigation: make implicit assumptions explicit in the context block (AEE-301).

4. **Format ambiguity.** "Respond formally" does not specify what formal means in this context. Formal can mean no contractions, can mean legal register, can mean bullet points only, or can mean specific section headers. Mitigation: show a format example or describe format requirements mechanically.

**Priority when instructions conflict:**

Models use two heuristics when multiple instructions could conflict: recency (later instructions in the prompt carry more weight) and salience (explicitly marked priorities such as "Most important:" or numbered lists carry more weight than prose). Engineers can exploit these heuristics: place the most important constraints last in the instruction block, and mark priority explicitly when conflict is possible.

- Instructions MUST be reviewed for internal contradictions before deployment. Two engineers should be able to read the instruction set independently and agree on what a compliant response looks like.
- Instructions SHOULD use explicit priority markers ("Most important:", "If in doubt, prioritize X over Y") when multiple constraints could conflict in production inputs.
- Instructions SHOULD use specific action verbs. Instructions using vague verbs (handle, process, address) MUST be rewritten before deployment.

## Deep Dive

### The Instruction Audit

An instruction audit is a structured review of an instruction set before deployment. It checks for four failure modes:

1. **Ambiguity check:** For each instruction, can it be interpreted in two or more ways? If yes, pick one interpretation and make it explicit.

2. **Contradiction check:** Does any instruction conflict with any other? Common conflicts: brevity vs. completeness, formality vs. accessibility, speed vs. accuracy. If a conflict exists, resolve it with an explicit priority statement.

3. **Implicit context check:** Does any instruction assume something the model cannot infer from the prompt? If yes, move that assumption to the context block.

4. **Scope check:** Does every instruction have an explicit scope? "Extract key points" is unscoped. "Extract up to five key points, each as a single sentence under 15 words" is scoped.

Running the audit before deployment catches the majority of instruction following failures at their source rather than in production.

### Worked Example

**Instruction set with three failure patterns:**

```
System: You are an AI assistant. Help users with their questions. 
Be professional but friendly. Always be thorough in your responses 
and cover all relevant information. Keep responses concise. 
Use our standard format. Make sure to address any follow-up questions 
the user might have.
```

Failure patterns present:
- **Contradiction:** "thorough... cover all relevant information" conflicts with "keep responses concise"
- **Implicit context:** "our standard format" — the model does not know what this is
- **Implicit context:** "follow-up questions the user might have" — anticipating unstated questions is impossible without constraints
- **Vague verb:** "Help users" — help how? Explain? Fix? Classify? Route?

**Rewritten instruction set:**

```
System: You are a customer support specialist for a SaaS product. 
Answer questions about account management, billing, and product features.

For each response:
1. Answer the stated question directly in the first paragraph.
2. If the answer requires steps, use a numbered list.
3. End with one sentence offering to clarify if needed.
4. Maximum 200 words.

Do not speculate about questions the user has not asked.
Do not provide information about competitor products.
```

Changes made:
- Role is specific (customer support specialist, scoped domain)
- "Help" replaced by "Answer questions about [scoped domain]"
- Contradiction resolved: numbered format plus 200-word limit replaces "thorough" and "concise"
- "Standard format" replaced by explicit structural rule
- "Follow-up questions" removed; replaced by specific offer-to-clarify sentence

## Best Practices

1. **Run the two-engineer test on every instruction set before deployment.** Have a colleague read your instructions independently and describe what a compliant response looks like. If their description differs from yours, the instructions are ambiguous — and the model will produce the same kind of variance.

2. **Prefer numbered lists over prose for instruction sets with more than three constraints.** Numbered lists make it easy for both you and the model to track which constraints have been applied. Prose instructions with multiple constraints embedded in paragraphs are harder to audit and harder for the model to enumerate.

3. **Fix instruction following failures by improving the instruction, not by adding more instructions.** When the model does not comply with an instruction, the instinct is to add "I said to [X]!" Adding instructions to fix non-compliance makes the instruction set longer and more contradictory. Instead, diagnose why the original instruction failed (ambiguity, contradiction, implicit context) and fix it at the source.
