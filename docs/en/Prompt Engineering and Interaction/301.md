# [AEE-301] Prompt Structure Fundamentals

## Context

Most prompt failures are structural. The model produces inconsistent output not because it is unreliable, but because the prompt has not communicated intent precisely enough — it uses vague action verbs, omits output format constraints, or conflates information that should be in separate components. Engineers who understand prompt anatomy can diagnose these failures systematically and fix them at the source rather than adding workarounds downstream.

## Design Think

The core claim: a prompt is not free-form text — it is a structured artifact with components that serve distinct engineering functions. Prompts without deliberate structure produce inconsistent results not because the model is unreliable, but because the engineer has not communicated intent unambiguously.

A well-formed prompt has four components:

1. **Role/persona** — who the model is in this interaction. Sets behavioral defaults, tone register, and the implied knowledge domain. "You are a senior software engineer reviewing a pull request" activates different response patterns than no role specification.

2. **Context block** — the information the model needs to complete the task that it cannot infer from the instruction alone. The current document, the user's situation, the constraints of the system it is operating in. The context block is scoped: it contains only what is relevant to this task, not a general knowledge dump.

3. **Instruction** — what the model must do. One primary action verb (summarize, extract, classify, rewrite, evaluate) with explicit scope and success criteria. The instruction is the contract: it must be specific enough that the model cannot satisfy it in multiple incompatible ways.

4. **Output format constraint** — how the output must be structured. JSON schema, markdown section headers, maximum length, required fields. This component is the one most often omitted and the one that causes the most downstream failures when missing.

**How components map to turns:**

The system turn holds the role and stable context (instructions that apply to all interactions). The user turn holds task-specific context and the instruction. The assistant prefill (when used) sets the output format by beginning the response in the required format.

**Component interaction:**

A well-written instruction can be undermined by a conflicting output format constraint. "Explain this concept clearly" conflicts with "respond in JSON." A missing output format constraint causes the model to choose its own format, which changes across runs.

**Negative space:**

Over-specified prompts are as fragile as under-specified ones. A 2,000-token system prompt that enumerates every possible edge case signals an engineer who does not trust the model — and produces a prompt that is sensitive to the exact wording of those enumerations. The right specification level is: enough to eliminate ambiguity, not enough to constrain the model's reasoning.

**RFC 2119:**

- Prompts deployed in production MUST have explicit output format constraints. A prompt without an output format constraint is a prompt that will produce different formats on different runs.
- Instructions SHOULD use unambiguous action verbs (summarize, extract, classify, rewrite) rather than vague directives (handle, process, work with, deal with).
- Role/persona specifications SHOULD be included when the task requires a specific behavioral register or knowledge domain. They MAY be omitted for simple, well-scoped extraction tasks where role does not materially affect output.

## Deep Dive

### Component Anatomy in Detail

**Role/persona:** The most effective role specifications are specific and grounded: "You are a technical writer reviewing documentation for a developer audience" works better than "You are a helpful assistant." The role sets implicit priors about tone, vocabulary, assumed reader knowledge, and what a good response looks like. Role specifications work because they activate training patterns associated with that role's communication style.

Role specifications do not add capabilities the model does not have. Telling the model it is an expert does not make it more accurate on tasks outside its training distribution. What they do is shift the model toward a more consistent behavioral register.

**Context block:** The context block is the place for grounding information: the document to be summarized, the code to be reviewed, the user's stated goal, the relevant constraints of the system. It is not the place for general background knowledge (which the model has from training) or instructions (which belong in the instruction component).

A common structural error is mixing context and instruction: "Here is the contract (context). Review it and identify any clauses that create liability risk (instruction). Also, our jurisdiction is California (more context) and we prefer plain language summaries (output format constraint)." Separating these into explicit components makes the prompt easier to test, version, and debug.

**Instruction:** The instruction component must pass what can be called the "two-engineer test": would two engineers, reading this instruction independently, agree on what a correct response looks like? If not, the instruction is underspecified. Common failure modes: vague verbs ("process," "handle"), missing scope ("summarize the document" — which sections? to what length?), and absent success criteria ("make this better" — better by what measure?).

**Output format constraint:** Output format constraints should specify: the structure (JSON object, markdown list, plain paragraph), the required fields or sections, and any length or count constraints. For JSON outputs, include the schema. For markdown outputs, show an example of the required section structure.

### Worked Example

**Weak prompt (no deliberate structure):**

```
Help me with this email. I need to respond to a customer complaint about a delayed shipment.
The customer is upset and wants a refund. Here is their email: [EMAIL]
```

Problems: No role. No output format constraint. "Help me with" is not an action verb. The word "refund" is mentioned without a constraint on whether to offer it. The model may respond with an email draft, a strategy, bullet points, or a mix.

**Restructured prompt (four components explicit):**

```
[Role]
You are a customer service specialist at an e-commerce company. You write professional,
empathetic responses that de-escalate situations while following company policy.

[Context]
Company policy: issue a 10% discount on the next order for delays under 7 days;
offer a full refund for delays over 7 days. This shipment is 4 days delayed.

[Instruction]
Write a response to the following customer complaint email that acknowledges the delay,
applies the correct policy, and closes on a positive note.

[Output format]
Respond with a single email draft. Subject line on the first line, blank line, then body.
150 words maximum. No bullet points.

[Customer email]
[EMAIL]
```

What changed: The role activates a customer service communication register. The context block contains the policy constraint (preventing the model from inventing a refund policy). The instruction uses a specific action verb with scope. The output format constraint specifies structure, length, and format.

## Best Practices

1. **Write the output format constraint before the instruction.** Deciding what the output must look like forces clarity about what the instruction is actually asking for. Engineers who write instructions first often produce ambiguous instructions because they have not yet committed to what success looks like.

2. **Test each component in isolation.** Change the role and see if output quality shifts. Remove the output format constraint and observe the variance. Add a conflicting instruction and see if the model detects it. Component-level testing reveals which parts of the prompt are load-bearing.

3. **Minimize the context block to what the current task actually needs.** Every token in the context block is a token not available for reasoning. If the same background information appears in every prompt, it belongs in the system prompt. If it is task-specific, include only the task-relevant subset.
