# [AEE-706] Error Recovery

## Context

Agent loops fail. Tools time out. The model refuses to act. History grows until it no longer fits the context window. The model gets stuck and repeats itself. These are not edge cases -- they are expected conditions in any production agent deployment. A harness that stops on the first failure is fragile. A harness that maintains the loop's ability to make progress despite errors is resilient.

Error recovery is not about preventing failures. It is about maintaining the loop's ability to make progress despite them.

## Design Think

**Resilience** is not about preventing errors -- it is about maintaining the loop's ability to make progress despite errors. A harness that stops on first failure is fragile; a harness with an error budget stays useful.

The error budget pattern is central: track consecutive failures. After N consecutive failures, escalate to a human or terminate with a diagnostic. This prevents infinite retry loops -- one of the most dangerous harness failure modes. An agent that retries indefinitely wastes resources and may cause compounding damage.

**RFC 2119:**

- The harness MUST NOT retry a failed tool call without a delay or a changed condition. Immediate retry of a failing call is not recovery -- it is amplification.
- The harness MUST terminate the loop when the error budget is exhausted. A harness that continues past budget exhaustion cannot distinguish a stuck loop from a healthy one.
- The harness SHOULD log every error and its recovery action. Without a record of what failed and what was attempted, post-mortem analysis is impossible.

## Deep Dive

### Error Taxonomy

| Error type | Cause | Recoverable? |
|---|---|---|
| Tool execution failure | Tool returns error result or throws exception | Usually (if transient) |
| Model refusal | Model declines to act (safety filter, capability limit) | Rarely |
| Context overflow | History exceeds context window | Yes — with truncation or summarization |
| Loop stall | Model produces no useful output across N turns | Sometimes |
| Timeout | Tool or model call exceeds time limit | Usually (if transient) |

### Per-Error Recovery Strategies

**Tool execution failure:**
Return `is_error: true` in the tool result. Let the model decide whether to retry, use a different approach, or report the failure. For transient errors, add backoff before retry.

```python
def handle_tool_failure(error: Exception, tool_name: str, attempt: int) -> ToolResult:
    is_transient = isinstance(error, (RateLimitError, NetworkError, TimeoutError))
    if is_transient and attempt < MAX_RETRIES:
        time.sleep(BACKOFF_SECONDS * (2 ** attempt))  # exponential backoff
        return None  # signal retry
    return ToolResult(
        is_error=True,
        content=f"{tool_name} failed: {str(error)}. Attempt {attempt}/{MAX_RETRIES}."
    )
```

**Context overflow:**
1. Summarize the oldest N turns into a compressed form.
2. Replace the summarized turns in history with the summary.
3. Continue the loop with the compressed history.

**Loop stall:**
1. Detect stall (see AEE-701 stall detection).
2. Inject recovery prompt: "You appear to be stuck. What is blocking you? Try a different approach."
3. If no progress after M additional turns, terminate with diagnostic.

**Timeout:**
Cancel the tool call, return `is_error: true` with a timeout message, continue the loop. Let the model decide whether to retry or proceed differently.

**Model refusal:**
Model refusals on a task indicate the task or the tool combination is outside the model's operational boundaries. Retrying the same prompt will not succeed. Log the refusal, escalate to a human, and terminate the loop.

### The Error Budget Pattern

Track consecutive failures. After N consecutive failures, escalate:

```python
class ErrorBudget:
    def __init__(self, max_consecutive: int = 3):
        self.max_consecutive = max_consecutive
        self.consecutive_failures = 0

    def record_success(self):
        self.consecutive_failures = 0

    def record_failure(self) -> bool:
        """Returns True if budget is exhausted."""
        self.consecutive_failures += 1
        return self.consecutive_failures >= self.max_consecutive

    def is_exhausted(self) -> bool:
        return self.consecutive_failures >= self.max_consecutive
```

When the budget is exhausted: terminate the loop and return a diagnostic message describing the last N failures and suggesting next steps.

### Recoverable vs. Unrecoverable Errors

| Error | Recoverable? | Recovery action |
|---|---|---|
| Rate limit exceeded | Yes | Backoff and retry |
| Network timeout | Yes | Retry with backoff |
| Tool returns unexpected format | Sometimes | Let model retry with rephrased input |
| Model refuses on every turn | No | Terminate; escalate to human |
| Context window exhausted after summarization | No | Terminate; checkpoint and restart in new session |
| Tool unavailable (deployment issue) | No (until fixed) | Terminate; alert on-call |

## Best Practices

1. **Implement the error budget as a first-class object, not a counter.** An error budget that is just an integer is easy to forget to reset. A first-class `ErrorBudget` object with `record_success()`, `record_failure()`, and `is_exhausted()` methods is explicit, testable, and impossible to misuse. Reset it on every success, not just on loop restart.

2. **Never retry immediately on failure.** Immediate retry of a failing call amplifies the failure. A minimum of one second backoff on first retry, doubling on each subsequent retry, is the baseline. Set a maximum backoff (e.g., 60 seconds) to prevent indefinitely long stalls.

3. **Terminate with a diagnostic, not a silent failure.** When the error budget is exhausted, return a diagnostic that describes what was tried, how many times, and what failed. This gives the user enough information to decide whether to retry, reframe the task, or escalate.
