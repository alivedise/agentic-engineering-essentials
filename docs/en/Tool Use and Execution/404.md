# [AEE-404] Browser and Computer Use

## Context

Browser and computer use enable agents to interact with the world through interfaces designed for humans. Where function calling invokes an API the engineer defined, browser and computer use operate on interfaces the engineer did not build and cannot control: websites, desktop applications, forms, and GUIs. This capability fills the gaps that structured APIs cannot — legacy systems, third-party sites, and interfaces that only expose a human-facing UI.

The trade-off is blast radius. A function calling tool can only do what its implementation allows. A computer use agent that can click, type, and navigate can do anything a human user can do — including actions that are destructive, irreversible, or outside the intended scope. Safety architecture is not optional for these tool types.

## Design Think

The core claim: browser and computer use enable agents to interact with the world through interfaces designed for humans — but they operate with much higher blast radius than function calling. They require explicit safety architecture before deployment.

**Browser automation vs. computer use:**

| | Browser Automation | Computer Use |
|---|---|---|
| Interface | Programmatic (DOM) | Visual (screenshot + actions) |
| Reliability | High — deterministic selectors | Lower — depends on screenshot quality |
| Scope | Web only | Any visible application |
| Token cost | Low (no screenshots) | High (screenshot per action) |
| Use when | Stable HTML structure exists | Legacy apps, non-web GUIs |

**The screenshot-action loop:**

Computer use operates as a perception-action loop:
1. The harness takes a screenshot of the current screen state
2. The screenshot is sent to the model as an image content block
3. The model decides on an action: click, type, scroll, key, or screenshot
4. The harness executes the action
5. A new screenshot is taken and the loop continues
6. The loop ends when the model emits a text response indicating task completion

Each iteration sends a full screenshot as a base64-encoded image — typically hundreds to thousands of tokens depending on resolution. Tasks with many screen interactions can incur substantial token costs. Estimate cost before deploying.

**When NOT to use browser/computer use:**

- When a stable API exists: use function calling instead. Computer use to interact with a site that has a usable API is unnecessary complexity.
- When the target interface changes frequently: DOM selectors and click coordinates break on UI updates.
- In production without an isolated environment: both tool types must run in isolation.

**RFC 2119:**

- Computer use agents MUST run in an isolated environment (VM, container) separate from the host system. A computer use agent on an uncontrolled desktop has access to every application, credential, and file visible on screen.
- Browser automation SHOULD be scoped to specific domains rather than granted unrestricted web access. An agent with unrestricted browser access can navigate to sites designed to execute prompt injection via page content.
- Token cost for computer use tasks MUST be estimated before production deployment. Tasks requiring many screen interactions have non-trivial token costs that are not apparent from the task description alone.

## Deep Dive

### Playwright for Browser Automation

Playwright provides programmatic control over Chromium, Firefox, and WebKit. It is the right choice for structured web automation where the target site has stable HTML.

```python
from playwright.sync_api import sync_playwright

def extract_product_prices(url: str) -> list[dict]:
    with sync_playwright() as p:
        browser = p.chromium.launch(headless=True)
        page = browser.new_page()
        page.goto(url)
        page.wait_for_load_state("networkidle")

        products = page.query_selector_all(".product-card")
        results = []
        for product in products:
            name = product.query_selector(".product-name").inner_text()
            price = product.query_selector(".price").inner_text()
            results.append({"name": name, "price": price})

        browser.close()
        return results
```

Playwright strengths:
- Deterministic: CSS selectors are stable when the HTML structure is stable
- Fast: no screenshot overhead; DOM queries are direct
- Structured extraction: returns typed data rather than raw text
- Network interception: can mock or intercept requests

### browser-use for Agent-Friendly Automation

browser-use is a library built on Playwright that provides a higher-level interface for LLM-driven web navigation. It handles the translation between natural language task descriptions and browser actions.

```python
from browser_use import Agent
from langchain_anthropic import ChatAnthropic

agent = Agent(
    task="Find the current price of the MacBook Pro 16-inch on apple.com",
    llm=ChatAnthropic(model="claude-opus-4-6"),
)

result = await agent.run()
print(result)
```

browser-use is appropriate when the task requires multi-step navigation with decision-making rather than a fixed extraction pattern.

### Anthropic Computer Use

Anthropic's computer use capability enables Claude to interact with a full desktop environment through screenshots and actions. It uses the standard messages API with a beta header.

Available action types:
- `screenshot`: capture the current screen (used to observe)
- `left_click`, `right_click`, `double_click`: click at specified screen coordinates
- `type`: type a string of text
- `key`: press a keyboard key or combination (e.g., `"ctrl+c"`)
- `scroll`: scroll at a position by a given amount
- `mouse_move`: move the cursor without clicking

```python
import anthropic
import base64

client = anthropic.Anthropic()

def computer_use_loop(task: str, take_screenshot, execute_action) -> str:
    """
    take_screenshot: () -> base64-encoded PNG string
    execute_action: (dict) -> None
    """
    messages = [{"role": "user", "content": task}]

    while True:
        screenshot = take_screenshot()
        messages[-1]["content"] = [
            {
                "type": "text",
                "text": task if len(messages) == 1 else "Current screen state:"
            },
            {
                "type": "image",
                "source": {
                    "type": "base64",
                    "media_type": "image/png",
                    "data": screenshot
                }
            }
        ]

        response = client.beta.messages.create(
            model="claude-opus-4-6",
            max_tokens=1024,
            tools=[{
                "type": "computer_20251124",
                "name": "computer",
                "display_width_px": 1280,
                "display_height_px": 800
            }],
            messages=messages,
            betas=["computer-use-2025-11-24"]
        )

        messages.append({"role": "assistant", "content": response.content})

        if response.stop_reason == "end_turn":
            for block in response.content:
                if block.type == "text":
                    return block.text

        tool_results = []
        for block in response.content:
            if block.type == "tool_use" and block.name == "computer":
                execute_action(block.input)
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": ""  # screenshot appended in next iteration
                })

        messages.append({"role": "user", "content": tool_results})
```

**Safety setup:** Computer use agents must run in an isolated environment. The recommended approach is a Docker container or VM with no access to the host system's files, credentials, or applications. Anthropic's computer use documentation includes a reference Docker setup.

## Best Practices

1. **Default to Playwright over computer use when the target has stable HTML.** Playwright is faster, cheaper, more reliable, and easier to debug. Computer use is the right choice when structured automation is not available — not as a general-purpose browser strategy.

2. **Scope browser automation to the minimum required domains.** Configuring Playwright to only navigate to specific origins reduces the risk of prompt injection via page content. An agent that can navigate anywhere can be directed to a page designed to override its instructions.

3. **Run computer use in a container with no access to host credentials.** The most common deployment mistake is running computer use on a developer's machine during testing and then moving to production without changing the isolation setup. Test with the same isolation configuration that will run in production.
