# [AEE-403] Code Execution

## Context

Code execution is the most capable and most dangerous tool type in an agent's repertoire. It enables arbitrary computation: data analysis, algorithm execution, dynamic logic, generating and running code to solve problems the model cannot solve in-context. It also executes arbitrary code from a source that cannot be fully trusted — the model's output. The two facts are inseparable.

Engineers who add code execution to an agent without a sandbox are not making a performance trade-off. They are giving the model the ability to run arbitrary code in their application's process, with their process's permissions, with access to their filesystem and network.

## Design Think

The core claim: code execution is the most capable and most dangerous tool type. It can perform arbitrary computation, but it requires a sandbox — executing model-generated code in an uncontrolled environment is a critical security failure.

**Code execution vs. function calling:**

Both are tool types. The difference is specificity:
- **Function calling** invokes a known, predefined operation with a defined schema. The model specifies arguments to a function the engineer wrote and reviewed.
- **Code execution** invokes an interpreter with arbitrary code the model generated. The model specifies the code itself — and the code can do anything the interpreter allows.

Use function calling when the operation is well-defined and bounded. Use code execution when the task requires dynamic computation that cannot be expressed as a fixed function: data transformations on unknown input structure, algorithm execution for problems stated at runtime, or mathematical computation the model cannot perform in-context.

**The sandbox requirement:**

A code sandbox enforces isolation between the executing code and the host system. At minimum, a sandbox must enforce:
- **Filesystem isolation**: code can only read/write designated paths; it cannot access the host filesystem
- **Network isolation**: outbound connections are blocked or limited to a defined allowlist; code cannot exfiltrate data
- **Process isolation**: code runs in a separate process or container from the host application; it cannot access host memory or credentials
- **Resource limits**: CPU time, memory, and execution timeout are bounded; runaway code cannot exhaust host resources

The question is not whether to sandbox — it is which provider fits the use case.

**RFC 2119:**

- Model-generated code MUST NOT be executed outside a sandboxed environment. Executing model output in the host process is a critical security failure regardless of prompt constraints.
- Sandboxes MUST enforce resource limits including CPU time, memory, execution timeout, and network access policy. A sandbox without resource limits is not a sandbox.
- Code execution tools MUST return stdout, stderr, and exit code as structured output. The model needs the full execution context to determine whether to retry or report failure.

## Deep Dive

### Provider Options

**E2B (Cloud Sandbox)**

E2B provides on-demand isolated sandbox environments for code execution. Each sandbox runs in an isolated container with a Python or JavaScript runtime, isolated filesystem, and configurable network policy.

```python
import e2b_code_interpreter as e2b

sandbox = e2b.Sandbox()

execution = sandbox.run_code("""
import pandas as pd
import json

data = [{"name": "Alice", "score": 92}, {"name": "Bob", "score": 78}]
df = pd.DataFrame(data)
result = df.sort_values("score", ascending=False).to_dict(orient="records")
print(json.dumps(result))
""")

print(execution.text)   # stdout
print(execution.error)  # stderr (if any)
sandbox.kill()
```

E2B characteristics:
- Network: configurable; default allows outbound HTTPS to package registries
- Persistence: sandboxes persist for a session and share state across multiple code blocks in the same session
- Cost: billed per compute-second

**Modal (Serverless Containers)**

Modal runs containerized workloads on demand with configurable CPU, memory, and optional GPU. For code execution in agents, Modal provides ephemeral function execution with strong isolation.

```python
import modal

app = modal.App()

@app.function(
    cpu=1,
    memory=512,
    timeout=30,
    network_file_systems={}  # no persistent storage
)
def execute_code(code: str) -> dict:
    import subprocess, sys
    result = subprocess.run(
        [sys.executable, "-c", code],
        capture_output=True, text=True, timeout=25
    )
    return {
        "stdout": result.stdout,
        "stderr": result.stderr,
        "returncode": result.returncode
    }

with app.run():
    output = execute_code.remote("print(2 + 2)")
```

Modal characteristics:
- GPU support: available for ML workloads requiring GPU compute
- Container images: fully configurable; install any packages
- Isolation: each invocation is a fresh container unless persistence is explicitly configured
- Default timeout: 5 minutes; maximum 24 hours

**Docker (Self-Hosted)**

For teams that cannot send data to external providers, Docker containers provide equivalent isolation on self-hosted infrastructure:

```python
import docker

client = docker.from_env()

def run_in_sandbox(code: str, timeout: int = 10) -> dict:
    try:
        output = client.containers.run(
            "python:3.11-slim",
            command=["python", "-c", code],
            remove=True,
            mem_limit="128m",
            cpu_quota=50000,       # 50% of one CPU
            network_disabled=True,
            read_only=True,
            tmpfs={"/tmp": "size=64m"},
            timeout=timeout
        )
        return {"stdout": output.decode("utf-8"), "returncode": 0}
    except docker.errors.ContainerError as e:
        return {"stdout": "", "stderr": str(e), "returncode": e.exit_status}
```

Adjust `cpu_quota` and `mem_limit` based on your tasks — 50,000 (50% of one CPU) and 128MB memory are conservative defaults suitable for untrusted code. Data analysis tasks may require higher limits.

Key Docker flags:
- `--network=none` / `network_disabled=True`: blocks all network access
- `mem_limit`: enforces memory ceiling
- `cpu_quota`: limits CPU usage
- `--read-only`: root filesystem is read-only
- `tmpfs /tmp`: writable in-memory temp directory

### Code Execution as a Tool

Defining code execution within the function calling protocol:

```json
{
  "name": "execute_python",
  "description": "Execute Python code in a sandboxed environment and return the output. Use for data analysis, mathematical computation, algorithm execution, or any task requiring dynamic computation. The code runs in an isolated container. Network access is restricted — external connections are blocked except to package registries. Only use when the computation cannot be expressed as a direct answer.",
  "input_schema": {
    "type": "object",
    "properties": {
      "code": {
        "type": "string",
        "description": "Valid Python code to execute. Print results to stdout — expression return values are not captured. Available packages: numpy, pandas, matplotlib. Maximum 10 seconds execution time."
      }
    },
    "required": ["code"]
  }
}
```

### Output Handling

Return full execution context as a structured tool result:

```json
{
  "type": "tool_result",
  "tool_use_id": "toolu_01ABC",
  "content": "stdout:\n[{\"name\": \"Alice\", \"score\": 92}, {\"name\": \"Bob\", \"score\": 78}]\n\nstderr:\n(empty)\n\nexit_code: 0"
}
```

If execution failed:

```json
{
  "type": "tool_result",
  "tool_use_id": "toolu_01ABC",
  "is_error": true,
  "content": "stdout:\n(empty)\n\nstderr:\nNameError: name 'df' is not defined\n\nexit_code: 1"
}
```

Returning stdout, stderr, and exit_code gives the model enough context to decide whether to retry with corrected code.

## Best Practices

1. **Pin the available package list in the tool description.** The model needs to know which packages it can import. A description that says "available packages: numpy, pandas, matplotlib, json" prevents the model from importing packages that are not installed in the sandbox, avoiding avoidable NameErrors.

2. **Use a fresh sandbox per stateless task; reuse for multi-step sessions.** Sandboxes that persist state across multiple tool calls are useful for multi-step data analysis where each step builds on the previous one. For independent tasks, use a fresh sandbox per call to avoid state leakage between invocations.

3. **Return exit code alongside stdout.** A model that only receives stdout cannot distinguish a clean zero-output execution from a silent failure. Always return exit code. On non-zero exit, include stderr.
