# [AEE-402] Tool Schema Design

## Context

A tool schema is what the model reads to decide whether to call a tool and what arguments to pass. The `description` field is not documentation for human readers — it is the model's primary signal for tool selection and invocation. Parameter descriptions are not optional metadata — they are the model's instructions for what value to provide in each field.

Most tool invocation failures trace to schema quality, not model capability. A model that invokes the wrong tool, passes incorrect argument types, or omits required parameters is almost always responding to an ambiguous or incomplete schema, not failing at tool use.

## Design Think

The core claim: a tool schema is a natural language interface, not just a type contract. The model uses parameter descriptions to decide when and how to invoke a tool — poorly written descriptions cause mis-invocation more often than type mismatches.

**Name design:**

Tool names are the first signal in selection. Names should be:
- Action-oriented: `search_web`, `create_calendar_event`, `get_user_profile` — not `tool1`, `calendar`, `user`
- Specific enough to distinguish from similar tools: `search_web` vs. `search_knowledge_base` vs. `search_documents` makes the intended scope clear
- Consistent in naming convention across all tools in a set: all snake_case; all verb-first

Tool names must match the pattern `^[a-zA-Z0-9_-]{1,64}$` — only letters, numbers, underscores, and hyphens, up to 64 characters.

**Description quality:**

The tool description must answer three questions:
1. What does this tool do?
2. When should I use it?
3. When should I NOT use it?

The third question is the most often omitted and the most often needed. In a multi-tool environment, the model must distinguish overlapping tools. Without negative guidance, it cannot.

**Parameter descriptions:**

Every parameter must have a description that specifies:
- What the value represents (not just what it is called)
- The expected format (ISO 8601 date, URL, comma-separated list)
- Valid ranges or constraints (maximum 200 characters, positive integer only)
- Edge case behavior ("pass null to retrieve all records; omit to use the caller's default")

**Type choices:**

- Use `enum` over `string` when the valid values are a fixed set. The model will stay within the enum.
- Use `boolean` over `string` for yes/no fields.
- Avoid deeply nested objects when flat parameters suffice. Flat schemas are easier for the model to invoke correctly.
- Mark parameters `required` when the tool cannot function without them. Document optional parameter defaults in the description.

- Every tool parameter MUST have a description. A parameter without a description forces the model to guess its purpose — and its guess may not match the tool's behavior.
- Tool descriptions MUST specify when NOT to use the tool when another tool in the same set could be confused for it.
- Tool names MUST be action-oriented and specific. Generic names (`tool1`, `helper`, `process`) are a schema design failure.
- `enum` types SHOULD be used whenever the valid values for a parameter are a known fixed set.

## Deep Dive

### Schema Anatomy

A complete tool definition for a document search tool:

```json
{
  "name": "search_documents",
  "description": "Search the internal knowledge base for relevant documents. Use when the user asks about company policies, product documentation, or internal processes. Do NOT use for general web search or external information — use search_web for those queries.",
  "input_schema": {
    "type": "object",
    "properties": {
      "query": {
        "type": "string",
        "description": "The search query using key terms from the user's question. Avoid filler words. Maximum 500 characters."
      },
      "document_type": {
        "type": "string",
        "enum": ["policy", "product", "process", "all"],
        "description": "Filter by category. Use 'policy' for HR/compliance documents, 'product' for technical specs, 'process' for workflows. Use 'all' when the category is unclear."
      },
      "max_results": {
        "type": "integer",
        "description": "Maximum number of documents to return. Must be between 1 and 20. Defaults to 5 if not specified."
      }
    },
    "required": ["query"]
  }
}
```

What makes this schema effective:
- The name distinguishes it from `search_web` (internal vs. external)
- The description answers when to use AND when NOT to use
- `query` specifies format and length constraint
- `document_type` uses `enum` with documented semantics for each value
- `max_results` specifies the valid range and default

### Worked Example: Redesigning a Failing Schema

**Original schema (causes invocation failures):**

```json
{
  "name": "calendar",
  "description": "Calendar tool",
  "input_schema": {
    "type": "object",
    "properties": {
      "action": {
        "type": "string"
      },
      "data": {
        "type": "object"
      }
    },
    "required": ["action", "data"]
  }
}
```

Failure modes:
- `name: "calendar"` is a noun, not an action — unclear what invoking this tool does
- `description: "Calendar tool"` provides no selection guidance at all
- `action` has no description and no `enum` — the model must guess valid values
- `data` is an untyped object — the model has no idea what structure to pass
- Both parameters are required but neither is explained

**Redesigned as three specific tools:**

```json
[
  {
    "name": "create_calendar_event",
    "description": "Create a new event on the user's calendar. Use when the user asks to schedule a meeting, add a reminder, or block time. Do NOT use to modify or delete existing events.",
    "input_schema": {
      "type": "object",
      "properties": {
        "title": {
          "type": "string",
          "description": "Event title. Keep concise (under 100 characters)."
        },
        "start_time": {
          "type": "string",
          "description": "Event start time in ISO 8601 format with timezone offset (e.g., '2026-04-15T14:00:00-07:00')."
        },
        "end_time": {
          "type": "string",
          "description": "Event end time in ISO 8601 format. Must be after start_time."
        },
        "attendees": {
          "type": "array",
          "items": { "type": "string" },
          "description": "Email addresses of attendees to invite. Pass an empty array or omit for personal reminders."
        }
      },
      "required": ["title", "start_time", "end_time"]
    }
  },
  {
    "name": "list_calendar_events",
    "description": "Retrieve upcoming events from the user's calendar. Use when the user asks what is scheduled, when they are free, or wants to review their calendar. Do NOT use to create or modify events.",
    "input_schema": {
      "type": "object",
      "properties": {
        "start_date": {
          "type": "string",
          "description": "Start of the date range to query, in YYYY-MM-DD format."
        },
        "end_date": {
          "type": "string",
          "description": "End of the date range, in YYYY-MM-DD format. Defaults to start_date + 7 days if omitted."
        }
      },
      "required": ["start_date"]
    }
  },
  {
    "name": "delete_calendar_event",
    "description": "Permanently delete a calendar event by ID. Use only when the user explicitly asks to delete or cancel a specific event. Do NOT use to modify event details — deletion is irreversible. Obtain event_id from list_calendar_events first.",
    "input_schema": {
      "type": "object",
      "properties": {
        "event_id": {
          "type": "string",
          "description": "The unique identifier of the event. Obtain this from list_calendar_events before calling."
        }
      },
      "required": ["event_id"]
    }
  }
]
```

What changed:
- Three specific tools replace one generic one — each has a clear action and distinct scope
- Every parameter has a description with format and constraints
- `start_time` and `end_time` specify ISO 8601 format explicitly
- `delete_calendar_event` says "deletion is irreversible" — reducing accidental destructive calls
- Each tool has an explicit "Do NOT use" clause

## Best Practices

1. **Write the "when NOT to use" clause before writing the description.** In a multi-tool environment, negative guidance is what the model actually needs to distinguish overlapping tools. Identify which adjacent tools could be confused with this one, then write exclusion clauses for each before filling in the positive guidance.

2. **Test your schema against inputs that should NOT trigger this tool.** After writing a description, send the model several requests that should NOT call this tool and verify it does not. An over-broad description will pull in adjacent queries and cause false invocations.

3. **Prefer multiple specific tools over one generic tool with an action parameter.** The pattern of a single tool with an `action` string parameter defeats the model's selection mechanism. The model selects tools by description — giving it distinct tools with distinct descriptions produces more reliable behavior than one monolithic tool with an undescribed `action` field.
