# [AEE-405] Tool Selection

## Context

Tool selection is the model's decision, on each turn, about which tool (if any) to invoke. This decision is driven entirely by the tool descriptions the engineer provides. The model has no direct access to what tools do — it reads the descriptions and infers when each applies. When selection fails, the failure trace almost always leads back to a description problem, not a model problem.

Understanding tool selection as a prompted decision — not a lookup, not a classification, but a reasoning step based on natural language guidance — changes how engineers approach the problem. The fix for selection failures is not prompting the model harder. It is writing better descriptions.

## Design Think

The core claim: tool selection failures are almost always description failures. When a model invokes the wrong tool or fails to invoke any tool, the cause is nearly always that the tool descriptions were ambiguous, overlapping, or incomplete — not that the model is incapable.

**How models select tools:**

The model reads all tool names and descriptions in the `tools` array before deciding whether to call a tool. It does not execute tools or inspect their implementations. Selection is a reasoning step: given the user's request and the available tools, which tool (if any) should be called, and with what arguments?

This means tool selection quality is bounded by description quality. A well-designed tool will not be selected correctly if its description is ambiguous. A poorly-designed tool may be selected spuriously if its description is over-broad.

**Selection failure modes:**

1. **Wrong tool selected (overlapping descriptions):** Two tools have descriptions that cover similar territory. The model picks one, sometimes incorrectly. Fix: add "when NOT to use" clauses that distinguish the tools from each other.

2. **No tool selected when one should be (vague trigger conditions):** The description does not make clear when to invoke. The model defaults to answering from training data rather than invoking the tool. Fix: add specific trigger conditions ("use this tool when the user asks about current prices, real-time data, or recent events").

3. **Over-invocation (description too broad):** The model calls a tool for requests it was not designed for. Common with descriptions like "use for any question about X" — "any" is broad enough to pull in adjacent queries. Fix: narrow trigger conditions; add "do NOT use for" clauses.

4. **Under-invocation (real-time data not signaled):** The model should invoke a tool but answers from its training data instead. Common for real-time data tools when the description does not signal that the data is time-sensitive. Fix: add "this tool provides current, live data — do not answer questions requiring up-to-date information without calling this tool first."

**Selection accuracy and tool count:**

Selection accuracy degrades as the number of available tools grows. With a small set, the model can reason precisely about each tool's scope. With a large set, relevant tools receive less attention and overlapping descriptions become harder to distinguish. Mitigation strategies:
- **Subset selection:** present only tools relevant to the current task
- **Dynamic tool loading:** use a classifier or embedding search to select which tools to include based on the user's request
- **Tool routing layer:** a first-pass call routes the request to the correct tool subset

**RFC 2119:**

- Tool descriptions MUST specify when NOT to use the tool when another tool in the same set could be confused with it.
- Tool sets with more than 10 tools SHOULD use dynamic tool loading or a routing layer rather than presenting all tools simultaneously.
- Tool selection accuracy MUST be tested against a representative sample of production inputs before deployment. Description quality cannot be verified by reading the schema alone.

## Deep Dive

### Testing Tool Selection

Tool selection is testable. Build a test suite that verifies which tool is selected for a set of representative inputs:

```python
import anthropic

client = anthropic.Anthropic()

def test_tool_selection(tools: list[dict], test_cases: list[dict]) -> dict:
    """
    tools: list of tool definitions (name, description, input_schema)
    test_cases: list of {"input": str, "expected_tool": str | None}
      expected_tool is None when no tool should be called
    Returns: {"total": int, "passed": int, "pass_rate": float, "failures": list}
    """
    results = {"total": len(test_cases), "passed": 0, "failures": []}

    for case in test_cases:
        response = client.messages.create(
            model="claude-opus-4-6",
            max_tokens=256,
            tools=tools,
            messages=[{"role": "user", "content": case["input"]}]
        )

        selected_tool = None
        for block in response.content:
            if block.type == "tool_use":
                selected_tool = block.name
                break

        if selected_tool == case["expected_tool"]:
            results["passed"] += 1
        else:
            results["failures"].append({
                "input": case["input"],
                "expected": case["expected_tool"],
                "got": selected_tool
            })

    results["pass_rate"] = results["passed"] / results["total"] if results["total"] > 0 else 0.0
    return results
```

Example usage with two overlapping tools:

```python
tools = [
    {
        "name": "search_web",
        "description": "Search the web for current information. Use for recent events, current prices, or information that changes over time. Do NOT use for questions about internal company policies or documents.",
        "input_schema": {
            "type": "object",
            "properties": {
                "query": {"type": "string", "description": "The search query."}
            },
            "required": ["query"]
        }
    },
    {
        "name": "search_documents",
        "description": "Search internal company documents and knowledge base. Use for questions about company policies, procedures, products, and internal processes. Do NOT use for general web search or current events.",
        "input_schema": {
            "type": "object",
            "properties": {
                "query": {"type": "string", "description": "The search query."}
            },
            "required": ["query"]
        }
    }
]

test_cases = [
    {"input": "What's the current price of gold?", "expected_tool": "search_web"},
    {"input": "What is our company's refund policy?", "expected_tool": "search_documents"},
    {"input": "What is the capital of France?", "expected_tool": None},
    {"input": "What were our Q3 sales numbers?", "expected_tool": "search_documents"},
    {"input": "Who won the championship last night?", "expected_tool": "search_web"},
]

results = test_tool_selection(tools, test_cases)
print(f"Pass rate: {results['pass_rate']:.0%}")
for failure in results["failures"]:
    print(f"FAIL: '{failure['input']}' → expected {failure['expected']}, got {failure['got']}")
```

Run this against any schema change that affects tool names or descriptions.

### Description Anti-Patterns

**Anti-pattern 1: Mirror descriptions**

```
search_web:       "Search for information on the web"
search_documents: "Search for information in documents"
```

Both descriptions say "search for information" — the model cannot determine which applies to a given query. Add distinguishing context: which web, which documents, what query types belong to each.

**Anti-pattern 2: Missing trigger conditions**

```
get_stock_price: "Gets the current stock price"
```

This does not tell the model when to call it. If the user asks "what's Apple's stock price?" the model may answer from training data (stale). Fix: add "Use this tool whenever a question requires a current or real-time stock price. Do not attempt to answer stock price questions from training data — prices change constantly."

**Anti-pattern 3: Over-broad inclusion**

```
answer_question: "Answers any question the user asks"
```

"Any question" matches every input. The model will call this for everything — including queries that should be routed to more specific tools. Never use "any" in tool trigger conditions.

## Best Practices

1. **Write test cases for selection before writing descriptions.** List the inputs that SHOULD trigger the tool and inputs that should NOT. This forces scope clarity before you write the description, and gives you an immediate test suite once the description is written.

2. **Run the selection test suite after every description change.** A change that fixes one failure may introduce another. Selection failures are often non-local — changing one tool's description can affect the selection behavior of other tools in the set.

3. **Diagnose failures before adding instructions.** When selection fails, the instinct is to add more instructions to the system prompt. This adds noise without fixing the root cause. Diagnose first: is it a missing trigger condition (add "use when"), an overlapping description (add "do NOT use when"), or an over-broad inclusion (narrow scope)?
