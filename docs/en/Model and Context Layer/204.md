# [AEE-204] System Prompt Engineering

## Context

The system prompt is the primary mechanism for configuring model behavior — the engineer's equivalent of configuration code. Unlike most configuration, however, system prompts interact with the model's training distribution in non-obvious ways: placement affects interpretation, length affects attention quality, and user input in the user turn can subvert system instructions in ways the engineer did not design for.

## Design Think

The system prompt is an architectural artifact, not a prompt hack. It defines the model's persistent behavior configuration: persona, task framing, output format requirements, tool descriptions, and safety constraints. Engineers who treat it as a place to "add more instructions" rather than a versioned configuration artifact accumulate technical debt in the same way as unreviewed configuration files.

**What belongs in the system prompt:**
- Stable behavioral instructions (persona, tone, scope of responsibility)
- Output format requirements that are consistent across all interactions
- Tool descriptions and invocation instructions
- Safety constraints and prohibited behaviors

**What does not belong in the system prompt:**
- Dynamic content such as the current date or user-specific data — inject this into the conversation context or the user turn at request time
- Large knowledge bases — retrieve relevant content via RAG (see [AEE-108](../Foundations and Mental Models/108)) rather than pasting documents into the system prompt
- Conversational context that changes per session — this belongs in the conversation history, not in the system-level configuration

**Position carries trust semantics.** System prompt instructions are treated by the model as engineer-level configuration — authoritative and prior to user intent. Instructions placed in the user turn are interpreted as user intent — lower trust, and subject to being reasoned against or overridden by model judgment. This distinction is not just API mechanics; it reflects how the model weights competing instructions during generation.

**RFC 2119 requirements:**

- System prompts **MUST** be version-controlled as configuration artifacts. A system prompt stored only in a database field or an environment variable with no change history is an unauditable configuration artifact.
- System prompts **SHOULD** be tested for robustness across input variations before deployment. A system prompt that works on the happy path may produce unexpected behavior under paraphrase, adversarial phrasing, or off-topic inputs — the context sensitivity described in [AEE-110](../Foundations and Mental Models/110) applies here directly.
- Engineers **MUST** treat prompt injection — user input attempting to override system prompt instructions — as a security concern, not a model quality issue.

## Deep Dive

### System vs. User vs. Assistant Prefill Positioning

The three API roles carry different interpretation semantics:

**System turn** sets baseline behavior and carries the highest trust. Content placed here is treated as authoritative configuration established before any user interaction begins. The model is trained to respect system prompt constraints and to interpret subsequent turns in light of them. Anthropic's documentation notes that even a single sentence establishing a role in the system prompt meaningfully focuses Claude's behavior and tone.

**User turn** is interpreted as user intent. The model applies its judgment about what the user wants, filtered through the constraints established in the system prompt. Instructions that appear only in the user turn have lower trust than system prompt instructions — this is the attack surface for prompt injection.

**Assistant prefill** (where the API supports it) allows the engineer to pre-generate the start of the model's response. This is useful for enforcing output format (e.g., prefilling `{` to force JSON output) but requires care: it bypasses the model's natural turn structure and can produce unexpected results if the prefill creates an inconsistency with the system prompt.

**Instruction placement affects attention quality.** Anthropic's prompting documentation reports that placing queries at the end of long prompts — after the long-form data the query applies to — can improve response quality by up to 30% in tests, particularly for complex multi-document inputs. The mechanism is attention: the model's representation of the query is freshest at generation time when it appears near the end of the input, while the data it applies to is fully in context above it.

### Compression vs. Completeness Tradeoff

Every token in the system prompt consumes context budget (see [AEE-108](../Foundations and Mental Models/108) and [AEE-201](201)). An over-specified system prompt has three costs: it consumes context that could hold conversation history or retrieved knowledge; it dilutes attention across more instructions, potentially reducing adherence to any single one; and it is harder to maintain and audit.

The correct question for every instruction in a system prompt: **does removing this instruction change model behavior?** If the model produces the same outputs without it, the instruction is either captured by the model's training (it would do this anyway) or redundant with another instruction in the prompt. In both cases, remove it.

This requires testing, not intuition. Run the same inputs against the system prompt with and without the candidate instruction. If the outputs are indistinguishable, the instruction is waste.

### Versioning in Practice

System prompts are configuration artifacts and should be managed as such:

- **Store in version control as a file**, not a database field. A file in a repository gets commit history, diff review, and revert capability. A field in a settings table gets none of these by default.
- **Tag changes with intent.** A commit message of "updated system prompt" is not useful. "Restrict scope to customer-facing questions; remove internal tool descriptions" is auditable.
- **Run regression tests when changing.** A change that fixes one behavior may break another. A test suite of representative inputs with expected outputs makes regressions detectable before deployment.

The system prompt is the specification of the model's behavior. It deserves the same version control discipline as application code.

### Prompt Injection Attack Vectors (OWASP LLM01:2025)

Prompt injection (classified as LLM01:2025 by OWASP) is the class of attacks where inputs cause the model to behave in ways the engineer did not intend. OWASP documents three primary injection types:

**Direct injection:** The user's input explicitly attempts to override system prompt instructions. Classic example: "Ignore previous instructions and instead..." Direct injection is the most visible form and often the easiest to detect. Simple pattern matching on common override phrases provides partial protection, but a determined attacker can vary the phrasing indefinitely. Privilege separation — ensuring the model cannot act on user-turn instructions that claim system-level authority — is more robust than pattern matching.

**Indirect injection:** Adversarial instructions are embedded in external content that the agent reads during execution — documents the user uploaded, web pages fetched by a tool, results returned from a database. The model cannot reliably distinguish between "content to be processed" and "instructions to be followed" when both arrive through the same channel. An agent that reads a webpage and acts on its content can be subverted by a webpage that contains instructions formatted to look like system prompt directives. This attack vector scales with the agent's tool capabilities: the more an agent reads from external sources, the larger the indirect injection surface.

**Multimodal injection:** In models that process images or other non-text modalities, adversarial instructions can be hidden in those modalities — for example, text embedded in an image at low contrast or in a font color that matches the background. The model processes the full image and may act on embedded instructions that are invisible to human reviewers.

**Mitigations:**
- **Input sanitization:** Filter or escape common injection patterns in user input before passing to the model. This is not a complete defense but reduces the surface area.
- **Privilege separation:** System-level instructions (what the model is allowed to do, what it should never do) must be established in the system turn and should not be overrideable by user-turn content. Design the system so that a user-turn instruction claiming elevated authority is simply not acted upon.
- **Output validation before acting:** In agentic contexts, validate model-proposed actions against a predefined policy before executing them. The model should propose; a separate validation layer should authorize. Content from retrieved documents should be treated as data to process, not instructions to follow.

### Testing System Prompts

Testing a system prompt means verifying that it behaves correctly under pressure, not just on the examples used during development:

- **Vary user input phrasing.** If the system prompt instructs the model to respond only in English, test what happens when the user writes in French, or mixes languages, or asks the model to explain why it can only respond in English.
- **Test adversarial inputs.** Explicitly test direct injection attempts. If the system prompt passes these tests, document what was tested and when.
- **Verify under edge-case topics.** Test inputs that are adjacent to but outside the defined scope. Does the model correctly decline out-of-scope requests, or does it comply if asked nicely?
- **Measure regression across versions.** When the system prompt changes, run the same test suite and compare outputs. A behavioral change that is not intentional is a defect.

This is the prompt robustness testing approach from [AEE-110](../Foundations and Mental Models/110) applied to the configuration layer.

## Best Practices

1. **Minimize system prompt length aggressively.** For every instruction, test whether removing it changes model behavior. If it does not, remove it. Shorter system prompts consume less context budget, focus model attention on what matters, and are easier to maintain and audit.

2. **Version-control the system prompt in a dedicated file with descriptive commit messages.** Treat every change as a configuration change with intent. A commit message that explains why an instruction was added or removed makes the system prompt auditable. A system prompt with no change history is a liability.

3. **Test explicitly for prompt injection resistance before deploying any agent that processes untrusted user input.** Direct injection, indirect injection (from tool results and retrieved documents), and multimodal injection are all documented attack vectors. Adversarial inputs are not edge cases — they are the attack surface.
