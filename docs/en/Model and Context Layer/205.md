# [AEE-205] Fine-Tuning Basics

## Context

Fine-tuning is frequently reached for too early and for the wrong reasons. Engineers apply it to fix hallucinating models, improve reasoning quality, or add knowledge the model lacks — and are disappointed when it does not help, or makes things worse. Understanding what fine-tuning actually does — and what it cannot do — is prerequisite to making this decision correctly.

## Design Think

Fine-tuning adjusts model weights on a specific dataset to improve performance on a narrow distribution. It is not a general-purpose improvement mechanism. It improves consistency, style, and efficiency for tasks the model can already perform. It does not teach new facts, and it does not reliably fix reasoning failures.

**Three valid reasons to fine-tune:**

1. **Style and format consistency at a level prompting cannot reliably achieve.** When a task requires highly specific output structure, tone, or formatting that few-shot prompting produces inconsistently, fine-tuning can train that consistency in.
2. **Cost and latency reduction through model distillation.** A large model's behavior on a narrow task can be distilled into a smaller fine-tuned model, reducing per-call cost and latency without meaningfully sacrificing quality.
3. **Domain-specific vocabulary or output structure the model handles poorly by default.** Specialized terminology, unusual output schemas, or domain conventions that fall outside the model's training distribution respond well to fine-tuning.

**Three invalid reasons to fine-tune:**

1. **Adding knowledge the model lacks.** Fine-tuning trains behavior patterns from examples — it is not a knowledge injection mechanism. If the model lacks facts or up-to-date information, use RAG (see [AEE-108](../Foundations and Mental Models/108)). Fine-tuning on a dataset containing target facts will not reliably make the model recall those facts at inference time.
2. **Fixing hallucination.** Fine-tuning can reduce hallucination frequency on the training distribution, but it does not eliminate the problem. A model that hallucinates on a general task will continue to hallucinate on edge cases and out-of-distribution inputs after fine-tuning.
3. **Substituting for prompt optimization.** Prompt engineering must be exhausted before fine-tuning is considered. Many problems attributed to model capability are actually prompt design problems. Fine-tuning a model with a poorly optimized prompt is compounding errors.

- Engineers **MUST** establish a baseline benchmark on the target task before fine-tuning. Without a measured baseline, there is no way to determine whether fine-tuning produced an improvement.
- Fine-tuning **MUST NOT** be attempted before prompt engineering is exhausted. Run structured prompt experiments — few-shot examples, chain-of-thought, explicit formatting instructions — and measure results before committing to fine-tuning.

## Deep Dive

### Full Fine-Tuning vs. LoRA and PEFT

**Full fine-tuning** updates every parameter in the model. This is expensive: it requires enough GPU memory to hold all model weights in a training-compatible state, optimizer state for every parameter, and gradients. For large models this is prohibitively costly for most teams. It also carries the highest risk of catastrophic forgetting — because every weight is adjusted, the model's general capabilities are most exposed to displacement by the narrow training distribution.

**LoRA (Low-Rank Adaptation)**, introduced by Hu et al. in "LoRA: Low-Rank Adaptation of Large Language Models" (arXiv:2106.09685), addresses this directly. Instead of updating the original weights, LoRA freezes the pre-trained model weights and injects trainable low-rank decomposition matrices alongside the frozen layers. Only these small matrices are trained. The original weights are never modified.

The efficiency gains are substantial. The paper reports that compared to full fine-tuning of GPT-3 175B with Adam: LoRA reduces the number of trainable parameters by 10,000 times, and reduces GPU memory requirement by 3 times. Despite this, the paper shows LoRA performs on par or better than full fine-tuning on RoBERTa, DeBERTa, GPT-2, and GPT-3 tasks, with no additional inference latency because the trained matrices can be merged into the base weights at serving time.

**PEFT (Parameter-Efficient Fine-Tuning)**, the Hugging Face library, provides implementations of LoRA and many related techniques — including prefix tuning, prompt tuning, and others — under a unified API. PEFT is the practical starting point for most fine-tuning work, providing tooling, documentation, and pre-integration with the transformers ecosystem.

### When to Use LoRA vs. Full Fine-Tuning

Start with LoRA for style adaptation, format consistency, and domain jargon. LoRA iterates faster, requires less compute, and carries lower risk of disrupting the base model's general capabilities. The frozen base weights act as a natural regularizer: the model can only adjust behavior through the low-rank matrices, which limits how far it can drift from its original capabilities.

Only escalate to full fine-tuning if LoRA provably cannot achieve the required performance and compute budget allows. This means running LoRA experiments first, measuring against the baseline, and determining that the performance gap cannot be closed through data quality improvements or hyperparameter tuning. Full fine-tuning as a starting point is unnecessary for most style and format adaptation tasks.

### Data Requirements

Quality over quantity is the dominant principle in fine-tuning data. The relationship between dataset size and fine-tuning quality is not linear: a small set of high-quality, diverse, deduplicated examples will consistently outperform a large noisy dataset. Noise in training data teaches the model to reproduce that noise.

Several properties matter most:

- **Format consistency.** Inconsistent formatting in training examples produces inconsistent formatting in outputs. If half the examples use JSON and half use plain text, the fine-tuned model will not reliably choose either. Enforce a single format throughout the training set.
- **Distribution coverage.** Training data that covers only common cases produces a model that handles common cases well and fails on edge cases. Deliberately include examples of rare inputs, edge cases, and expected failure modes. The fine-tuned model's behavior on inputs outside the training distribution is largely undefined.
- **Deduplication.** Duplicate examples overweight those cases in training. A training set with 50 near-identical examples of one pattern will produce a model biased toward that pattern at the expense of others.

There are no universal minimum example counts — requirements depend on task complexity, model size, and target performance level. Start small, measure, and scale up only when measurement shows more data is helping.

### The Eval-Before-Fine-Tune Requirement

The baseline benchmark is not optional. Before writing a single training example:

1. Define the evaluation metric for the task — exact match, BLEU, human rating, task success rate, or a custom scoring function appropriate for the domain.
2. Construct a held-out evaluation set that represents the full target distribution, including edge cases.
3. Run the evaluation against the base model with the best prompt you can produce. This is the baseline score.
4. Record the baseline score with the model version, prompt version, and evaluation set version.

After fine-tuning, run the identical evaluation against the fine-tuned model. The comparison is valid only if the evaluation is identical. If the improvement over baseline is not significant, fine-tuning did not help — and the result should lead back to diagnosing what the actual problem is (data quality, evaluation design, or the original premise for fine-tuning).

This discipline also guards against a subtle failure mode: fine-tuning that improves scores on the training distribution while degrading performance on the actual target distribution. If the evaluation set was constructed from training-adjacent examples, the improvement may not generalize.

### Catastrophic Forgetting and Over-Specialization

Fine-tuning on a narrow task can reduce the model's performance on tasks it previously handled well — a phenomenon known as catastrophic forgetting. The model's weights, which encode broad general capabilities, are adjusted toward the narrow training distribution. General capabilities that were not reinforced in the fine-tuning data may degrade.

This risk is mitigated but not eliminated by LoRA: because the base weights are frozen, general capabilities are more protected. Full fine-tuning carries the highest catastrophic forgetting risk.

A practical mitigation: evaluate the fine-tuned model on general benchmarks — or on a representative sample of the full task distribution it is expected to handle — before and after fine-tuning. A model that improved on the target task but regressed significantly on general tasks may not be suitable for production deployment if those general tasks are part of its operational envelope.

Related is **over-specialization**: the fine-tuned model performs well on examples drawn from the same distribution as the training data but worse on slightly different inputs in production. Monitor the production input distribution after deployment. If production inputs drift from the training distribution — different phrasing, different domains, different user populations — the fine-tuned model's advantage may erode or reverse. Periodic evaluation against production samples is required to detect this.

## Best Practices

1. **Exhaust prompt engineering before fine-tuning.** Run structured prompt experiments — few-shot examples, chain-of-thought, explicit formatting instructions. Many "fine-tuning needed" problems are prompt design problems. Fine-tuning without a well-optimized baseline prompt will obscure whether fine-tuning is actually the right intervention.

2. **Start with LoRA or QLoRA rather than full fine-tuning.** It is faster to iterate, requires less compute, and carries lower risk of catastrophic forgetting. Only escalate to full fine-tuning if LoRA provably cannot achieve the required performance after genuine iteration on data quality and hyperparameters.

3. **Establish a baseline benchmark before touching training data.** Define the eval, run it against the prompted base model, and record the score. This number is the standard against which fine-tuning is justified. Without it, there is no way to know if fine-tuning helped, hurt, or made no difference.
