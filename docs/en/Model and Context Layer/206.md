# [AEE-206] Model Selection in Production

## Context

Most teams default to the frontier model during development and never revisit the decision. The result is production systems that pay for capability they do not use. Model selection in production is an empirical decision requiring task-specific measurement — not a prestige decision based on leaderboard rankings.

## Design Think

The right model for a production task is the one that passes your task-specific quality bar at the lowest cost and acceptable latency. General benchmark rankings are a starting point for shortlisting candidates, not a selection criterion. Two models with nearly identical MMLU scores may produce dramatically different results on your actual task.

**Capability overshoot** is the most common cost failure mode. Frontier models excel at complex multi-step reasoning, ambiguous instructions, and novel tasks. They are overqualified — and overpriced — for tasks a smaller model handles reliably: classification, extraction, summarization of structured inputs, format conversion. Using a frontier model for these tasks is unnecessary cost and latency.

**Three measurement axes for production model selection** (extending the framework in [AEE-111](../Foundations and Mental Models/111)):

1. **Task-specific quality.** Measured on a representative sample of your actual task. Not benchmark scores — your task. A coding benchmark predicts performance on coding benchmarks; it may not predict performance on your specific code review task.
2. **Cost.** Input token price × input volume + output token price × output volume, adjusted for caching where applicable. The full cost stack, not the headline per-token rate.
3. **Latency.** Time to first token (TTFT) for interactive applications; throughput (tokens per second) for batch workloads.

- Model selection **MUST** be validated on a representative task sample before production deployment. General benchmarks may inform the candidate shortlist; they do not substitute for task-specific measurement.
- Engineers **MUST NOT** select a model based solely on general benchmark rankings. Benchmark performance does not predict task-specific performance.

## Deep Dive

### Reading Benchmarks Critically

Benchmark scores carry three systematic risks that make them unreliable as primary selection criteria.

**Benchmark contamination.** If a model's training data included questions or answers from the benchmark test set, the model's score on that benchmark is inflated relative to its performance on genuinely unseen inputs. This is a well-documented risk in the field. The practical consequence: high benchmark scores do not reliably predict performance on inputs the model has not seen before — which is the category that matters for production.

**Task distribution mismatch.** Benchmarks measure performance on their specific task distribution. A state-of-the-art coding benchmark score measures performance on competitive programming problems, algorithm implementation, and similar tasks. It does not measure performance on your code review workflow, your internal API documentation generation, or your test case synthesis task. Domain and task specificity matter.

**Benchmark saturation.** When multiple frontier models approach ceiling performance on a benchmark (above 90%), the benchmark loses discriminating power. Small differences in scores near the ceiling may reflect measurement noise more than genuine capability differences. Live, task-specific evaluations (covered in the AEE-800 series) are the appropriate tool for production decisions. Use Artificial Analysis (<https://artificialanalysis.ai/>) to track live, multi-dimensional performance data including latency, throughput, and quality across providers — it is more useful than static leaderboard snapshots.

### Cost Modeling in Practice

Per-token pricing requires modeling both input and output separately, since providers typically charge different rates for each. Output tokens cost more than input tokens for the same model.

**Prompt caching (Anthropic):** For workloads where the same large prompt prefix is reused across many requests — system prompts, document context, few-shot examples — prompt caching dramatically reduces input costs. Cache reads are charged at 0.1× the standard input token price (a 90% discount). Cache writes cost 1.25× the standard input token price for a 5-minute TTL, or 2× for a 1-hour TTL. The net cost reduction depends on the cache hit rate: a workload that reads from cache 20 times for each write at the 5-minute TTL recovers the write premium and achieves close to the 90% discount on net input spend. Model this explicitly using your expected prompt structure and request volume.

**Batch APIs (OpenAI and others):** Many providers offer batch processing at roughly 50% of real-time pricing, with a 24-hour latency SLA. For workloads that are not latency-sensitive — bulk document processing, offline evaluation runs, nightly data enrichment — batch pricing can cut per-request cost in half without any quality tradeoff. The OpenAI batch API (<https://platform.openai.com/docs/guides/batch>) is the reference implementation; other providers have equivalent offerings.

On high-volume workloads, a 2× price difference between candidate models compounds into significant absolute cost. A model that costs half as much and passes the quality bar is the correct choice even if it scores lower on a benchmark.

### Latency Modeling

**Time to first token (TTFT)** is the duration from sending the request to receiving the first token of the response. For streaming applications — chat interfaces, interactive tools, real-time analysis — TTFT determines perceived responsiveness. The user sees the first token before generation completes, so TTFT drives user experience more than total generation time.

**Throughput** (tokens per second after the first token) matters for long-form generation: reports, summaries, documents. A model with higher throughput produces the same output in less wall-clock time.

Smaller models generally have lower TTFT and higher throughput at the same hardware budget. If your task is latency-sensitive and a smaller model passes the quality bar, the latency difference often justifies the selection independent of cost.

### Provider Lock-In Risk

Providers differ in API schema, authentication, available models, rate limits, deprecation timelines, and pricing. This creates lock-in risk that is easy to underestimate at the start of a project.

**API format differences** mean that switching providers requires code changes unless abstracted. OpenAI, Anthropic, and Gemini have different request and response schemas despite partial standardization efforts.

**Model deprecation** is routine. Providers typically give 3–6 months notice before retiring a model version, but the migration timeline is not always negotiable if a replacement model produces different output characteristics.

**Pricing changes** happen without advance notice. A model that is economical today may be repriced.

**Mitigation: use an abstraction layer.** LiteLLM (<https://github.com/BerriAI/litellm>) provides a unified OpenAI-compatible interface to 100+ LLM providers. A call to `completion(model="anthropic/claude-sonnet-4-20250514", ...)` uses the same interface as `completion(model="openai/gpt-4o", ...)`. Switching providers or models becomes a configuration change rather than a code change. This also enables A/B testing of models in production without application-layer modifications.

### Routing in Production

Not every request in a production system has the same complexity. A user query that asks for a one-sentence summary is a different task than a query that requires synthesizing conflicting information across multiple documents. Treating all requests identically means paying frontier model prices for simple requests.

**LLM routing** addresses this by directing requests to model tiers based on complexity or type. Research on unified routing and cascading approaches (arXiv:2410.10347) shows that combining routing and cascading strategies achieves significant cost reduction while maintaining quality. The paper demonstrates that cascade routing consistently outperforms using individual models alone.

**Classifier-based routing:** A small, fast model classifies each incoming request by type or complexity and routes it to the appropriate tier. Simple requests go to cheaper models; complex requests escalate to the frontier tier.

**Cost-based routing:** Route to the cheapest model that has historically met the quality bar for a given request class. Track per-class quality metrics in production to maintain the routing table.

**Fallback chains:** If Model A fails (timeout, error, or output below quality threshold), retry on Model B. This provides resilience against provider outages and graceful degradation when a cheaper model cannot handle an edge case.

## Best Practices

1. **Before selecting a model for production, build a representative eval set of 50–200 examples from your actual task.** Run all candidate models against it. The model that passes your quality bar at the lowest cost wins. General benchmark rankings inform the shortlist; task-specific evaluation determines the selection.

2. **Model the full cost stack before committing.** Account for input price, output price, caching discount if the workload structure supports it, and batch discount if latency requirements permit. On high-volume workloads, a 2× price difference compounds into significant absolute cost — the arithmetic is worth doing explicitly before production deployment.

3. **Build a provider abstraction layer from the start.** Switching models or providers should be a configuration change, not a code change. This also enables A/B testing of models in production. LiteLLM is a practical starting point for teams that want a unified interface without building the abstraction themselves.
