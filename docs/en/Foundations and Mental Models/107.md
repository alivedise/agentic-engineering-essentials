# [AEE-107] Goals, Tasks, and Plans

## Context

Agents receive goals from humans and must decompose them into executable steps. The quality of this decomposition determines whether the agent succeeds or fails: a well-specified goal with clear task boundaries and verifiable completion criteria almost always produces better outcomes than a vague goal handed to a capable model. Most agent failures in production trace to one of three root causes: poor goal specification (the goal was ambiguous or contradictory), missing completion criteria (no definition of "done" exists so the agent cannot verify success), or rigid plans that cannot replan when a step fails. These are engineering failures, not model failures.

## Design Think

The core claim: goal → task → plan is a hierarchy with distinct engineering concerns at each level. Conflating these three concepts produces systems that either cannot start (the goal is too vague to decompose) or cannot recover (the plan is too rigid to adapt when a step fails).

**Goal:** What the user wants. Goals are often ambiguous, underspecified, or conflicting. They frequently contain implicit assumptions about context, constraints, and acceptable trade-offs that the user has not articulated. The agent must either clarify the ambiguity before acting or adopt a reasonable, explicit interpretation and surface it to the user. A goal is not an engineering artifact — it is an expression of intent. It must be translated into engineering artifacts before execution begins.

**Task:** A discrete, executable unit of work with a verifiable completion criterion. Goals decompose into tasks. A task without a completion criterion is not a task — it is a direction. "Research the topic" is a direction. "Return a 3-bullet summary of the top 5 search results for query X, where each bullet is under 30 words" is a task: it has a start state (query X), an output specification (3 bullets, 5 results, 30 words), and a verifiable completion criterion (the output either meets the spec or it does not). Completion criteria are runtime evals — they are the mechanism by which the harness verifies that each step was completed correctly before proceeding to the next.

**Plan:** The ordered sequence of tasks the agent will execute to achieve the goal. Plans may be static (generated upfront and executed as written) or dynamic (replanned as execution proceeds based on the outputs of completed tasks). Static plans work for well-understood, stable workflows. Dynamic plans are required when later tasks depend on the results of earlier tasks in ways that cannot be fully specified upfront — which is most non-trivial agentic work. An agent that cannot replan when a task fails is a fragile script with a language model attached.

**RFC 2119:**

- Every agent task MUST have a completion criterion that can be verified programmatically or by the model.
- Goals that cannot be decomposed into tasks with completion criteria SHOULD be escalated to the human for clarification before execution begins.
- Plans MUST be revisable: agents that cannot replan when a task fails will fail on any non-trivial goal.
- Agents SHOULD surface ambiguity in the goal before beginning execution, not after producing incorrect output.

## Deep Dive

### Hierarchical Task Planning vs. Flat Task Lists

A flat task list is a sequence of independent actions. It works when tasks have no dependencies — when the output of task N is not required by task N+1. Most non-trivial agent goals involve dependent tasks: task N+1 needs the result of task N, or task N+3 needs to know whether task N+1 succeeded before deciding what to do. Flat lists fail here because they have no representation of dependency and no mechanism for conditional branching.

Hierarchical task planning decomposes a high-level goal into sub-goals, then sub-goals into tasks, then tasks into actions. The hierarchy makes dependencies explicit: task N+1 is a child of task N, not a sibling of equal priority. This representation supports both dependency tracking and selective replanning — when task N fails, only the subtree rooted at task N needs to be replanned, not the entire plan.

The BabyAGI architecture demonstrates the simplest dynamic task management loop: execute the current task, evaluate the result, create new tasks based on the result, reprioritize the task list, and repeat. Even this minimal loop handles a property that flat lists cannot: the task list itself changes as execution proceeds.

### Dynamic Replanning

Replanning decisions arise when a task fails, when a task's output is unexpected, or when new information changes the validity of future tasks. Three replanning strategies cover most cases:

1. **Retry with modification:** Re-execute the failed task with adjusted parameters (different search query, different API endpoint, reduced scope). Appropriate when the task specification was valid but the execution encountered a transient or correctable error.
2. **Subtree substitution:** Replace the failed task and its dependents with an alternative sub-plan that achieves the same sub-goal via a different path. Appropriate when the original task is fundamentally blocked.
3. **Goal escalation:** Surface to the human that the plan cannot be completed as designed and request clarification or new instructions. Appropriate when no alternative path exists within the agent's capabilities.

The harness must implement replanning as explicit logic, not as an implicit property of the model. A model may generate a new plan when prompted, but the harness must decide when replanning is warranted, what information to include in the replanning prompt, and how to integrate the new plan with the already-completed work.

### Goal Ambiguity in Production

Goal ambiguity manifests in the worst possible failure mode: the model proceeds on an incorrect interpretation and produces confident, internally consistent, wrong output. This is worse than a task failure that throws an error, because the harness may not detect it. The output looks correct at a surface level; the error only becomes apparent when downstream systems consume the output or a human reviews the final result.

The pattern: the user provides a goal with an implicit assumption that the agent does not share. The agent selects the most plausible interpretation given its training distribution, executes the plan, and delivers a result that is technically consistent with its interpretation but misaligned with the user's intent. No error was raised; no signal was generated. The failure is silent.

Mitigations: prompt the model to identify unclear requirements and surface them as questions before execution begins. Treat goal clarification as a first-class step in the agent loop, not as an exception handler. If the user cannot or will not clarify, document the interpretation adopted and surface it in the agent's output so the human can evaluate whether it was correct.

### Completion Criteria as Runtime Evals

Completion criteria are not administrative documentation — they are executable specifications. Each criterion should be implementable as a check that the harness runs after task execution: does the output match the type and structure specification? Does it satisfy domain-specific constraints? Does it satisfy length, format, or content requirements?

This is the connection to AEE-102 (RL + Verifiable Rewards): completion criteria are the runtime equivalent of verifiable reward signals. A task whose completion criterion can be verified automatically is a task that can be measured, and a task that can be measured can be improved. A task with no completion criterion is a black box: the harness cannot know whether it succeeded or failed, and cannot provide the feedback signal needed to improve performance over time.

## Best Practices

1. **Write completion criteria when writing agent tasks, not after.** If you cannot state how to verify that a task completed correctly, the task is underspecified. The act of writing the completion criterion often surfaces ambiguity in the task definition itself — which is better discovered before execution than after.
2. **Build replanning into the harness from the start.** An agent that cannot adapt its plan when a step fails is a fragile script, not an agent. Replanning is not an edge case — it is the normal path for non-trivial goals. Design the harness to expect and handle plan revision, not to treat it as an error condition.
3. **Prompt the model to identify unclear requirements before execution.** Surface ambiguity as a structured clarification request, not as a bad output. A question asked before execution costs one round-trip; bad output produced after a full execution run may cost the entire run and the downstream consequences of the incorrect result.
