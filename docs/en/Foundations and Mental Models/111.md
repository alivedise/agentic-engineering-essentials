# [AEE-111] Model Selection for Agentic Tasks

## Context

Choosing the right model for an agentic task is not obvious, and the consequences of choosing wrong compound quickly. Frontier models are expensive and slow; smaller models may fail on tasks requiring multi-step reasoning or reliable tool use. Engineers who default to the most powerful model available for every task pay 10–50x more per task than necessary on the sub-tasks that do not require frontier capability, accumulate latency on the critical path that users experience, and build systems that are difficult to optimize because the model tier is not a design variable. Model selection is an architectural decision with cost, latency, quality, and reliability consequences — and it can be optimized without sacrificing quality.

## Design Think

The core claim: model selection for agentic tasks requires reasoning across three axes — capability, cost, and latency — and matching them to task requirements. Defaulting to the most powerful model available is not an architecture; it is the absence of one.

**1. Capability axis.**

Map the task to its capability tier requirement before selecting a model (see AEE-104 for the full capability tier taxonomy). As a working heuristic:

- **Tier 0–1 (classification, extraction, formatting, simple summarization):** These tasks are well-served by smaller, cheaper models. They involve pattern-matching over well-defined input schemas and do not require multi-step reasoning or planning. The quality delta between a Tier 1 and a frontier model on these tasks is small; the cost delta is large.
- **Tier 2 (multi-step reasoning, tool use, code generation):** These tasks require reliable instruction following across multiple steps and benefit from stronger models. Errors here propagate — a failed tool call or incorrect intermediate result can cascade through subsequent steps.
- **Tier 3+ (goal decomposition, long-horizon planning, ambiguous open-ended tasks):** Frontier models are appropriate here. The cost is justified by the complexity of the task and the compounding risk of error in long execution sequences.

**2. Cost and latency axis.**

Published benchmark scores are not a substitute for task-specific measurement. The same model may perform very differently on your task profile than on standard academic benchmarks. Before committing to a model, measure on representative samples of your actual workload.

Two measurement priorities differ by deployment context:

- **Synchronous, user-facing tasks:** latency dominates. A task that takes 8 seconds on a frontier model and 2 seconds on a smaller model is a user experience choice, not just a cost choice. Measure p50, p95, and p99 latencies under realistic concurrency.
- **Batch and async tasks:** cost per task dominates. Measure token consumption on representative inputs and multiply by the per-token price for each candidate model. A 10x cost difference at batch scale is a monthly infrastructure budget line, not a rounding error.

**3. Model routing.**

Complex agentic systems should route different sub-tasks to different models rather than running a single model for all operations. This is an architectural decision made at design time, not a runtime optimization.

A practical routing taxonomy:

- **Coordinator / planner:** frontier model — responsible for goal decomposition, long-horizon reasoning, and deciding which tools to invoke. Quality here has high leverage because errors propagate to every sub-task.
- **Execution sub-tasks:** cheaper models — classification, structured extraction, reformatting, simple summarization. These are high-volume, low-risk operations where a cheaper model performs comparably.
- **Verification / critic:** mid-tier or frontier model — checking intermediate outputs for quality or consistency. This is a lower-volume operation where quality matters more than cost.

Research on routing and cascading (arXiv 2410.10347) demonstrates that optimal cascade strategies — attempt with a cheaper model first, escalate to a more capable model if a quality threshold is not met — achieve 5x or greater cost savings at negligible overall performance loss compared to running a frontier model on every request. The key insight is that a large fraction of real-world queries are simple enough that the cheaper model succeeds; only the harder queries escalate.

**RFC 2119:**

- Engineers MUST measure cost and latency on their specific task and token profile before committing to a model for production. Published benchmarks on standard datasets are not a substitute for measurement on representative production inputs.
- Engineers SHOULD use model routing in systems with diverse task types rather than a single model for all tasks. Single-model architectures systematically overpay on simple tasks and may underperform on complex ones.
- Systems MUST NOT be benchmarked on synthetic or demo inputs and deployed to production without measurement on representative production inputs. Demo inputs are typically shorter, cleaner, and more structured than production inputs; performance on demos does not predict production performance.

## Deep Dive

### Model Routing Architectures

The simplest routing architecture is static routing: classify each incoming task by type at design time and assign it to a model tier. This requires a task taxonomy but is fully predictable in cost and latency.

Dynamic routing adds a lightweight classifier that labels each request at runtime and routes it to the appropriate model. The classifier itself should be a small, fast model (or a rule-based system) to avoid adding latency. The LLM routing production guide (TianPan.co, 2025) documents that shifting 50% of queries from a frontier model to a smaller model reduces costs approximately 14x on that segment — with the critical caveat that the task mix must be profiled first to identify which 50% can safely downgrade.

### The LLM Cascade Pattern

The cascade pattern adds a quality gate to static routing. The request is first attempted with a cheaper model. If the output meets a defined quality threshold (e.g., a confidence score, a schema validation check, or a secondary model's assessment), it is returned. If not, the request escalates to a more capable model.

The academic analysis of cascade strategies (arXiv 2410.10347) derives the optimal cascade policy for minimizing cost subject to a quality constraint and shows that the optimal policy is surprisingly aggressive in its use of cheaper models — because the distribution of real-world query difficulty is heavily skewed toward simple queries. The practical implication: the quality threshold, not the model tier, is the primary design lever. A threshold that is too high will escalate unnecessarily; a threshold that is too low will accept poor outputs. Calibrate the threshold empirically on held-out data.

### Building a Cost/Latency/Quality Measurement Matrix

Before production deployment, build a measurement matrix with the following dimensions:

- **Task types:** list the distinct task categories your system performs (e.g., intent classification, document summarization, code generation, tool-call planning).
- **Candidate models:** the 2–4 model tiers you are considering.
- **Measurements per cell:** quality score on representative inputs (using your task-specific evaluation criteria), median latency, p95 latency, and cost per 1,000 tasks at your expected token profile.

The matrix makes the tradeoff explicit and creates a documented baseline. It also makes model upgrades and downgrades auditable: when a new model release claims improved performance, you can re-run the matrix and make the comparison against the same evaluation criteria.

### Harness Design and Model Abstraction

Different models may require different prompting conventions, tool call formats, function call schemas, and error handling patterns. A harness that is tightly coupled to a specific model's API format will require significant rework each time the model selection changes.

The engineering principle is model-agnostic interface design: the harness should define a model adapter interface that abstracts over provider-specific API formats. Swapping models — or routing different tasks to different models — should require changing a configuration value, not rewriting tool routing logic, prompt templates, or response parsing code. This is especially important in routing architectures where multiple models may be in use simultaneously.

## Best Practices

1. **Build a cost/latency/quality measurement matrix for your task types before production deployment.** Measure on representative production inputs, not synthetic or demo inputs. Synthetic inputs are typically shorter, cleaner, and more uniform than production inputs, and performance on them does not predict production performance. The matrix is also a living artifact: update it when new model versions are released or when your task distribution changes.

2. **Design model-agnostic harness interfaces.** The harness should abstract over model-specific API formats so that swapping models — or routing different tasks to different models — requires changing a configuration parameter, not rewriting tool routing, prompt templates, or response parsing. Coupling the harness to a specific model's conventions creates architectural debt that compounds every time model selection changes.

3. **Start with a frontier model to establish a quality baseline, then systematically downgrade task by task to find the minimum viable tier.** The minimum viable tier is the cheapest model that meets your quality threshold on representative tasks from that category. "We can use a cheaper model here" should be a conclusion drawn from measurement, not an assumption. Work task-type by task-type so that each downgrade is isolated and its quality impact is measured before the next one.
