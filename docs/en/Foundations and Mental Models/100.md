# [AEE-100] What Is an Agent?

## Context

The word "agent" is used loosely in AI discourse -- sometimes meaning a simple chatbot with a system prompt, sometimes meaning a fully autonomous system that runs for hours without human input. Without a shared definition, engineers cannot reason clearly about what they are building or evaluate whether their system is fit for purpose.

## Design Think

An **agent** is a system that perceives its environment, makes decisions, and takes actions to achieve a goal, potentially over multiple steps, using one or more tools or capabilities beyond text generation.

The key properties that distinguish an agent from a plain LLM call:

1. **Multi-step reasoning** -- the system can plan and execute a sequence of actions, not just respond once
2. **Tool use** -- the system can act on the world (read files, call APIs, execute code, browse the web)
3. **State** -- the system maintains context across steps (conversation history, memory, intermediate results)
4. **Goal-directedness** -- the system acts toward an objective, not just answers a question

A system gains "agentness" in proportion to how many of these properties it exercises autonomously. A chatbot has state. A RAG system has tool use. A fully autonomous coding agent has all four.

### Agent Taxonomy

Classical AI agent literature distinguishes three broad categories:

- **Reactive agents** -- no internal world model; respond to stimuli directly (stimulus-response). Fast, simple, but cannot reason about the future.
- **Deliberative agents** -- maintain an explicit model of the world; plan before acting. Can reason across time horizons but are slower and require an accurate world model.
- **Hybrid agents** -- most production LLM-based agents. The model serves as a learned, implicit world model; planning emerges from in-context reasoning rather than explicit symbolic logic.

### Boundary Definition

> A system MUST exhibit at least tool use or persistent multi-step reasoning to be called an agent in the AEE sense. A single LLM call with a system prompt is NOT an agent.

This RFC 2119 boundary exists to keep engineering conversations precise. "We are building an agent" should carry specific engineering implications -- harness design, eval strategy, state management, permission model -- that do not apply to a single-call LLM integration.

## Deep Dive

### Reactive vs. Deliberative Agents

The classical AI agent literature distinguishes reactive agents -- those that respond to stimuli without an internal world model -- from deliberative agents -- those that maintain a model of their environment and plan before acting. LLM-based agents blur this distinction: the model functions as a learned world model, and planning emerges from prompting rather than explicit symbolic reasoning. In practice, most production LLM agents are hybrid: they react to tool observations at each step while using the model's in-context reasoning as a form of lightweight deliberation.

### The PEAS Framework

The academic PEAS framework (Russell & Norvig) provides a formal lens for characterizing any agent system:

- **Performance** -- the metric that defines success (tests passed, task completed, user rating)
- **Environment** -- the world the agent operates in (tools available, external APIs, file system)
- **Actuators** -- how the agent acts on the world (tool calls, text output, code execution)
- **Sensors** -- how the agent perceives the world (input tokens, tool results, observations)

Applying PEAS to a new agent design surfaces gaps: an agent with no clear Performance measure has no defined success criterion; an agent with no defined Environment has unbounded scope.

### The "Agentness" Dial

Rather than a binary agent/non-agent classification, it is more useful to reason about degree of agentness -- how many of the four core properties the system exercises, and how autonomously:

| System | Tool Use | Multi-Step | State | Goal-Directed | Agentness |
|--------|----------|-----------|-------|--------------|-----------|
| Single LLM call | No | No | No | No | Minimal |
| RAG pipeline | Limited | No | No | No | Low |
| Function-calling assistant | Yes | Limited | Session | Partial | Medium |
| ReAct loop with tools | Yes | Yes | Task | Yes | High |
| Long-horizon coding agent | Yes | Yes | Persistent | Yes | Full |

This dial metaphor matters because many production systems sit in the middle range. Engineering decisions -- harness design, eval strategy, permission model -- should match the agentness level actually required.

## Best Practices

1. **Establish your agentness level before designing the harness.** A system that only needs one tool call does not need a full ReAct loop with replanning. Harness complexity should match agentness requirements.
2. **Define PEAS for every new agent design.** Unanswered PEAS components are design gaps. If you cannot define Performance, you cannot evaluate the agent; if you cannot define Environment, scope will expand unexpectedly.
3. **State management is the hardest part.** Multi-step reasoning and tool use are well-supported by modern model APIs. State -- knowing what happened in prior steps, what the current goal is, what has been tried -- is where most agent failures occur in production.
