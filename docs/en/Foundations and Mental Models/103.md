# [AEE-103] Agent vs. Chatbot

## Context

Engineers who treat agents like chatbots under-engineer their systems: they omit the harness, skip permission modeling, and build no eval infrastructure, then are surprised when the system produces unauditable side effects. Engineers who treat chatbots like agents over-engineer them: they add tool routing, state machines, and lifecycle management to what is fundamentally a request-response loop, adding cost and fragility without benefit. The architectural difference is well-documented: chatbots are "stateless prediction" systems (request-response loop) while agents operate as "stateful reasoning" systems (perception-action loop), and the transition point is defined by tool use plus persistent state across steps. Getting this classification right at design time is one of the highest-leverage decisions a team makes.

## Design Think

The core claim: the distinction between agent and chatbot is architectural, defined by specific measurable properties — not by UI, marketing label, or brand.

A **chatbot** is a system optimized for text exchange with a human. It produces a response per turn. It may maintain session memory (conversation history) but it does not take actions in external systems and is not designed to execute multi-step plans. Its failure modes are bounded: a bad response is a bad response.

An **agent** is a system that exercises one or more of: tool use, multi-step reasoning, persistent state management, or goal-directed execution. The key marker is the ability to take actions that have effects outside the system — writing to a database, calling an API, executing code, sending a message. The framing from arXiv 2503.12687 is useful: "chatbots offer assistance, agents offer labor." A chatbot helps you think; an agent acts on your behalf.

The engineering consequences follow directly:

- A chatbot MAY have memory within a session but MUST NOT be designed with the expectation that it will act autonomously on external systems.
- An agent MUST be designed with infrastructure appropriate to its agentness: a harness that wraps the execution loop, tool routing that constrains what the agent can call, a permission model that limits blast radius, and an eval harness that verifies outcomes.
- A system that looks like a chatbot in its UI but makes autonomous tool calls IS an agent for engineering purposes, and MUST be designed and evaluated accordingly — the UI presentation is irrelevant to architectural classification.
- Engineers MUST NOT apply agent infrastructure to systems that are genuinely chatbots: the overhead is unwarranted and the added complexity creates new failure modes without solving any real problem.

The MarkTechPost consolidation of vendor definitions sharpens this: a chatbot focuses on "interpreting a question and returning an answer," while an agent carries "a request from intent to outcome across connected systems." The transition from the first to the second is not incremental — it is a phase change that requires redesign, not an upgrade.

## Deep Dive

### The Blurring in Modern Products

Many products market themselves as "AI assistants" or "AI copilots" while being architecturally agents. A product that books a calendar appointment, files a support ticket, or executes a code change is an agent, regardless of whether its interface is a chat window. The UI presentation creates the perception of a chatbot while the engineering reality is an agent. This mismatch is a root cause of underengineered deployments.

The "Stop Building Chatbots" practitioner analysis characterizes the distinction operationally: a chatbot is "triggered by human input and uses NLP/retrieval to produce a response"; an agent is "triggered by goal or event and uses reasoning, planning, and tool use to produce an outcome." The trigger and the output type are both different.

### Why UI Does Not Determine Architectural Category

Architectural category is determined by what the system does, not how it presents itself. A command-line tool that calls one API is not an agent. A chat interface that orchestrates five tool calls, maintains a goal across turns, and writes results to a database is an agent. The interface is a rendering choice; the architecture is a consequence of the system's responsibilities.

This matters because tooling, observability, and evaluation requirements all follow from architectural category. An agent needs distributed tracing on its tool calls, audit logs of its external actions, and regression tests against representative task sequences. A chatbot needs response quality eval and latency monitoring. Applying the wrong tooling set to either system produces gaps.

### The Transition Case: Chatbot Upgraded with Tool Use

Consider a customer service chatbot. In its original form: it receives a message, looks up knowledge base articles, and returns a suggested response. Architectural category: chatbot. Failure mode: bad suggestions. Blast radius: zero — it cannot act.

The team then adds one tool: the ability to issue refunds up to $50 automatically. The system now makes external state changes. Architectural category: agent. Failure mode: incorrect or fraudulent refunds. Blast radius: real and financial.

This transition requires a deliberate redesign — not an incremental change. The new system needs: a permission model (which accounts, which thresholds), audit logging (who asked, what was issued, when), an eval harness (does it refuse out-of-policy requests?), and rate limiting (how many refunds per session). None of these were needed for the chatbot. All of them are necessary for the agent.

## Best Practices

1. **Classify by architectural properties, not UI.** Ask: does this system make tool calls? Does it maintain state across steps toward a goal? If yes to either, it is an agent. Classify it as an agent regardless of how it presents to the user.
2. **Do not apply agent infrastructure to genuine chatbots.** Harnesses, tool routing, and permission models add complexity. A system that only produces text responses does not need them and will be harder to maintain if they are added speculatively.
3. **Treat the chatbot-to-agent transition as an architectural milestone, not an incremental change.** When a chatbot gains its first tool that acts on an external system, stop, redesign the system with full agent infrastructure (harness, permission model, audit log, eval harness), and re-review its threat model before deploying.
