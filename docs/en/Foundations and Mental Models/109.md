# [AEE-109] How LLMs Work — A Practitioner's Mental Model

## Context

Engineers who lack a mental model of how LLMs work make bad architectural decisions. They over-trust the model on tasks it fundamentally cannot perform reliably, under-trust it on tasks it handles well, and misattribute failures to bugs in their application code when the root cause is a structural property of the model. A correct mental model does not require deep ML knowledge — but it does require understanding five foundational properties of how LLMs generate output. This article provides that minimum viable model, calibrated for reasoning about agent behavior, failure modes, and design tradeoffs.

## Design Think

The core claim: LLMs are autoregressive token predictors with a fixed context window. Understanding this single sentence explains most of the agent failure modes practitioners encounter in production.

**1. Tokens, not words.**

LLMs operate on tokens — subword units produced by a tokenization algorithm applied before the model ever sees any text. One token corresponds to approximately 0.75 English words on average, but the ratio is not uniform. Non-English languages, source code, mathematical notation, and special characters often tokenize at significantly different densities. CJK characters (Chinese, Japanese, Korean) frequently produce more tokens per character than English text does. Cost, context window limits, and per-request latency are all measured in tokens, not characters or words. An engineer who estimates context usage by character count will produce systematically wrong estimates.

**2. Autoregressive generation.**

Each output token is predicted as a probability distribution over the model's vocabulary, conditioned on all tokens that precede it in the context (the prompt plus any tokens already generated). The model generates left to right, token by token. There is no global planning pass before generation begins, no revision mechanism, and no re-reading of the output already produced. The model commits to each token in sequence and cannot revise it. This is why a model can begin a sentence with apparent confidence and then produce a contradiction four tokens later — the later tokens are predicted from what came before, not from an independently verified claim.

**3. The context window is the model's only memory.**

There is no persistent state between API calls. Everything the model knows at inference time must be present in the input tokens. Previous conversations, user preferences, intermediate reasoning from a prior call, and any world knowledge that is not in the context are all invisible to the model unless explicitly included. A model given a 128K-token context window has exactly 128K tokens of memory per inference call — and nothing else.

**4. Attention, not retrieval.**

The model does not look up facts in a database or traverse a knowledge graph. During inference, attention layers compute weighted relationships between tokens in the current context, and feedforward layers apply transformations learned from training data. When the model produces a factual claim, it is producing the statistically most probable continuation given the training corpus — not retrieving a verified fact. This is why the model can produce fluent, detailed, and completely wrong factual claims: the mechanism optimizes for token probability, not truth.

**5. The model cannot reason about what it does not know it does not know.**

The model has no built-in uncertainty signal. It does not have a metadata layer that tracks which outputs are high-confidence versus speculative. It produces fluent, similarly-formatted output regardless of whether it has strong training-data grounding for the claim. Confident-sounding output is a property of the generation process, not a reliability signal. A model that is wrong will sound identical to a model that is right.

**RFC 2119:**

- Engineers designing agentic systems MUST internalize this mental model. The five properties above — tokenization, autoregression, context-bounded memory, attention-based prediction, and absent uncertainty signals — directly explain the hallucination, context overflow, inconsistent behavior, and unexpected latency issues that dominate production agentic system failures.
- Engineers MUST count tokens, not characters, when estimating cost and context window usage. Character-based estimates are systematically inaccurate.
- Engineers SHOULD NOT interpret fluent, confident model output as evidence of correctness. Output style is not correlated with output accuracy.

## Deep Dive

### Transformer Architecture in One Paragraph

Input text is first tokenized into a sequence of integer IDs. Each ID is mapped to a high-dimensional embedding vector. The stack of transformer layers then processes the full sequence: in each layer, multi-head self-attention computes, for every token position, a weighted sum over the values of all other token positions — weighting determined by learned query-key dot products. This lets every token "attend to" every other token in context. Feedforward sublayers apply position-wise learned transformations after attention. After all layers, the final token position's output is projected through a linear layer and softmax to produce a probability distribution over the full vocabulary. The model samples from this distribution to produce the next token, appends it to the context, and repeats. This sequential procedure is the autoregressive loop. The foundational paper proposing the full transformer architecture is Vaswani et al., "Attention Is All You Need," arXiv 1706.03762 (NeurIPS 2017), which has accumulated over 173,000 citations as of 2025.

### Tokenization for Practitioners

The dominant tokenization algorithm for modern LLMs is Byte Pair Encoding (BPE). BPE works by starting with a vocabulary of individual bytes or characters and iteratively merging the most frequently co-occurring adjacent pair into a single new token. This process repeats until a target vocabulary size (typically 50,000–100,000 tokens) is reached. The result is a vocabulary of subword units — common English words become single tokens, rarer words are split into two or three subword tokens, and very rare strings are split down to individual bytes.

The practical engineering implications:

- The same word may tokenize differently depending on context (capitalization, position after a newline, whether it is preceded by a space).
- English text tokenizes efficiently at roughly 1 token per word.
- Code tokenizes at variable density depending on language and style: long variable names, indent-heavy code, and languages with complex syntax can produce significantly more tokens than equivalent logic in compact code.
- CJK text typically produces 1–2 tokens per character, compared to English's ~1 token per ~4 characters for common words.
- Numbers and arithmetic expressions tokenize inconsistently: the number "1234" might tokenize as one token, two tokens, or four tokens depending on the tokenizer and context.

The Hugging Face LLM Course provides an authoritative step-by-step explanation of BPE, and Karpathy's `minbpe` (GitHub) implements the full algorithm in approximately 200 lines of Python — the most useful resource for engineers who learn from code.

### Why Hallucination Happens

Hallucination is not a bug in the conventional sense. It is the model behaving correctly according to its training objective.

During pre-training, the model is optimized to predict the next token given the preceding context, over a very large corpus of text. That corpus contains factual errors, confident-sounding speculation, and statements that were true when written but have since become false. The model learns to reproduce all of it with similar fluency. It has no mechanism to distinguish "this is a fact I am confident in" from "this is something that sounds like a fact I am generating."

The theoretical grounding is provided by Xu et al., "Hallucination is Inevitable: An Innate Limitation of Large Language Models" (arXiv 2401.11817): because LLMs cannot learn all computable functions from finite training data, they will necessarily produce incorrect outputs when used as general problem-solvers. This is a structural property, not a fixable training artifact.

The comprehensive 2025 survey by arXiv 2510.06265 identifies attention mechanism weaknesses and capability misalignment as architectural contributors. Separately, arXiv 2509.18970 documents how hallucination compounds in multi-step agentic contexts: an error introduced in step 2 of a 10-step reasoning chain is carried forward and built upon by every subsequent step, amplifying the original mistake into a large final error. This compounding effect is why agentic systems are more susceptible to hallucination damage than single-turn chatbots, even at identical per-step error rates.

## Best Practices

1. **Count tokens, not characters.** Before building any production system involving a context window, integrate a tokenizer library (such as the one published by the model provider) to instrument actual token counts. Character-based estimates undercount CJK and code inputs substantially and produce incorrect cost and limit projections.

2. **Do not infer model accuracy from model confidence or output fluency.** Fluency and confidence in phrasing are outputs of the generation process, not reliability signals. Test correctness programmatically wherever the output will be acted upon: run assertions, execute generated code, verify structured output against a schema. See AEE-102 for the role of verifiable rewards in training systems that self-check reasoning.

3. **Minimize prompts without losing precision.** Verbose prompts are not safer or more reliable than concise prompts. Unnecessary length consumes context capacity, increases cost, and can dilute the model's attention on the instructions that matter. The engineering principle is: every token in the prompt should be there because removing it would degrade output quality.
