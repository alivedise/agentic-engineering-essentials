# [AEE-503] Skill Design

## Context

Writing behavioral guidance that the agent follows is not enough. The guidance must be packaged into a skill that can be invoked reliably, maintained over time, and composed with other skills. The packaging decisions — what the skill is named, what its invocation condition says, where its scope boundaries are drawn — determine whether the skill is a reusable asset or a one-time prompt.

Most skill libraries accumulate artifacts rather than assets. The difference is design discipline: an asset was designed with a specific purpose, a clear trigger, and defined limits. An artifact was written for a specific moment and never given those things.

## Design Think

The core claim: a skill that does too much gets ignored or misapplied. The single most common skill design failure is scope creep: a skill that started as "how to debug" becomes "how to do everything when something is wrong." Skill design is the discipline of drawing scope boundaries that hold under real usage.

**The scope boundary problem:**

Skills grow because it is easier to add guidance than to create a new skill. A debugging skill gets extended with test-writing guidance (because tests prevent bugs), then refactoring guidance (because clean code is easier to debug), then code review guidance (because review catches bugs). Each addition feels locally justified; the aggregate is a skill that applies to everything.

When a god-skill is loaded, it competes with every other skill for the agent's attention. The result is inconsistent behavior: the agent follows some instructions from the god-skill, ignores others, and the practitioner cannot predict which.

**Single responsibility:**

Each skill should have exactly one primary purpose. The test: can you state the skill's purpose in one sentence without "and"? If not, the skill has more than one purpose.

**Naming conventions:**

- **Verb-noun format**: `diagnose-root-cause`, `write-commit-message`, `review-pull-request`. The verb encodes what the agent does; the noun specifies the domain.
- **Domain prefix for org skills**: `backend:api-design`, `frontend:component-review`. Prevents naming collisions across teams.
- **Specificity over generality**: `generate-jest-test-cases` is better than `testing`. A specific name is an implicit scope boundary.

**The invocation condition as a design constraint:**

Writing the invocation condition forces clarity about scope. If you cannot write it in one sentence, the skill is not well-scoped.

- Too broad: "when the user has a problem with their code"
- Too narrow: "when the user pastes a Python stack trace from a Django app on Linux"
- Well-scoped: "when the user reports unexpected behavior and asks for help identifying the cause"

**Writing the "when NOT to use" section:**

This is the most commonly skipped part of skill design. Negative invocation conditions prevent a skill from being applied to requests it was not designed for, especially requests that look similar but require a different skill.

A debugging skill should specify: "Do NOT use this skill when the user is asking for a code review of working code, when the user wants tests written, or when the user wants a refactor."

- A skill MUST have exactly one primary purpose, stateable in one sentence without "and."
- Skills MUST specify at least one condition under which they should NOT be invoked, particularly when another skill could be confused for it.
- Skill names SHOULD be action-oriented and unambiguous.

## Deep Dive

### Worked Example: From God-Skill to Three Skills

**Before:** A single `engineering-assistant` skill that has grown over time.

```markdown
---
name: engineering-assistant
description: Help with engineering tasks
---

When the user has an engineering problem, help them solve it.

If there's a bug: ask for the error message and stack trace. Identify the root cause. Suggest a fix.

If there's a test to write: analyze the code under test. Write tests for the happy path, edge cases, and error conditions.

If there's code to review: check for correctness, readability, performance, and security. Provide feedback with specific line references.

If there's a refactor: identify code smells. Propose a cleaner structure. Preserve behavior.

When in doubt, ask the user what they need.
```

Problems:
- **Invocation condition**: missing. "Help with engineering tasks" is too broad to auto-select.
- **Scope**: undefined. Covers debugging, testing, code review, and refactoring — four distinct activities.
- **"When NOT to use"**: absent.
- **Conflict risk**: any specialized skill for these activities will conflict with this one.

**After:** Three well-bounded skills.

```markdown
---
name: diagnose-root-cause
description: Identify the root cause of unexpected software behavior. Use when the user reports a bug or unexpected output and asks why it is happening.
invocation: when the user reports unexpected behavior and asks for help identifying the cause
scope: root cause identification only — does not write fixes, tests, or refactors
---

When diagnosing a bug:
1. Ask for the error message, stack trace, and minimal reproduction steps if not provided.
2. Identify the failing assumption: what did the code expect vs. what happened?
3. Trace the execution path to the point of divergence.
4. State the root cause as a specific claim: "The bug is caused by X when Y."
5. Stop here. Do not propose a fix unless explicitly asked.

Do NOT write test cases. Do NOT refactor the code. Do NOT conduct a code review.
Use `write-test-cases` or `review-pull-request` for those tasks.
```

```markdown
---
name: write-test-cases
description: Write test cases for a specific function or module. Use when the user wants test coverage written, not when they are debugging existing failures.
invocation: when the user asks for tests to be written for specific code
scope: test case authoring only — does not debug, refactor, or review
---

When writing tests:
1. Identify the function or module under test and its public interface.
2. List the scenarios: happy path, boundary values, error conditions, edge cases.
3. Write tests in the project's existing test framework (detect from imports or ask).
4. Each test should test one behavior. Name tests after the behavior: `test_returns_empty_list_when_no_items`.

Do NOT diagnose bugs in existing tests. Do NOT refactor the code under test.
Use `diagnose-root-cause` if the tests are failing unexpectedly.
```

```markdown
---
name: review-pull-request
description: Review code changes for correctness, readability, and maintainability. Use when the user shares code for review, not when they are debugging or writing tests.
invocation: when the user asks for a code review or shares a diff/PR for feedback
scope: code review feedback only — does not debug, write tests, or implement changes
---

When reviewing code:
1. Read the full diff before commenting.
2. Organize feedback by severity: blocking issues, important suggestions, minor notes.
3. For each issue: state what, why it matters, and how to fix it.
4. End with an overall assessment: approve, approve with suggestions, or request changes.

Do NOT write replacement code. Do NOT debug issues found during review — flag them.
Do NOT write tests — recommend them.
```

The three skills together cover the same territory as the god-skill. Each is independently invocable, testable, and composable. A practitioner can use `diagnose-root-cause` without loading test or review guidance.

### Testing a Skill Before Publishing

Before promoting a skill from personal to team scope:

1. **Positive test**: present 5 requests that should trigger this skill. Does the agent apply the skill's guidance correctly for each?
2. **Negative test**: present 5 requests that look similar but should NOT trigger this skill. Does the agent avoid applying it?
3. **Boundary test**: present 3 requests at the edge of the scope definition. Is behavior consistent?

If any positive test fails, the skill body needs clarification. If any negative test fails, the "when NOT to use" section is insufficient. This is the same methodology as AEE-306 (prompt robustness testing), applied to skills.

## Best Practices

1. **Start every skill design by writing the invocation condition and scope, not the body.** If you write the body first, the scope will fit whatever you wrote rather than whatever the skill needs to cover.

2. **Test the negative case explicitly.** Ship no skill to team or org scope that has not been tested against requests it should NOT handle. A skill that fires on everything is worse than no skill.

3. **Decompose at the first sign of "and."** When you catch yourself writing "if X do this, if Y do that" in a skill body, stop. Extract the X-skill and the Y-skill; let the harness do the routing.
