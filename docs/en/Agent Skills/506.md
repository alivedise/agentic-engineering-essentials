# [AEE-506] Skill Governance

## Context

A skill library with good management practices (AEE-505) — named owners, versioned skills, a registry — can still fail without governance. Governance is the operational layer: the policies, processes, and enforcement mechanisms that determine who can publish, how skills evolve, and how they are retired. Without governance, management practices drift.

The need for governance appears at a specific threshold: when the number of skills exceeds what any individual can hold in their head, and when a broken or outdated skill causes real operational problems. For most org-scale skill libraries, this threshold arrives around 20–30 active skills.

## Design Think

The core claim: skill governance is the operational layer that keeps skill management from collapsing under its own weight. Without governance, org-scale skill libraries accumulate outdated skills, contradictory standards, and no deprecation path. Governance defines who can publish, how skills are reviewed, and how they are retired.

**What constitutes a breaking change for a skill:**

For code, a breaking change breaks the API contract. For skills, a breaking change causes agents using the skill to behave differently in a way the consumer did not expect:
- Removing an invocation condition that consumers relied on
- Narrowing the scope so the skill no longer applies where it previously did
- Adding a hard constraint that overrides consumer-level customization
- Changing the expected output format (e.g., a code review skill that gave line-specific feedback now gives summary-only feedback)

Changes that are NOT breaking: improving clarity without changing behavior, adding "when not to use" guidance that doesn't affect existing use cases, updating examples.

**Versioning scheme:**

Semantic versioning (semver) applies directly:
- **MAJOR**: breaking change (behavior change, scope narrowing, constraint addition)
- **MINOR**: non-breaking addition (new guidance, expanded scope, new examples)
- **PATCH**: non-breaking fix (clarity improvement, typo correction, URL update)

MAJOR version bumps at org scope require a deprecation period before the old version is removed. Consumers must be able to pin to the old MAJOR version while they migrate.

**Deprecation lifecycle:**

1. **Deprecated**: still functional but marked deprecated in the registry. The skill body includes: "This skill is deprecated as of vX.0.0. Use `replacement-skill` instead. Will be removed on [date]."
2. **Sunset**: no longer updated; bugs are not fixed. Removal date confirmed.
3. **Removed**: removed from registry and distribution. Consumers who pinned continue to work; consumers following latest receive a missing skill error.

Minimum deprecation window at org scope: 30 days for a MINOR replacement (same purpose, better implementation); 90 days for a MAJOR replacement (different behavior, requires consumer adaptation).

**RFC 2119:**

- Skills MUST NOT be removed from org scope without a deprecation period communicated in the skill itself and in the registry.
- Org skill publishers MUST document breaking changes between major versions, including: what changed, why it changed, and how consumers should migrate.
- Audit trails for agent sessions SHOULD include the active skill set and versions at session start.

## Deep Dive

### Quality Gates Before Org Publication

Before a skill is promoted to org scope:

```markdown
## Org Skill Publication Checklist

- [ ] Single primary purpose — stated in one sentence without "and"
- [ ] Invocation condition defined — specific enough to auto-select
- [ ] "When NOT to use" section present — at least one exclusion specified
- [ ] Scope defined — explicit boundaries on what the skill covers
- [ ] Version assigned — following semver, starting at 1.0.0
- [ ] Owner named — an individual or team accountable for maintenance
- [ ] Variation tested — 5 positive cases and 5 negative cases passing
- [ ] Conflict check — no invocation condition overlap with existing org skills
- [ ] Changelog entry — describes what this version adds or changes
```

High-stakes skills (those active in every engineer's session that can affect production systems) warrant additional review: a trial period in team scope, multiple reviewers, and sign-off from a security or compliance stakeholder.

### Access Control

**Personal**: no control needed.

**Team**: any team member can propose; at least one other team member must review. A PR comment is sufficient.

**Org**: a designated group (DX team, platform team, or governance committee) approves publications. Separation of roles: the skill author proposes; the approver is a different person. No self-approval for org scope.

**Maintaining org skills**: the named owner can publish PATCH and MINOR updates without additional approval. MAJOR updates require the same review process as new skills.

### Audit Trails

An agent session that ran with skill version X and produced unexpected output cannot be diagnosed without knowing that version X was active. Session audit records must include:

```json
{
  "session_id": "sess_abc123",
  "timestamp": "2026-04-14T10:00:00Z",
  "active_skills": [
    { "name": "diagnose-root-cause", "version": "2.1.0", "scope": "org" },
    { "name": "backend:api-design", "version": "1.0.0", "scope": "team" }
  ],
  "active_tools": ["bash", "read_file", "search_web"]
}
```

This connects to AEE-406 (audit logging for tool calls): the same session record that captures tool calls should capture the active skill set. A skill update that changes agent behavior is as significant as a tool permission change.

### Governance Anti-Patterns

**The skill graveyard:** Skills in the org registry with no owner, no recent updates, and no consumers — but they remain "active" because no one went through the effort to deprecate them. The graveyard grows over time and degrades the registry's signal-to-noise ratio. Prevention: quarterly ownership audits; any skill without a confirmed owner for 90 days is auto-deprecated.

**The shadow personal skill:** A practitioner creates a personal skill that overrides an org skill because the org skill does not fit their workflow. The override is never reviewed, never surfaced to the team, and silently diverges from the org standard. When that practitioner's agent produces different output than everyone else's, the shadow skill is invisible to diagnosis. Prevention: org skills should log when they are overridden; practitioners should be encouraged to propose changes to org skills rather than maintaining silent overrides.

**The monolith skill:** One skill that covers everything for a domain — the org-level god-skill. It started focused and was extended over time because updating one file is easier than creating new skills and updating the registry. By the time the problem is visible, the monolith is too large to decompose without breaking consumers. Prevention: enforce the single-purpose checklist at org publication; treat every "and" in a skill body as a decomposition signal.

## Best Practices

1. **Publish the deprecation notice inside the skill body, not just in the registry.** When an agent loads a deprecated skill, the deprecation notice is part of what it reads. The agent can surface the deprecation to the practitioner inline, at the moment of use.

2. **Treat a skill update that changes behavior as a deployment, not an edit.** A behavioral change in an org skill affects every engineer in the organization. Apply the same care as a production deployment: announce in advance, provide a migration path, maintain the old version during the transition.

3. **Separate the active registry from the archive.** Deprecated and sunset skills should be in a separate section of the registry. Practitioners looking for skills to use should not wade through deprecated artifacts.
