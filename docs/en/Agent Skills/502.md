# [AEE-502] The Agent Skill Ecosystem

## Context

Agent skills exist in every major AI coding tool, but they are called different things: slash commands, rules, instructions, workflows, plugins. Each platform has its own invocation model, distribution mechanism, and scope of behavioral guidance. A practitioner who knows only their own tool's system is missing patterns and techniques proven in other contexts.

This article surveys the current state of the agent skill ecosystem — what exists, how each system works, and what they share despite their different names.

## Design Think

The core claim: the agent skill ecosystem is fragmented but converging. Skills exist under many names across different platforms, but they share a common pattern: packaged behavioral guidance loaded into agent context at invocation time. Understanding the landscape lets practitioners adopt proven patterns rather than reinventing them.

The convergence pattern — what every platform's skill system shares:

1. A named unit of behavioral guidance
2. A mechanism for loading that guidance into the agent's context
3. Some form of invocation trigger (explicit command, file presence, or condition matching)
4. A storage format (file, database entry, or API configuration)

The differences are in degree: how much structure is enforced, how composable skills are, how skills are distributed and versioned.

## Deep Dive

### The Agent Skills Open Standard

Before surveying individual platforms, it is worth naming the common substrate. The [Agent Skills open standard](https://agentskills.io) defines an agent skill as "a folder of instructions, scripts, and resources that agents can discover and use to do things more accurately and efficiently." The standard was originally developed by Anthropic and then released for broader ecosystem adoption. As of April 2026, over 30 tools implement the standard, including Claude Code, Gemini CLI, GitHub Copilot, Cursor, VS Code, Amp, and others.

The structural unit of a skill under this standard is a directory containing `SKILL.md` as the required entrypoint, with optional supporting files (templates, examples, scripts, reference docs). The specification is at [agentskills.io/specification](https://agentskills.io/specification) and the GitHub repository is at [github.com/agentskills/agentskills](https://github.com/agentskills/agentskills).

When platforms below are listed as implementing this standard, it means they recognize this directory structure and load `SKILL.md` when the skill is invoked.

### Claude Code Skills

Claude Code (the CLI) implements the Agent Skills open standard. Documentation is at [code.claude.com/docs/en/skills](https://code.claude.com/docs/en/skills).

**Storage paths:**

- Personal (user-level): `~/.claude/skills/<skill-name>/SKILL.md`
- Project-level: `.claude/skills/<skill-name>/SKILL.md`
- Plugin-level: `<plugin>/skills/<skill-name>/SKILL.md` (skills installed via the Claude Code plugin manager live inside the plugin's own directory tree)

The legacy `.claude/commands/` path continues to work for backward compatibility. Your existing `.claude/commands/` files keep working, but `.claude/skills/` is the recommended path for new skills.

**Invocation:** Users type `/skill-name` to invoke a skill explicitly. Skills can also be loaded automatically when Claude determines their description matches the current task. Both explicit invocation and automatic matching are supported.

**Note on terminology:** Earlier Claude Code documentation used "custom commands" and "slash commands" for this feature. The current documentation uses "skills" as the unified term, reflecting conformance with the Agent Skills open standard.

### Superpowers

[Superpowers](https://github.com/obra/superpowers) is an open-source agentic skills framework and software development methodology created by Jesse Vincent (obra) and community contributors. It is not an Anthropic product — it is a community plugin that installs a curated set of skills into Claude Code.

**Installation:** Via the Claude Code plugin system:

```
/plugin install superpowers@claude-plugins-official
```

The plugin listing is at [claude.com/plugins/superpowers](https://claude.com/plugins/superpowers). Distribution is through the Claude Code plugin manager and GitHub. There is no dedicated npm package for the project.

**Invocation:** Skills trigger automatically when their description matches the task context. Users can also invoke directly with `/skill-name` (for example, `/brainstorming`, `/execute-plan`). Both auto-invocation and manual invocation are supported.

**Relationship to Claude Code skills:** Superpowers installs skills that conform to the Claude Code skills system. When users install Superpowers via `/plugin install`, the plugin's skills become available alongside any skills the user has defined personally or at the project level.

### Cursor

Cursor has two overlapping behavioral guidance features: Cursor Rules and Cursor Skills.

**Cursor Rules** are the older and more widely used system. Documentation is at [cursor.com/docs/rules](https://cursor.com/docs/rules). Project rules live in `.cursor/rules` as markdown files and are version-controlled. Cursor supports both `.md` and `.mdc` extensions. MDC files support YAML frontmatter with fields including `description`, `alwaysApply`, and `globs` — allowing rules to be applied always, on file-pattern match, or based on description matching.

```
.cursor/rules/
  commit-style.mdc
  api-conventions.mdc
```

A typical MDC rule file:

```yaml
---
description: "Apply when the user asks about API design or REST endpoints"
globs: ["src/api/**"]
---

Follow our REST conventions: ...
```

**Cursor Skills** implement the Agent Skills open standard. The skills documentation is at [cursor.com/docs/context/skills](https://cursor.com/docs/context/skills). Cursor is listed on agentskills.io as an implementor pointing to that URL. The relationship between Cursor Rules and Cursor Skills is that Rules are Cursor's native behavioral guidance system, while Skills represent Cursor's participation in the cross-platform Agent Skills standard. As of April 2026, both systems are active in Cursor; practitioners using Cursor for team-shared behavioral guidance can choose either approach.

The older single-file `.cursorrules` format is not mentioned in current Cursor documentation. The canonical path is `.cursor/rules/*.mdc`.

### Custom GPT Instructions (OpenAI)

OpenAI Custom GPTs load behavioral guidance as persistent system-level instructions that apply to every conversation with that GPT. Unlike uploaded Knowledge files (which are used for retrieval), instructions are structural — they define how the GPT behaves, not what it knows.

Instructions are configured via the GPT Builder interface and stored as part of the GPT definition. There is no file-based storage path; instructions are entered as text in the builder UI and associated with the GPT record in OpenAI's systems.

**Invocation:** Instructions are injected automatically into every conversation. There is no per-session invocation trigger; the GPT's instructions are always active.

**Distribution:** A Custom GPT is shared as a URL or published to the GPT Store. The instructions travel with the GPT definition, not as a standalone artifact.

Relevant OpenAI documentation:
- [Creating a GPT](https://help.openai.com/en/articles/8554397-creating-a-gpt)
- [Key guidelines for writing instructions](https://help.openai.com/en/articles/9358033-key-guidelines-for-writing-instructions-for-custom-gpts)

Note: As of April 2026, the OpenAI Help Center blocks automated content fetching. The URLs above are confirmed via search index; specific behavioral details of the instructions system are sourced from search result summaries.

### Platform Comparison

| Platform | Format | Invocation | Distribution | Composability | Versioning |
|----------|--------|------------|--------------|---------------|------------|
| Claude Code | Directory + `SKILL.md` (Agent Skills standard) | `/skill-name` or auto-match by description | File system; plugin manager | Multiple skills compose at invocation time | File-based; use git for version control |
| Superpowers (plugin) | Directory + `SKILL.md` (via Claude Code) | `/skill-name` or auto-match | `/plugin install superpowers@claude-plugins-official`; GitHub | Composes with other Claude Code skills | Plugin versioning via plugin manager |
| Cursor Rules | `.mdc` or `.md` files in `.cursor/rules/` | `alwaysApply`, glob match, or description match | Git-committed with repository | Multiple rules apply simultaneously | File-based; use git |
| Cursor Skills | Directory + `SKILL.md` (Agent Skills standard) | Auto-match or explicit invocation | File system; cross-platform | Composes per standard | File-based; use git |
| Custom GPT (OpenAI) | Text field in GPT Builder UI | Always active (system prompt injection) | GPT URL or GPT Store | Single instruction set per GPT | GPT record versioning (builder UI) |

### The Convergence Pattern

Despite their differences in terminology and tooling, these systems share a common structure. Every platform's skill system:

1. **Names the guidance unit.** Whether it is a "skill," a "rule," a "custom GPT," or a "command," each platform has a named thing that represents a unit of behavioral guidance.

2. **Loads guidance into context.** Every system injects the guidance into the agent's context — as a system prompt, prepended instructions, or a loaded file — before the agent processes the request.

3. **Has an invocation mechanism.** The mechanism differs: always-on (Custom GPT), explicit command (`/skill-name`), condition matching (Cursor MDC frontmatter, Superpowers auto-match), or pattern-based (glob). But every system has one.

4. **Uses a storage format.** Files (Claude Code, Cursor), text fields (Custom GPT Builder), or database entries. File-based systems gain version control as a side effect; API-stored systems need separate versioning discipline.

The platforms that have moved toward the Agent Skills standard (Claude Code, Cursor, Gemini CLI, and others) are converging on the directory-plus-`SKILL.md` format specifically because it solves the versioning and distribution problems that ad-hoc text fields and single markdown files cannot. A skill structured as a directory can contain examples, templates, reference documents, and automation scripts alongside the core instructions — a richer artifact than a text field allows.

The remaining fragmentation is in distribution and composability. Custom GPTs are distributed as application instances; Agent Skills-conformant platforms distribute skills as files or packages. Composability in Custom GPTs is limited (one instruction set per GPT); in Agent Skills-conformant platforms, multiple skills compose naturally at invocation time.

## Best Practices

1. **Choose your skill format based on distribution needs, not familiarity.** If a skill needs to travel with a repository and apply across a team, file-based formats (Claude Code skills, Cursor rules/skills) are the natural choice — they version-control alongside the code they govern. If you are building a standalone assistant to share with non-technical users via a URL, Custom GPT instructions are appropriate. Match the format to the distribution channel, not to the tool you happen to use most.

2. **Add explicit invocation conditions even when your platform does not require them.** Cursor's `alwaysApply: true` and Custom GPT's always-on injection are convenient, but they mean every interaction carries the full weight of all active guidance. Explicit invocation conditions — a `description` field in Cursor MDC frontmatter, an invocation condition in a `SKILL.md` header — allow the harness or agent to select the right skill for the right task, reducing noise and improving reliability. Write the invocation condition first, before the body.

3. **Treat composability as a feature to design toward.** Single-skill systems (one big rule file, one Custom GPT instruction block) are easy to start with and hard to maintain. Platforms that support multiple composing skills allow you to build modular guidance libraries where skills for debugging, commit authoring, code review, and domain conventions stay separate and can be updated independently. When adopting a new platform, structure your guidance from the start as multiple named units rather than a single monolithic block.
