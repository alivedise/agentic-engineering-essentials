# [AEE-500] Skills vs. Tools

## Context

As agentic systems grow more sophisticated, two distinct abstraction layers have emerged for extending agent capabilities: **tools** and **skills**. These terms are used interchangeably in some frameworks but represent meaningfully different concepts. Conflating them leads to poorly designed agent architectures where behavior is fragile, hard to reuse, and difficult to maintain.

## Design Think

**Tools** are interfaces that connect an agent to the external world. They are executable capabilities: run code, read a file, call an API, search the web. Tools are stateless, side-effectful, and defined by their input/output schema. The agent decides when and how to call them.

**Skills** are packaged behavioral patterns that guide how an agent approaches a class of problems. They are loaded instruction sets: context that shapes the agent's reasoning, decision-making style, and sequence of actions for a particular task type. Skills are stateless in terms of side effects but stateful in terms of the agent's approach.

The key distinction:

| | Tools | Skills |
|--|-------|--------|
| What it is | Executable capability | Behavioral guidance |
| Invoked by | Agent (function call) | Harness or agent (context injection) |
| Side effects | Yes (writes, reads, calls) | No (shapes reasoning only) |
| Reusability unit | Schema + implementation | Prompt/instruction set |
| Example | `bash()`, `read_file()`, `search()` | `superpowers:debugging`, `superpowers:tdd` |

An agent MAY use both simultaneously: skills guide its approach to a problem, tools let it act on the world.

Skills SHOULD be designed with a single clear purpose. A skill that tries to do too many things becomes as brittle as a God object. A skill that is too narrow provides no leverage over just giving the instruction inline.
