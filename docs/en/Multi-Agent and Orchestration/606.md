# [AEE-606] Multi-Agent Failure Modes

## Context

Single-agent debugging is relatively straightforward: find the turn where the model produced wrong output, examine its context, identify the cause. Multi-agent debugging is different. The wrong final output may not originate at the last agent in the chain — it may originate at an earlier agent whose error propagated undetected through several correct-looking handoffs.

Understanding multi-agent failure modes means knowing not just what fails, but where in the coordination layer to look.

## Design Think

**Why single-agent debugging mental models fail in multi-agent systems**

In a single agent, a bad output traces to one model call. You examine the prompt, the context, and the model's response — the failure domain is a single turn. In a multi-agent system, a bad final output can be the composed result of multiple individually-plausible intermediate outputs. Each handoff looked correct. Each worker returned a well-formed result. The error was not visible at any individual step — it only became visible when the composed result was evaluated as a whole.

Applying single-agent debugging to this situation leads engineers to inspect the last agent in the chain first, because that is where the bad output appeared. But the last agent may have done exactly what it was asked to do, with the input it was given. The source of the failure is upstream — at an earlier handoff, a shared state write, or a delegation step that passed the wrong context forward.

**The coordination layer as the primary failure domain**

Multi-agent systems introduce a failure domain that single-agent systems do not have: the coordination layer. Handoffs, shared state, and delegation are not implementation details — they are the mechanisms through which multi-agent systems fail in ways that individual agents do not. A perfectly capable worker producing a perfectly valid output can still cause a system failure if the output contract was wrong, if another worker wrote conflicting state, or if the orchestrator delegated the wrong task.

**The five failure modes**

Multi-agent system failures cluster into five modes, each located in the coordination layer:

1. **Coordination breakdown** — a worker's output does not match what the orchestrator expected: wrong format, wrong scope, missing required fields.
2. **State inconsistency** — two workers write to shared state and produce conflicting results that cannot both be correct.
3. **Cascade failure** — one worker's bad output is passed to downstream workers as valid input; the error propagates through the dependency chain.
4. **Infinite delegation** — an orchestrator delegates a task; the worker re-delegates it; the loop continues until the system exhausts its budget or hangs.
5. **Silent success** — a worker reports success and returns a well-formed output, but the output is subtly wrong; no error is raised.

Each failure mode has a different location in the coordination layer, a different detection mechanism, and a different recovery strategy. Diagnosing a multi-agent failure requires identifying which mode is active before deciding where to look.

- Multi-agent systems MUST log each worker invocation with its input, output, and outcome.
- Output contract validation MUST happen at every fan-in point.
- Orchestrators MUST set a maximum delegation depth to prevent infinite delegation.

## Deep Dive

### Failure taxonomy

| Failure mode | Description | Detection | Recovery |
|---|---|---|---|
| Coordination breakdown | Worker produces output the orchestrator cannot use (wrong format, wrong scope, wrong contract) | Output contract validation at fan-in | Re-dispatch with clearer contract or error handling |
| State inconsistency | Two workers write to shared state and produce conflicting results | Conflict detection at merge | Conflict resolution strategy or serial execution |
| Cascade failure | One worker's failure causes downstream workers to fail on bad input | Output contract validation at each handoff | Fail-fast with error boundary at the failed stage |
| Infinite delegation | An orchestrator delegates a task, the worker re-delegates it, loop continues | Step counter / delegation depth limit | Maximum delegation depth; detect and abort cycles |
| Silent success | Worker completes task, reports success, but output is subtly wrong; no error raised | Review agent or automated test | Review loop pattern for quality-sensitive outputs |

### 1. Coordination breakdown

**What it looks like**

The orchestrator receives a result it cannot interpret — wrong format, missing required fields, or wrong scope (the worker solved a different problem than the one asked). The system either errors at the fan-in step or silently proceeds with an unusable result.

**Why it happens**

The output contract was underspecified: the worker was not told precisely what format the orchestrator needed. Alternatively, the worker had insufficient context to understand the full scope of its task, and solved a narrower version. In some cases, the worker encountered a problem it could not solve and produced a best-effort result that technically conforms to the format but answers the wrong question.

**Detection**

Validate worker output against the output contract before using it in any downstream task. Structured output types — JSON schema, typed responses — catch most coordination breakdowns automatically, because the validation is mechanically enforced rather than assumed. An unvalidated fan-in is a coordination breakdown waiting to happen.

**Recovery**

Re-dispatch with a more explicit output contract. If the contract was already explicit and the worker still produced wrong output, the issue may be worker capability — the task requires more context, a different prompt, or a more capable model.

---

### 2. State inconsistency

**What it looks like**

Two workers each produced correct-looking results in isolation, but when combined they produce a contradiction: two functions with the same name, two documentation sections with conflicting facts, two database migrations that both modify the same column in incompatible ways. Neither worker failed; the failure is a property of their combination.

**Why it happens**

Parallel workers that share mutable state — a file, a database, a shared context document — without a concurrency strategy will produce inconsistency whenever their writes overlap. The inconsistency is not visible during execution; it only becomes visible at the merge step, after all workers have completed.

**Detection**

Diff/conflict detection at merge. If workers write to shared state, the merge step must actively check for conflicts before accepting results. Do not assume that two well-formed results are compatible.

**Recovery**

Apply a conflict resolution strategy: one version wins (with defined priority), human arbitration, or a merge agent that synthesizes the conflicting results into a coherent whole. Alternatively, eliminate shared state and use message-passing: each worker produces an independent output, and a dedicated synthesis step combines them.

---

### 3. Cascade failure

**What it looks like**

Worker A fails or produces wrong output. Worker B receives Worker A's output as input and fails, or produces wrong output downstream. Worker C receives Worker B's output and fails further. The error propagates through the dependency chain, often growing in magnitude at each step.

**Why it happens**

The orchestrator treats Worker A's output as valid and passes it to Worker B without verification. There is no output contract validation between stages. By the time the failure is visible in the final output, multiple workers have already processed and amplified the original error.

**Detection**

Validate outputs at each handoff. Do not pass unvalidated output from one worker to the next. If Worker A's output cannot be validated against the contract Worker B expects, the failure should be surfaced at that boundary — not propagated forward.

**Recovery**

Fail-fast at the first failed stage: when Worker A's output fails validation, stop the pipeline at that boundary and do not dispatch Worker B. Re-dispatch from the failed stage after diagnosing and fixing the input or the worker's prompt. Propagating a known bad output through the remaining pipeline wastes resources and makes the failure harder to diagnose.

---

### 4. Infinite delegation

**What it looks like**

An orchestrator delegates Task X to Worker A. Worker A cannot complete Task X, but instead of returning a failure, it re-delegates Task X (or a variant of it) to Worker B. Worker B re-delegates to Worker C. The loop continues until the system exhausts its token budget, incurs runaway cost, or hangs indefinitely.

**Why it happens**

A worker that cannot complete its task but also cannot return a failure will delegate away the problem. This occurs when: (a) the worker's task description is ambiguous enough that it can be reasonably re-delegated to a sub-agent, or (b) the error contract does not allow the worker to express failure — so the only available action is further delegation. The root cause is an incomplete error contract: workers must be able to say "I cannot complete this task" in a structured way that the orchestrator can handle.

**Detection**

A delegation depth counter attached to each task. When a task is delegated, the counter increments. At each delegation step, the counter is checked against a maximum threshold. If the counter exceeds the threshold, the delegation is aborted and a structured failure is returned rather than delegating again.

**Recovery**

Set a maximum delegation depth before deployment. Three levels is a practical upper bound for most systems; beyond three levels, the system is almost always over-delegating or has a design problem in its task decomposition. On hitting the limit, return a structured failure to the orchestrator rather than delegating again. The orchestrator must have a defined behavior for this failure: escalate to a human, simplify the task, or fail the request cleanly.

---

### 5. Silent success

**What it looks like**

The worker reports success, returns a well-formed output, and the orchestrator accepts it. No error is raised. The output passes structural validation. But the output is subtly wrong: plausible but incorrect, complete but not coherent, formatted correctly but semantically invalid. The failure is invisible until downstream tasks consume the output or a human reviews the final result.

**Why it happens**

There is no check between what the worker returned and what the orchestrator actually needed. The output contract validates structure but not semantics. A worker that produces a correctly-formatted but wrong answer satisfies the structural contract while failing the semantic one.

**Why it is the most dangerous failure mode**

Silent success produces no error signal. Cascade failures at least fail loudly — the pipeline stops, an exception is raised, something is visibly broken. Silent success propagates through every downstream task, potentially reaching the final output before anyone notices. By the time it is detected, multiple downstream tasks may have consumed and acted on the wrong result, making recovery expensive.

**Detection**

A review agent: a second agent that evaluates the worker's output against intent and quality criteria, not just format. For high-stakes outputs, a human checkpoint is appropriate. Semantic validation — checking that the output answers the right question, not just that it is well-formed — is the only reliable detection mechanism for silent success.

**Recovery**

The review loop pattern (established in AEE-605): if the reviewer identifies a silent success, it returns FAIL with structured feedback, and the worker revises. If the review loop itself passes bad output, the reviewer's prompt is underspecified — the reviewer is not checking the right criteria. Diagnosing a review loop that consistently passes wrong output requires examining the reviewer's prompt and its evaluation criteria, not the worker.

### The diagnostic protocol

When a multi-agent system produces wrong final output, apply this protocol to locate the failure:

1. **Identify the earliest divergence point.** Work backward from the final output to find the earliest point in the chain where the output first diverged from expected. Do not assume the failure is at the last step.

2. **Check the input and output contracts at that point.** Was the input contract correct? Did the worker receive the context it needed? Was the output contract well-defined? Could the worker have produced a correct output given what it was asked?

3. **Check the handoff.** Was the worker's output validated before being passed to the next stage? Was it used correctly by the downstream task?

4. **Check the logs.** Which worker invocations were involved? What were their inputs and outputs? At which invocation did the output first become wrong?

This protocol works backward from the symptom to the source. It applies regardless of which failure mode is active — the mode determines what you find when you look at the coordination layer, not whether you look there.

---

### What to log

To make post-mortem diagnosis tractable, log the following per worker invocation:

- **Worker ID and role** — which agent type handled the task
- **Task description** — hash or full text, to reconstruct what the worker was asked
- **Input** — hash or structured content, to reconstruct what the worker received
- **Output** — hash or structured content, to reconstruct what the worker produced
- **Duration** — total time for the invocation
- **Success/failure and error code** — outcome of the invocation, with error type if failure

Without these fields, post-mortem diagnosis requires re-running the system to reproduce the failure. With them, the coordination layer can be inspected offline.

## Best Practices

1. **Validate output contracts at every fan-in.** An unvalidated fan-in is a coordination breakdown waiting to happen. Structural validation catches coordination breakdowns; it does not catch semantic failures. Both checks are necessary at every point where worker outputs are consumed.

2. **Set a maximum delegation depth before deployment.** Three levels is a practical upper bound for most systems. Beyond that, the system is almost always over-delegating. Define the maximum and define the behavior when it is reached — not at runtime, but as part of the system's design.

3. **Treat silent success as the failure mode to engineer against first.** The other four failure modes produce observable symptoms: the pipeline errors, the merge conflicts, the delegation loops. Silent success is invisible until you check. Build semantic review into any workflow where downstream tasks cannot be easily reversed. A review agent is not optional for quality-sensitive outputs — it is the only mechanism that catches failures the error contract cannot see.
