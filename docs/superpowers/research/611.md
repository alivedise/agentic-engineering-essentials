## Research notes

- **Version churn is real and load-bearing for the article.** As of 2026-04-28, v0.8 is labeled "Stable" and v0.9 is the new normative spec (released 2026-04-17), and a `v0_10` folder already exists in the repo's specification tree. The article should treat the message-type names (`createSurface` / `updateComponents` / `updateDataModel` / `deleteSurface` / `action` / `error`) as the v0.9 canon and call out that v0.8's older names (`beginRendering`, `surfaceUpdate`, `dataModelUpdate`) are still in the wild — many third-party integrations were authored against v0.8.
- **Disambiguation hazard confirmed.** `github.com/a2a-community/a2a-ui` is a separate community project for managing A2A agents and is NOT Google A2UI. Any GitHub citation must be under the `google` org.
- **AG-UI vs A2UI framing — primary source available.** The v0.9 blog provides the cleanest Google-side framing ("Any agent that already speaks AG-UI can drive A2UI v0.9 on day zero"). Combine with the v0.9 spec's transport-agnostic statement to make the relationship rigorous: A2UI is the payload contract; AG-UI is one of the transports it can ride. This is the single most important framing for AEE-611's relationship to AEE-610.
- **Status terminology drift.** The launch blog and the repo README call A2UI "Public Preview." The a2ui.org spec index labels v0.8 "Stable" and v0.9 "Draft." Both are accurate snapshots; the article should harmonize by saying the project is in Public Preview, with v0.8 as the current stable spec and v0.9 as the just-published next normative version.
- **Adopter quotes are uneven.** a2ui.org/ecosystem/a2ui-in-the-world/ has named-attributed quotes for Opal (Dimitri Glazkov), Gemini Enterprise (Fred Jabbour), Flutter GenUI (Vijay Menon), CopilotKit/AG-UI (Atai Barkai), and AG2 (Mark Sze). Vercel json-render and OpenClaw Canvas are listed without standalone pull quotes — cite them as ecosystem mentions rather than testimonials.
- **Component model nuance for the writer.** The flat-list-plus-adjacency design is not just a serialization detail — it's chosen so LLMs can stream components incrementally and so the `component:` discriminator is a fixed string rather than a dynamic key. The evolution guide quote ("much easier for LLMs to generate consistently than a dynamic key") is the clean justification for the Surface and Component Model section.
- **Flutter docs caveat.** docs.flutter.dev/ai/genui labels the `genui` package as alpha ("The `genui` package is in alpha and is likely to change") — worth noting in Best Practices when recommending Flutter integration as a starting path.
